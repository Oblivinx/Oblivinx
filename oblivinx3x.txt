This document is the single authoritative source of truth for implementing
Oblivinx3x v2.0 ("Nova"). Follow these conventions:

  [v2]       = Feature new or replaced in v2.0.0 vs. v1.2.0.
               Must be implemented as described.
  [KEEP]     = Feature carried over unchanged from v1.2.0.
  [BREAK]    = Breaking change from v1.2.0; migration path documented.
  [OPTIONAL] = Feature-flagged, can be compiled out for embedded targets.
  [IMPL NOTE]= Direct, unambiguous implementation guidance.

Implementation language: Rust (crates/) for the core engine.
Bindings: JavaScript via Neon (packages/oblivinx3x/), Python via PyO3,
         C FFI (libovn2.h), Go (cgo), WASM (wasm-bindgen).
Build output: ovn_neon.dll/.so/.dylib → rename to ovn_neon.node → copy to native/.

All sections are numbered. Cross-references use "Section N.N" notation.
When sections conflict, later sections override earlier ones.

Target compile profiles:
  oblivinx3x-full        — all features, x86-64 + ARM64 desktop/server.
  oblivinx3x-embedded    — MCU profile, no dist/ANN/FTS, ≤ 2 MB runtime.
  oblivinx3x-wasm        — WASM/browser with OPFS backend, ≤ 4 MB gzipped.
  oblivinx3x-edge        — serverless (Workers/Lambda) with R2/S3 backing.

CHANGELOG — v1.2.0 → v2.0.0 "Nova"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

MAJOR ADDITIONS:
  + I/O abstraction rewritten on io_uring (Linux) / IOCP (Windows) /
    kqueue (macOS/BSD); NVMe passthrough (OP_URING_CMD) optional.
  + Buffer Pool replaced: segmented-LRU → ARC (Adaptive Replacement Cache)
    with CLOCK-Pro fallback for small-memory targets.
  + AHIT v2.0: adds a Learned Index tier (PGM++ / LMG hybrid) between
    Hot Zone and Cold Zone. AHIT v1 B+tree kept as the default fallback.
  + OBE2 encoding: SIMD-accelerated decoder (AVX-512, NEON), dictionary
    encoding for hot string fields, 10-35% smaller than OBE v1.
  + Columnar Storage Engine (Hybrid Row-Columnar / HTAP): documents auto-
    promoted to a columnar mirror when access pattern is analytical.
    Arrow IPC and Parquet read/write interop.
  + Vector subsystem v2: DiskANN graph, RaBitQ quantization, SPFresh
    incremental updates, LSM-VEC for billion-scale dynamic workloads.
    Hybrid search (BM25 + vector + fuzzy) with Reciprocal Rank Fusion (RRF).
  + Security Layer: Queryable Encryption (equality + range + prefix/suffix
    + substring), column-level AES-256-GCM-SIV, row-level security (RLS),
    OCSF-schema audit log, key rotation, envelope encryption via external
    KMS (AWS KMS, Azure Key Vault, GCP KMS, Vault, file-based dev KMS).
  + Concurrency v2: BEGIN CONCURRENT for multi-writer MVCC, Hybrid
    Logical Clocks (HLC) for distributed ordering, Deterministic
    Transactions mode (Calvin-style) for cross-shard ACID.
  + Change Data Capture (CDC): log-based, durable, replayable, separate
    from Change Streams. Supports Debezium-compatible envelope format.
  + Incremental View Maintenance (DBSP-based): materialized views now
    refresh incrementally for every aggregation stage, not just a subset.
  + Edge / WASM / Browser target: OPFS-backed persistence, IndexedDB
    fallback, bi-directional sync with primary (Turso-style embedded
    replicas).
  + AI/Embedding Layer: pluggable embedding providers, native RAG
    primitives (chunk, embed, search, rerank in one pipeline), cached
    embeddings stored alongside documents.
  + OpenTelemetry native: traces, metrics, logs; Prometheus exporter
    now an OTEL wrapper.

MAJOR REMOVALS / REPLACEMENTS:
  - Removed: Neon's raw buffer exposure in JS bindings (was v1 leak).
  - Replaced: TTL Reaper → Unified Expiry Worker (TTL + CDC + GC coalesced).
  - Replaced: WAL "fsync per record" default → group commit is now default
    (D1 now means "group commit"; old D1 becomes D1-Strict).
  - Removed: in-memory-only Savepoint restriction; savepoints can now be
    durably checkpointed if pragma durable_savepoints = true.

BREAKING CHANGES (all auto-migrated on first open):
  [BREAK-1] File Header Page 0 layout extended; magic bumped to 0x4F564E32
            ('OVN2'). Old v1 files detected by 0x4F564E58 and auto-migrated
            read-only; write-migration requires explicit db.migrate_v1_v2().
  [BREAK-2] OBE Type Tag 0x12 (RelRef) semantics extended; new tags
            0x13..0x1A added. Readers ≥ v1.2.0 will see v2-specific tags
            as "unknown field" and skip them (forward-compat by design).
  [BREAK-3] New Page Types 0x09..0x10 added; old code must be rebuilt.
  [BREAK-4] Pragma values that used boolean-only now accept string variants.

NON-BREAKING IMPROVEMENTS:
  * All existing MQL operators continue to work unchanged.
  * All existing JS API signatures preserved; new options are additive.
  * v1 .ovn files are readable by v2 in compatibility mode.

Section 1 — Ecosystem Analysis 2026
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

This section refreshes v1's ecosystem analysis with systems that matured
or emerged between late 2024 and early 2026. The selection is chosen to
surface architectural patterns that Oblivinx3x v2.0 adopts or explicitly
rejects.

1.1 Document and Multi-Model Databases
──────────────────────────────────────

1.1.1 MongoDB 8.x (2024-10 → 2026)
Storage Engine: WiredTiger retained, with time-series now column-compressed
                on-ingest (eliminating a prior decompress-recompress path).
Key 8.0/8.2 additions:
  - Queryable Encryption: range, prefix, suffix, substring queries on
    fully-encrypted fields (Cryptography Research Group's encrypted-search
    algorithm). Client-side encryption; server never sees plaintext.
  - Time-series block processing: 200% faster aggregations via columnar
    compression written directly on insert.
  - Ingress admission queue for backpressure.
  - 36% faster reads, 59% higher update throughput, 50× faster resharding.
  - OCSF (Open Cybersecurity Schema Framework) audit format.
  - Vector search: $vectorSearch stage integrates with full-text via
    $rankFusion (Reciprocal Rank Fusion) on Atlas; Community Edition
    since 8.2 includes the mongot binary for self-managed deployments.

Oblivinx implication: adopts Queryable Encryption pattern, OCSF audit
schema, RRF-based hybrid search, and columnar time-series storage.

1.1.2 FerretDB 2.x
PostgreSQL-backed MongoDB wire-protocol. Demonstrates that document APIs
can ride on relational engines. Proves the reverse direction works too —
reinforces v2's decision to keep the Relational Compatibility Layer.

1.1.3 Couchbase Capella Columnar (2024-2025)
Added a separate columnar service for analytics over operational data.
Relevant to Oblivinx's new HTAP engine (Section 15).


1.2 Embedded and Edge-Native Databases
───────────────────────────────────────

1.2.1 Turso Database (Rust rewrite of SQLite, 2024-2026)
Not a fork — a full rewrite. Key design decisions:
  - Async I/O via io_uring on Linux; equivalent async paths on other OSes.
  - BEGIN CONCURRENT for multi-writer MVCC (replaces SQLite's single-writer).
  - Change Data Capture (CDC) for real-time change tracking.
  - Embedded Replicas: each client has a local SQLite file that syncs to
    a primary; reads are sub-ms local, writes forwarded to primary.
  - Bi-directional sync with offline support (beta).
  - Incremental computation via DBSP for view maintenance and query
    subscriptions.
  - Full-text search via Tantivy.
  - Multi-process WAL coordination via a .tshm sidecar file.
  - Encryption at rest (experimental).

Oblivinx implication: all of the above are adopted in v2.0, adapted to
the document model. DBSP is the basis for the new IVM section.

1.2.2 libSQL (SQLite fork)
Full SQLite wire + file compatibility. Adds vector search as a native
data type, embedded replicas, and remote HTTP access. Inherits SQLite's
single-writer limitation — Turso Database was the answer.

1.2.3 DuckDB 1.x
Embedded OLAP. Vectorized execution engine (Morsel-driven parallelism).
Columnar. Great for the "analytics on the same file as transactions"
pattern. Oblivinx v2's Columnar Engine (Section 15) borrows the morsel
idea for vectorized scans.

1.2.4 Cloudflare D1 / Durable Objects (2024-2026)
D1 is SQLite behind an HTTP API; Durable Objects embed SQLite (up to 10
GB per object) for strongly-consistent stateful serverless compute.
Architectural lesson: the "database per tenant / per agent" model scales
economically when the engine is embedded, not client-server.

1.2.5 Cloudflare R2 / S3 Vectors (2025)
Object-storage-backed vector indexes. Oblivinx v2's optional remote
backing-store mode adopts this pattern: .ovn2 files can be cold-tiered
to R2/S3, with hot pages cached locally.

1.2.6 KuzuDB 0.6+ (2025)
Embedded graph database with Cypher; columnar node/edge storage;
vectorized Hash Join. Validates the columnar-for-graph pattern. Oblivinx
stays document-first but adopts vectorized joins for the HTAP engine.

1.2.7 SurrealDB 2.x
Multi-model (document + graph + KV + full-text). Written in Rust. Live
queries (similar to change streams). Validates the "one embedded engine,
many paradigms" thesis that underpins Oblivinx.


1.3 Vector-First and AI-Native Databases
────────────────────────────────────────

1.3.1 turbopuffer (2024-2026)
Object-storage-native vector DB. SPFresh + RaBitQ. Proves that
billion-scale vector search can be done with O(cents) per million
vectors by offloading cold shards to S3-class storage.

1.3.2 LanceDB (Lance format)
Columnar vector DB. Arrow-compatible. Validates co-locating vectors and
document metadata in the same row-group. Oblivinx v2's HTAP engine
extends this idea to full documents.

1.3.3 Weaviate 1.27+, Milvus 2.5+, Qdrant 1.12+, Pinecone Serverless
All adopted RaBitQ-class quantization in 2025 for 30-50% memory
reduction at equal recall. Consensus: HNSW is losing market share to
DiskANN + RaBitQ for disk-resident billion-scale workloads, while HNSW
remains the champion for in-memory, million-scale workloads.

1.3.4 RaBitQ (SIGMOD 2025)
Practical, asymptotically optimal quantization. Adopted by Elasticsearch,
Lucene (as "BBQ"), CockroachDB (CSPANN), turbopuffer, VectorChord,
Volcengine OpenSearch. De facto standard for scalar/binary quantization.

1.3.5 DiskANN v2 (Microsoft Research) and FreshDiskANN / SPFresh
DiskANN: Vamana graph stored on SSD, sub-10ms recall@10 ≥ 0.95 on
1-billion-vector datasets with only a few GB of RAM.
SPFresh (SOSP 2023, production-deployed by 2025): incremental in-place
update for billion-scale vector search.
LSM-VEC (2025): pairs a hierarchical proximity graph with an LSM-tree
backend for dynamic workloads; outperforms DiskANN/SPFresh on dynamic
billion-scale benchmarks.

Oblivinx implication: v2 supports HNSW (for in-memory/small), DiskANN
(for disk-resident/large), and LSM-VEC-style dynamic as compile-time
options. RaBitQ quantization is applied uniformly.


1.4 Learned and Adaptive Index Research
────────────────────────────────────────

1.4.1 PGM-Index (Ferragina & Vinciguerra, VLDB 2020)
Fully-dynamic learned index with provable worst-case bounds. Up to
three orders of magnitude less space than B+ trees at equal query
performance in the static case; up to 71% faster with 1140× less space
fully-dynamic.

1.4.2 PGM++ (VLDB 2025)
Hybrid branchless search + layer-skipping. Up to 2.31× PGM-Index
speedup at comparable space; up to 1.56× over SOTA baselines.

1.4.3 ALEX (SIGMOD 2020) and descendants
Gapped-array learned index; updatable in-place. Good under moderate
workload drift; degrades under adversarial drift.

1.4.4 LIPP (VLDB 2021)
Updatable learned index with precise positions; beats ALEX 2.9× and
B+ tree 13.5× on read-heavy workloads.

1.4.5 LMG-Index / LMIndex (preprint late 2025)
Linear-model + gapped array hybrid. Outperforms ALEX/SWIX/B+ by up to
11× point queries, 5× range queries at equal or lower memory.

1.4.6 Adaptive Radix Tree (ART)
Still the in-memory SOTA for variable-length keys; 8× faster than
B+ tree for point lookups on many workloads. Does not replace B+ tree
for range scans.

Oblivinx implication: v2 adopts ART for the in-memory hottest tier of
AHIT v2, PGM++ (with LMG-style segmentation) for the mid tier, and
B+ tree for the cold tier — a true three-tier hybrid (Section 4.2).


1.5 Modern Storage and I/O
──────────────────────────

1.5.1 io_uring (Linux 5.1+, mature since 6.x)
By 2025-2026, systematic studies (Jasny et al., VLDB 2025/2026) show:
  - Registered Buffers: +11% YCSB throughput (zero-copy DMA).
  - NVMe passthrough (OP_URING_CMD): additional +20% (skips block layer).
  - IOPOLL: +21% (polled completions, no interrupt).
  - SQPOLL (dedicated kernel poll thread): +32% (no syscalls).
  - Out-of-memory OLTP (TPC-C): up to 12.5× over blocking I/O.
  - Zero-copy network send/recv: 2.5× over epoll+copy for payloads > 1 KiB.

PostgreSQL 18 io_uring integration: +11-15% cold-scan throughput.

Oblivinx implication: v2 mandates io_uring on Linux for file I/O.
Ring-per-thread architecture. Registered buffers for the Buffer Pool.
Optional NVMe passthrough pragma for dedicated storage deployments.

1.5.2 Direct I/O and SPDK
Direct I/O (O_DIRECT) bypasses the OS page cache. SPDK goes further:
user-space NVMe driver, polling, no kernel involvement.
Oblivinx v2 supports O_DIRECT; SPDK is deferred to v2.2.

1.5.3 PMEM and CXL Memory
Persistent Memory (Optane) was EOL'd, but CXL-attached memory became
mainstream in 2025. CXL 2.0+ enables memory pooling and large cheap
RAM tiers. Oblivinx v2 adds an optional CXL memory tier for the
Buffer Pool and for AHIT Hot Zone overflow.

1.5.4 Kernel-Bypass Networking (DPDK, AF_XDP)
Relevant for distributed mode (Section 12). Oblivinx-Dist uses
AF_XDP when available, falls back to io_uring zero-copy.


1.6 Concurrency and Transaction Research
────────────────────────────────────────

1.6.1 Hybrid Logical Clocks (HLC)
Adopted widely (CockroachDB, YugabyteDB, FaunaDB). Provides monotonic,
causally-consistent timestamps across distributed nodes without the
TrueTime hardware requirement. Oblivinx v2 uses HLC for TxID in
distributed mode.

1.6.2 Serializable Snapshot Isolation (SSI) — Cahill Algorithm
Kept from v1. Extended with v2's dangerous-structure detection for
write skew across attached databases.

1.6.3 Deterministic Databases (Calvin, FaunaDB, PlanetScale Vitess)
Instead of MVCC conflict detection, pre-order transactions globally
and then execute deterministically. Oblivinx v2 offers this as an
opt-in mode for cross-shard transactions only.

1.6.4 BEGIN CONCURRENT
Turso Database feature. Multiple writers proceed concurrently within
their own MVCC snapshots; conflicts detected at commit. Oblivinx v2
adopts this as the default write path; old single-writer WAL becomes
an opt-in compatibility mode.


1.7 Security Research
─────────────────────

1.7.1 MongoDB Queryable Encryption (GA 2023 equality; 8.0 range; 8.2 substring)
Encrypted-index data structure: deterministic tagging of each encrypted
value to a small set of "tags" that the server can match without
decryption. Key property: client holds keys, server sees only tags.
Oblivinx v2's QE (Section 13.3) implements the same primitive, reusing
the well-reviewed cryptographic design.

1.7.2 Row-Level Security (RLS) — PostgreSQL model
Declarative per-row policies evaluated on read/write. Oblivinx v2
implements RLS on top of triggers + query rewriting (Section 13.4).

1.7.3 OCSF (Open Cybersecurity Schema Framework)
Standardized audit log format. MongoDB 8.0 adopted it. Oblivinx v2
emits audit records in OCSF JSON by default.

1.7.4 EU Cyber Resilience Act (CRA, first enforcement Sept 2026)
Mandates 24-hour vulnerability reporting and security-by-design for
any connected product. Oblivinx v2 ships with a CRA-ready posture:
signed releases, SBOM export, hardened defaults, audit logs on.


1.8 Observability
─────────────────

OpenTelemetry is now the default for traces/metrics/logs (CNCF graduated
2024). Prometheus/StatsD are special cases of OTEL exporters. Oblivinx
v2 emits OTEL natively; Prometheus exporter is a thin wrapper.

Section 2 — Gap Discovery (Updated for 2026)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The five gaps from v1 remain valid. Four new gaps emerged by 2026.

Gap 1 — No Embedded Native Document Database [UNCHANGED]
No production-grade, embedded document DB exists without a server.
SQLite JSON1 is bolted on. Turso is relational, not document. Realm is
mobile-only. Oblivinx3x's primary raison d'être.

Gap 2 — Write Amplification in Embedded Stores [UNCHANGED]
SQLite B-tree: 5-10× amp. RocksDB LSM: 10-30×. Turso improves writes
but is still a B-tree. Oblivinx's hybrid B+/LSM + Learned Index
addresses all three.

Gap 3 — Inefficient Compound Indexing [UNCHANGED]
Leftmost-prefix rule still hurts. v2 keeps Skip-Column Optimization
(SCO) from v1 and layers Learned Index on top for further gains.

Gap 4 — Memory Pressure [PARTIALLY CLOSED, NEW FRONTIER]
v1's WSE-adjusted buffer pool was good. In 2026 the new frontier is
mixed-hardware memory (DRAM + CXL + PMEM) and the need for an allocator
that spans them. Oblivinx v2 ships a tiered Buffer Pool.

Gap 5 — No Lightweight LSM + ACID System [CLOSED BY TURSO, NEW ANGLE]
Turso proved this is possible for relational. Nobody has done it for
documents + vectors + full-text + graph in one embedded file.
Oblivinx v2 is the answer.

Gap 6 — No Embedded Document DB with Queryable Encryption [NEW]
MongoDB has QE, but requires a server. No embedded document DB offers
QE. This is a moat for Oblivinx v2 in regulated industries (healthcare,
finance, HR, legal).

Gap 7 — No Embedded DB with Bi-Directional Sync [NEW]
Turso has it for relational (beta). Nobody has it for documents +
vectors. Critical for AI agents that run offline-first.

Gap 8 — No Embedded DB with Hybrid Row-Columnar Storage [NEW]
DuckDB is OLAP-only. MongoDB time-series is columnar but MongoDB is
not embedded. Turso is row-only. Oblivinx v2 offers HTAP in a single
file, with the columnar mirror generated transparently.

Gap 9 — No AI-Native Embedded DB [NEW]
Embedded DBs treat vectors as an index type. Oblivinx v2 treats the
full RAG pipeline (chunk → embed → search → rerank) as first-class,
with pluggable embedding providers, cached embeddings, and filtered
ANN by document attributes.

4.2 AHIT v2.0 — Three-Tier Adaptive Index Tree [v2]

[IMPL NOTE] AHIT v2 is a three-tier structure. Implement as three nested
data structures sharing one logical index namespace. Promotion/demotion
moves entries between tiers; the external API never distinguishes them.

  Tier 0 (Hottest, Memory):  Adaptive Radix Tree (ART)
    - Variable-length key support, path compression, lazy expansion.
    - O(k) point lookups where k = key length (not log N).
    - 5-10× faster than B+ tree for point lookups on typical workloads.
    - Bounded size (default 32 MB per collection; configurable).

  Tier 1 (Hot, Memory):      Learned Index (PGM++ with LMG-style segmentation)
    - Piecewise-linear approximation of the key→position mapping.
    - O(log log N) height for billions of keys.
    - Up to 1000× smaller than B+ tree at equal query speed.
    - Handles bulk-loaded / static-ish subtrees (e.g. historical partitions).
    - Rebuild cost amortized by incremental segment splits (LMG style).

  Tier 2 (Cold, Disk):       B+ Tree + immutable SSTables (AHIT v1 retained)
    - Classic B+ tree for the permanent store.
    - L0..Ln SSTables for recent writes (LSM write intake).
    - Bloom filters per SSTable (1% FP rate, k=7).

Promotion/Demotion logic (AHIT v2 Promoter thread):
  Cold → T1:   When a B+ tree subtree's access counter exceeds
               (default 500 accesses / 60 s) AND subtree size ≤ 1 MiB,
               bulk-load into a new learned-index segment. The subtree
               keeps a "shadow" in B+ tree until T1 segment is verified
               (CRC match after bulk-load).
  T1 → T0:     When an individual key in a learned segment exceeds
               (default 1000 accesses / 60 s), insert into ART.
  T0 → evict:  LRU eviction when ART > max_size. Evicted key re-enters T1.
  T1 → evict:  When a learned segment's hit rate < 10% for > 120 s,
               drop its in-memory model (B+ tree fallback takes over).

Why three tiers beat v1's two tiers:
  - ART eliminates tree-height for ultra-hot keys (e.g. top-100 users).
  - Learned index delivers billion-scale point-lookup speed in RAM with
    1000× less space than a full in-memory B+ tree. This is the enabling
    technology for keeping most of the index in RAM on edge devices.
  - B+ tree / SSTable remains the durable, range-scan-friendly foundation.

Write path (simplified): all writes hit the MemTable → replicate to
Tier 0 (ART) and Tier 2 (B+/LSM). Tier 1 is built lazily from Tier 2
during compaction, never directly written by the write path.

Read path (simplified):
  1. Check Tier 0 (ART): O(k). If hit, return.
  2. Check Tier 1 (Learned): O(log log N) + small local search. If hit, return.
  3. Check MemTable: O(log N). If hit, return.
  4. Check Tier 2 (B+/SSTable) via Bloom filter: O(1) miss detection.
     On hit: O(log N).

Complexity gain over AHIT v1:
  Point lookup avg: O(log N) → O(k) for hottest 1-5% of keys,
                    O(log log N) for next 10-20%,
                    O(log N) for the rest.
  Space per key (index, typical): 40-60 bytes → 1-8 bytes for learned tier.

[IMPL NOTE] Learned Index safety: the learned model can misfit under
workload drift. Implement a bounded-error fallback: if the predicted
position is wrong by more than epsilon, fall back to exponential search
within the segment (ALEX-style). If the fallback rate exceeds 10%,
rebuild the segment.


4.3 Hybrid Storage Engine (Row + Columnar / HTAP) [v2]

[IMPL NOTE] Every collection has a primary row-oriented representation
(OBE2-encoded documents in Data Segment pages). Additionally, columns
that are accessed analytically (detected by the Workload Profiler) are
maintained as columnar mirrors in ColumnChunk pages (Section 5.5).

Row representation (always present):
  OBE2-encoded documents in the Data Segment. This is the source of
  truth for all writes, single-document reads, and MVCC versioning.

Columnar mirror (optional, per-column):
  For each "hot analytical" column, ColumnChunk pages store:
    - Dictionary-encoded values (for low-cardinality columns)
    - Run-length encoded values (for sorted or clustered columns)
    - Bit-packed values (for small-range integer columns)
    - Zone maps (min, max, null-count per 64 KiB chunk)
    - Optional Arrow-compatible on-disk layout (for zero-copy export)

Detection of "hot analytical" columns:
  The Workload Profiler (running every 60 s) tracks:
    scans_per_column    : how many full-collection scans touch this field
    aggregations_on     : how many $group / $sum / $avg target this field
    rows_read_per_query : avg K for queries touching this field
  Column qualifies for columnar mirroring when:
    scans_per_column + aggregations_on ≥ 10 per minute for 5 minutes
    AND avg rows_read_per_query ≥ 1000
  Developer override: db.setColumnar('users', 'email', true) to force.

Consistency:
  The columnar mirror is maintained by the Columnar Flusher background
  worker. It reads the WAL after commit and appends to ColumnChunks.
  Reads from the columnar mirror use the snapshot TxID: rows with TxID
  beyond the chunk's watermark fall back to the row store for that
  query. This gives column scans consistent snapshot semantics.

Cost-based routing (Query Engine):
  For an aggregation that can be answered from the columnar mirror,
  the optimizer routes there when:
    (K × PAGE_COST_columnar) + (K × DECODE_COST_columnar)
      < (K × PAGE_COST_row) + (K × DECODE_COST_row)
  Typically wins for K ≥ 10,000 with selectivity ≥ 10%.

Storage overhead:
  Columnar mirror adds 0.3-0.6× the row storage (well-compressed).
  Disable per collection via pragma columnar_mirror = false.


4.4 Document Layer v2 (OBE2)

Implements OVN Binary Encoding v2 (OBE2, Section 5.4). Handles:
  - Document parsing and field extraction (SIMD-accelerated where available)
  - Dot-notation path resolver for nested access
  - Type system and type validation
  - Optional JSON Schema validation per collection
  - Dictionary-encoded hot fields (40-60% storage reduction for string-heavy
    collections beyond v1's Field-ID trick)
  - Forward-compatible unknown-tag skipping (for readers older than the writer)

OBE2 adds these type tags (0x13..0x1A) on top of v1's 0x01..0x12:
  0x13  Vector            (f16/f32/i8 quantized, dimension varint)
  0x14  EncryptedField    (QE-compatible encrypted blob + tags)
  0x15  BigDecimal        (arbitrary precision decimal)
  0x16  DateTimeTZ        (timestamp + timezone offset minutes)
  0x17  Duration          (signed i64 nanoseconds)
  0x18  JSONPatch         (RFC 6902 delta, for update operators)
  0x19  EmbeddingRef      (points to a cached embedding in Embedding Cache)
  0x1A  TaggedUnion       (sum type with variant tag)

Schema validation behavior (unchanged from v1):
  validationLevel:  'strict' (default) | 'moderate'
  validationAction: 'error' (default) | 'warn'

[IMPL NOTE] OBE2 decoder uses AVX-512 where available, NEON on ARM64,
scalar fallback on WASM. The decoder is a pure function on a byte slice —
never allocates, never copies. This is essential for the zero-copy
vectorized query executor (Section 4.6, Section 7).


4.5 MVCC Transaction Layer v2 (HLC, BEGIN CONCURRENT)

MVCC at the document level. Each write creates a new document version
stamped with a TxID. Previous versions retained until GC'd.

TxID generation (v2):
  Single-node mode:       monotonic u64 (hardware clock + sequence).
  Distributed mode:       Hybrid Logical Clock (HLC) — 64-bit timestamp
                          (48b physical ms + 16b logical counter).
                          HLC guarantees causal ordering across nodes
                          without requiring synchronized clocks.

Isolation levels (pragma isolation_level):
  'read_committed'       — see only committed data (cheap; MVCC skip).
  'snapshot' (default)   — consistent snapshot at txn start.
  'serializable'         — SI + Cahill SSI anti-dependency detection.
  'deterministic' [opt]  — Calvin-style pre-ordering (cross-shard only).

Concurrent writers — BEGIN CONCURRENT [v2 NEW]:
  By default, multiple writers may proceed in parallel, each with its
  own MVCC snapshot. Conflicts are detected at commit:
    - Lower-TxID writer wins (optimistic).
    - Higher-TxID writer retries with a fresh snapshot.
  Retry is automatic (up to retryable_writes limit).
  Legacy single-writer mode: pragma concurrent_writes = false.

[IMPL NOTE] v2's Savepoints are optionally durable:
  pragma durable_savepoints = true → savepoints write special WAL records
    (SVPT_BEGIN / SVPT_ROLLBACK). On crash, the txn can be resumed at the
    last savepoint rather than discarded entirely.
  pragma durable_savepoints = false (default) → v1 in-memory-only behavior.


4.6 Query Engine v2 (Vectorized Execution)

Pipeline: Lexer → Parser → AST → Logical Plan → Physical Plan → Vectorized Executor.

Vectorized Executor (new in v2):
  Operators process batches of 1024-4096 rows at a time, not one row at a
  time. Each operator reads from columnar input batches (Arrow-compatible
  layout) and writes to columnar output batches. This enables:
  - SIMD-accelerated predicate evaluation (equality, range, string prefix).
  - Cache-friendly tight loops (branch-predicted, prefetch-friendly).
  - Morsel-driven parallelism (à la DuckDB): a single query can be
    split across N threads working on disjoint morsels.

For row-oriented queries (single-document CRUD), the executor falls back
to the classic tuple-at-a-time path. The switch is automatic based on
plan cardinality.

Optimizer transformations (v2 adds to v1):
  1. Predicate pushdown (v1).
  2. Index selection (v1).
  3. Sort merging (v1).
  4. Geospatial plan selection (v1).
  5. Zone-map pruning [v2]: skip pages whose min/max ranges exclude the predicate.
  6. Columnar routing [v2]: route analytical queries to the columnar mirror.
  7. Vector + filter rewrite [v2]: push scalar filters into filtered ANN.
  8. Hybrid search fusion [v2]: combine BM25 + vector via RRF in a single plan.
  9. Learned-index-aware costing [v2]: factor tier hit probability into cost.
  10. Batch consolidation [v2]: merge adjacent $match stages into one SIMD predicate.

4.11 I/O Abstraction Layer [v2 NEW]

[IMPL NOTE] All storage I/O flows through a single trait:
  trait IoEngine {
    async fn read(&self, handle: FileId, offset: u64, buf: &mut [u8]) -> Result<usize>;
    async fn write(&self, handle: FileId, offset: u64, buf: &[u8]) -> Result<usize>;
    async fn fsync(&self, handle: FileId) -> Result<()>;
    async fn fdatasync(&self, handle: FileId) -> Result<()>;
    async fn range_sync(&self, handle: FileId, off: u64, len: u64) -> Result<()>;
    fn register_buffer(&self, buf: &[u8]) -> BufferHandle; // for zero-copy
    fn capabilities(&self) -> IoCapabilities;
  }

Concrete implementations (chosen at compile time or runtime):
  IoEngineUring     — io_uring on Linux 5.6+. Ring-per-thread.
                      Registered buffers + optional NVMe passthrough.
  IoEngineIocp      — Windows IOCP (Completion Ports). 
  IoEngineKqueue    — macOS/BSD kqueue (with posix_aio fallback).
  IoEnginePosix     — Blocking pread/pwrite + thread pool (fallback).
  IoEngineOpfs      — Browser OPFS (Origin Private File System).
                      Sync API in a dedicated Worker; async to caller.
  IoEngineIndexedDb — Browser IndexedDB fallback (when OPFS unavailable).
  IoEngineObject    — S3 / R2 object storage (edge/serverless).
                      Range-GET with local page cache.

Runtime selection (Linux example):
  if kernel ≥ 5.6 && io_uring available: IoEngineUring.
  else if POSIX AIO available:           IoEnginePosix (AIO path).
  else:                                   IoEnginePosix (thread-pool path).

NVMe passthrough activation (opt-in pragma nvme_passthrough = true):
  - Requires raw device access (/dev/ngXnY).
  - Skips Linux block layer entirely.
  - Highest throughput (+20% over plain io_uring on PCIe 5 NVMe).
  - Requires CAP_SYS_RAWIO or root; unsafe without device-exclusive access.

SQPOLL mode (opt-in pragma sqpoll_cores = N):
  Dedicates N kernel polling threads for io_uring. Eliminates submission
  syscalls; highest throughput but pins CPU cores. Use only on dedicated
  DB hosts.

Direct I/O (opt-in pragma direct_io = true):
  O_DIRECT on Linux / FILE_FLAG_NO_BUFFERING on Windows. Bypasses OS page
  cache in favor of Oblivinx's own Buffer Pool. Strongly recommended on
  servers with dedicated NVMe; not recommended on laptops or shared hosts.

Section 5 — .OVN2 Database File Format
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

.OVN2 is a single-file, page-oriented, self-describing binary format.
It is a strict superset of v1's .ovn — v1 readers handle v2 files up
through the common segments but cannot read v2-only segments.
All integers: little-endian. All strings in headers/metadata: UTF-8.

5.1 File Header (Page 0 — Always 4096 bytes)

Offset   Size   Field
──────────────────────────────────────────────────────────────
0x0000    4     Magic Number: 0x4F564E32 ('OVN2')   [BREAK from v1 'OVNX']
0x0004    2     Format Version Major (0x0002)
0x0006    2     Format Version Minor (0x0000)
0x0008    4     Page Size in bytes (default 4096; valid: 512..65536)
0x000C    8     Total file size in bytes
0x0014    8     Root Page Number of Primary B+ Tree (row store)
0x001C    8     Root Page Number of Metadata B+ Tree
0x0024    8     WAL Segment Start Offset (page number)
0x002C    8     WAL Segment Size in bytes
0x0034    4     Number of collections
0x0038    4     Checksum (CRC32C of bytes 0x0000..0x0034)  [v2: CRC32C not CRC32]
0x003C    8     Flags (64-bit, see 5.1.1)                   [v2: widened from 4B]
0x0044    8     Creation timestamp (Unix ms)
0x004C    8     Last checkpoint timestamp (Unix ms)
0x0054    16    Database UUID (128-bit random)
0x0064    64    Application identifier (UTF-8, null-padded)
0x00A4    8     Root Page Number of Columnar Segment index     [v2 NEW]
0x00AC    8     Root Page Number of Vector Segment index       [v2 NEW]
0x00B4    8     Root Page Number of CDC Log                    [v2 NEW]
0x00BC    8     Root Page Number of Security Metadata          [v2 NEW]
0x00C4    8     Root Page Number of AI/Embedding Cache         [v2 NEW]

0x0124    4     Feature Bitmap Length (N)                      [v2 NEW]
0x0128    N     Feature Bitmap (which OBE2 tags / segments used)
...       ...   Reserved for future use
0x0FE0    16    HLC state snapshot (last persisted)            [v2 NEW]
0x0FF0    16    Shadow copy: magic + version + page size (corruption detection)

12   HAS_RLS               Row-Level-Security policies present.          [v2]

14   HAS_LEARNED_INDEX     AHIT Tier-1 learned-index segments present.   [v2]

5.2 Segment Layout (v2)

Segments are page-aligned. Segment Directory at Page 1 maps type codes
to [start_page, end_page] ranges.

Code  Name                    Contents
──────────────────────────────────────────────────────────────────────
0x01  Header Segment          File header + DB metadata + coll registry.
0x02  Data Segment (row)      OBE2 documents in leaf pages + overflow.
0x03  Index Segment           AHIT v2 (ART snapshot + learned-index
                              segments + B+tree + SSTable blocks +
                              inverted index + BK-tree + autocomplete
                              + R-tree/S2 + HNSW/DiskANN graph +
                              ViewPage snapshots).
0x04  WAL Segment             Write-ahead log records.
0x05  Metadata Segment        Collection schemas, statistics, session
                              registry, idempotency cache, relation
                              defs, view defs, trigger hook registry,
                              pragma state, zstd dictionaries, relation
                              index maps, user/role registry [v2].
0x06  Blob Segment            BLOBs > 16 MB, chunked at 255 KB.
0x07  Change Stream Log       Append-only circular buffer of events.
0x08  Attached DB Index       Alias → .ovn2 path map.
0x09  Columnar Segment        [v2] ColumnChunks per (collection, field).
0x0A  Vector Segment          [v2] Per-collection HNSW/DiskANN graph(s)
                              + RaBitQ codebook(s) + SPFresh update logs.
0x0B  CDC Log                 [v2] Durable, replayable CDC events.
0x0C  Security Metadata       [v2] QE schema, RLS policies, column-level
                              key map, audit log ring buffer head/tail,
                              KMS key references, key-rotation history.
0x0D  Embedding Cache         [v2] content-hash → cached-embedding map.
0x0E  Zone Map Segment        [v2] Per-page min/max/null-count sketches.
0x0F  Learned-Index Segment   [v2] PGM++ model parameters + segments.
0x10  Audit Log Segment       [v2] OCSF-schema JSON ring buffer.

5.2.1 Columnar Segment (0x09) Detail [v2]

Per (collection, column) pair:
  ColumnChunk page (0x0A in Page Header): 64 KiB default.
  Layout:
    [32B Page Header]
    [8B  Field Id (2-byte, padded) + Physical Type tag]
    [8B  Watermark TxID (max TxID covered by this chunk)]
    [4B  Row count in chunk]
    [4B  Encoding flags: DICT|RLE|BITPACK|PLAIN|DELTA]
    [varint Dictionary length (0 if not DICT-encoded)]
    [Dictionary bytes (if DICT)]
    [Values bytes (encoded per flags)]
    [Zone Map: min (type-specific) | max | null_count | distinct_hint]
    [4B CRC32C]
  Tombstone handling:
    Rows deleted or updated-since-watermark are tracked in a "row-delete
    bitmap" stored at the end of the ColumnChunk. Readers apply the
    bitmap before returning data.
  Chunk rewriting:
    Columnar Flusher rewrites chunks when delete bitmap density > 30%.

5.2.2 Vector Segment (0x0A) Detail [v2]

Per (collection, vector-field, index-type) triple:
  - Codebook page (RaBitQ codebook; centroids, residuals).
  - Graph pages (HNSW layers OR DiskANN Vamana neighbors; 2048 neighbors
    per node stored as 16-bit IDs; compressed with vectorized delta
    encoding).
  - Update log pages (SPFresh: batched inserts/deletes applied during
    Vector Compaction).
  - Full-precision vector pages (for re-ranking top-K candidates).
  - Quantized vector pages (RaBitQ-compressed, 1 bit per dimension typical).

5.2.3 CDC Log Segment (0x0B) Detail [v2]

[IMPL NOTE] CDC Log is DIFFERENT from Change Stream Log (0x07):
  Change Stream Log: 64 MB circular buffer, best-effort, subscribers
                     that fall behind are invalidated.
  CDC Log:           Durable, monotonically-growing, explicitly sized
                     (default up to 10 GB). Designed for Debezium-
                     compatible downstream consumption with guaranteed
                     delivery via checkpointed offsets.

CDC record format:
  [8B  TxID]
  [8B  HLC timestamp]
  [4B  collection_id]
  [16B doc_id]
  [1B  operation: c(create) | u(update) | d(delete) | t(truncate) | s(schema)]
  [4B  source_partition_id (for distributed)]
  [varint pre_image_length]  [pre-image OBE2]
  [varint post_image_length] [post-image OBE2]
  [varint relation_context_length]
    [relation_id, cascade_reason, originating_tx_id] × N
  [4B CRC32C]

Retention policies (pragma cdc_retention):
  'size:10GB'        Drop oldest when exceeds size (default).
  'time:7d'          Drop records older than 7 days.
  'checkpoint'       Drop records that ALL subscribers have acknowledged.

5.2.4 Security Metadata Segment (0x0C) Detail [v2]

Contents:
  User/Role registry:
    user_id (UUID) | username | password_hash (Argon2id) | roles[] | created_at
  Role registry:
    role_name | privileges[] | inherits_from[]
  QE schema (per collection, per field):
    field_path | cipher_suite | encryption_type (eq/range/prefix/suffix/substr)
    | key_id | tag_count | index_structure_root_page
  RLS policies:
    collection | policy_name | on_operations (r/w/u/d) | policy_expression_ast
  KMS key map:
    collection | column | key_id | key_version | algorithm | kms_reference
  Key-rotation history:
    key_id | created_at | rotated_at | status (active/retiring/retired)
  Audit log ring head/tail pointers:
    ring_start_page | ring_end_page | next_seq_no

5.2.5 Embedding Cache Segment (0x0D) Detail [v2]

Content-addressed cache:
  Key:  SHA-256 of (provider | model | input_text_bytes)
  Value: embedding vector (quantized RaBitQ + full-precision fallback)
  TTL: configurable (default no expiry; LRU-evict when size exceeded).
Hit path avoids calling the external embedding provider; deterministic
across Oblivinx instances with the same cache.

5.2.6 Zone Map Segment (0x0E) Detail [v2]

Per data page (row-oriented AND columnar):
  [8B  page_number]
  [varint per-field-count]
    For each indexed scalar field:
      [2B field_id]
      [type-specific min value]
      [type-specific max value]
      [4B null_count]
      [4B total_count]
      [8B distinct_count_hll]  (HyperLogLog sketch, 64-bit count estimate)
  [4B CRC32C]
Used for predicate pushdown: optimizer checks if pred_min > page_max or
pred_max < page_min → skip entire page. This alone reduces I/O by 60-95%
on scans with selective predicates.

5.2.7 Learned-Index Segment (0x0F) Detail [v2]

Per AHIT Tier-1 segment:
  [16B segment_id]
  [8B  key_count]
  [8B  min_key_bytes_offset, max_key_bytes_offset]
  [2B  epsilon (error bound)]
  [2B  model_count (number of PGM layers)]
  For each PGM layer:
    [8B layer_size]
    [per-segment: slope (f64), intercept (f64), first_key (varint)]
  [8B pointer to raw keys (for verification)]
  [4B CRC32C]

5.3 Page Layout

Every page is exactly [page_size] bytes (default 4096).
Every page begins with a standard 32-byte Page Header.

Page Header (32 bytes):
  Offset  Size  Field
  0x00     4    Page Type:
                  0x01 = Leaf B+tree page
                  0x02 = Interior B+tree page
                  0x03 = Overflow page (large field values)
                  0x04 = WAL record page
                  0x05 = Free page
                  0x06 = BlobChunk page
                  0x07 = ChangeStreamRecord page
                  0x08 = ViewPage (materialized view snapshot)
                  0x09 = ColumnChunk page                [v2]
                  0x0A = VectorGraph page                [v2]
                  0x0B = VectorCodebook page             [v2]
                  0x0C = CDC record page                 [v2]
                  0x0D = AuditLog record page            [v2]
                  0x0E = EmbeddingCache page             [v2]
                  0x0F = ZoneMap page                    [v2]
                  0x10 = LearnedIndex page               [v2]
                  0x11 = SSTableIndex block              [v1, renumbered]
                  0x12 = BloomFilter page                [v1, renumbered]
                  0x13 = SecurityMetadata page           [v2]
                  0x14 = RelationIndex page              [v1]
  0x04     8    Page Number (0-indexed)
  0x0C     8    Transaction ID (TxID) that last modified this page
  0x14     2    Number of document slots (leaf pages only; 0 for others)
  0x16     2    Free space offset (bytes from start of payload area)
  0x18     4    Right sibling page number (B+ tree leaf chain; 0 = last leaf)
  0x1C     4    CRC32C checksum of page bytes 0x20 to end (v2 uses CRC32C, not CRC32)

[IMPL NOTE] v2 uses CRC32C (Castagnoli polynomial) instead of CRC32.
CRC32C has hardware acceleration on x86 (SSE4.2 CRC32 instruction) and
ARM64 (CRC32CX instruction) — 10-20× faster than software CRC32.

5.4 OBE2 Binary Encoding [v2]

[IMPL NOTE] OBE2 retains v1's Document Structure and Field Structure
verbatim, and extends the Type Tag table. v1 documents remain readable.

OBE2 Document Structure (unchanged from v1):
  [4 bytes]   Document total length
  [16 bytes]  Document ID
  [8 bytes]   Transaction ID
  [1 byte]    Tombstone flag
  [varint]    Number of fields
  [Fields...]

OBE2 Field Structure (unchanged from v1):
  [varint]    Field name length (>0: follows UTF-8 name; ==0: follows 2-byte Field ID)
  [bytes]     Name or Field ID
  [1 byte]    Type tag
  [...]       Value bytes

Type Tag Table (full):
  Tag   | Type                 | Encoding
  ------+----------------------+---------------------------------------------------
  0x01  | Null                 | No value bytes
  0x02  | Boolean True         | No value bytes
  0x03  | Boolean False        | No value bytes
  0x04  | Int32                | 4B LE (i32)
  0x05  | Int64                | 8B LE (i64)
  0x06  | Float64              | 8B IEEE 754 (f64)
  0x07  | String               | varint byte_length + UTF-8 bytes
  0x08  | Binary               | varint byte_length + raw bytes
  0x09  | Document (nested)    | Recursive OBE2 document (with own length prefix)
  0x0A  | Array                | varint element_count + elements (each w/ type tag)
  0x0B  | Timestamp            | 8B Unix epoch ms (u64)
  0x0C  | ObjectId             | 12B: 4B unix time + 5B machine ID + 3B counter
  0x0D  | Decimal128           | 16B IEEE 754 decimal128
  0x0E  | Varint               | 1-9B LEB128 signed integer
  0x0F  | BlobRef              | 16B blob_id + 8B total_size
  0x10  | GeoPoint             | 16B: 8B lat (f64) + 8B lng (f64)
  0x11  | GeoJSON              | varint byte_length + GeoJSON UTF-8
  0x12  | RelRef                                                             [v1]
  0x13  | Vector                                                             [v2]
        |   varint dim + 1B element_type (0x01=f32 | 0x02=f16 | 0x03=i8Q) +
        |   dim * sizeof(element_type) bytes
  0x14  | EncryptedField                                                     [v2]
        |   1B scheme (0x01=QE-eq | 0x02=QE-range | 0x03=QE-prefix |
        |              0x04=QE-suffix | 0x05=QE-substring | 0x06=AEAD-only) +
        |   16B key_id + varint nonce_len + nonce + varint ciphertext_len +
        |   ciphertext + varint tag_count + tag_count × 16B tags +
        |   16B auth_tag (AES-GCM-SIV)
  0x15  | BigDecimal                                                         [v2]
        |   varint scale + varint coefficient_byte_length + coefficient_bytes
  0x16  | DateTimeTZ                                                         [v2]
        |   8B Unix ms + 2B tz_offset_minutes (signed)
  0x17  | Duration                                                           [v2]
        |   8B signed nanoseconds (i64)
  0x18  | JSONPatch                                                          [v2]
        |   varint op_count + op_count × JSONPatch operation (RFC 6902)
  0x19  | EmbeddingRef                                                       [v2]
        |   32B content-hash pointing into Embedding Cache
  0x1A  | TaggedUnion                                                        [v2]
        |   1B variant_index + variant value (per-union schema)

[IMPL NOTE] Forward compatibility: when a v1 reader encounters tag ≥ 0x13
it MUST treat the field as opaque and skip it using the encoded length.
Oblivinx v2 mandates that every new tag include an explicit length prefix.
Old v1.2 code will NOT correctly skip 0x13+ without this — use the v2
minimum reader version requirement in the header's Feature Bitmap.

5.5 Columnar Chunk Format (ColumnChunk page 0x0A) [v2 NEW]

See Section 5.2.1 for the page layout. Here we specify encodings:

Encoding modes (chosen per-chunk by the Columnar Flusher):

DICT (dictionary) — best for low-cardinality strings / enums:
  [varint num_distinct_values]
  For each distinct: [varint value_length][value_bytes]
  Then: [bit-packed indices of N rows, log2(num_distinct) bits each]

RLE (run-length) — best for sorted or clustered columns:
  Series of (run_length: varint, value_bytes) pairs.

BITPACK — best for small-range integer columns:
  [varint min_value][varint max_value][bit_width: u8]
  [packed deltas of N rows, bit_width bits each]
  Actual value = min_value + delta.

DELTA — best for monotonic columns (timestamps, IDs):
  [base_value: varint][N-1 deltas: varint each]

PLAIN — fallback: raw values with 0x00 null mask.

Each chunk stores encoding_flags (u16) with bitmask of modes actually used
(can combine, e.g., DICT+BITPACK for the indices into a dictionary).

5.6 Compression

v2 extends v1's two-level compression:

Page-level (Data, Index, Columnar):
  Options: 'none' | 'lz4' | 'zstd-1' | 'zstd-3' (default) | 'zstd-19' | 'zstd-dict'
  LZ4: decode > 4 GB/s; ratio ~2-2.5×.
  Zstd-3: decode ~1.5 GB/s; ratio ~3-4×. New v2 default.
  Zstd-dict: shared dictionary, 2-3× better ratio for small records.
  Compression is transparent to readers: PageHeader.CompressionMode byte
  at offset 0x20 indicates raw vs. compressed.

Document-level (for docs > 512 B, independent of page compression):
  Zstd with optional per-collection trained dictionary.
  Dictionary auto-trained from the first 10,000 documents; stored in
  Metadata Segment; rebuilt lazily when collection grows 10× since last
  training.

Bit-packing (within ColumnChunks only):
  Minimum bit width for integer ranges. E.g., age 0..150 needs 8 bits.

WAL is NEVER compressed (must be readable after partial crashes).
Security metadata is NEVER compressed (encrypted, so entropy is high).

5.7 Encryption at Rest [v2]

Collection-level AES-256-GCM-SIV (misuse-resistant; same nonce is safe):
  DEK (Data Encryption Key): 256-bit, per collection.
  KEK (Key Encryption Key): held in external KMS (never written in plaintext).
  DEK is wrapped by KEK; wrapped DEK stored in Security Metadata Segment.

Page encryption:
  Every data / index / columnar page (NOT WAL — see below) can be
  encrypted. Encrypted page layout:
    [32B Page Header (clear; includes CRC32C of ENCRYPTED payload)]
    [12B AES-GCM-SIV nonce]
    [page_size - 32 - 12 - 16 bytes of ciphertext]
    [16B AES-GCM-SIV authentication tag]
  Nonce derivation: HKDF(DEK, page_number || TxID)
  This guarantees nonce uniqueness without a counter register.

WAL encryption:
  WAL records are encrypted record-by-record (not page-by-page) using
  the same DEK + HKDF nonce derivation based on (record_sequence, TxID).

Key rotation:
  db.rotateKey(collection) generates a new DEK, re-wraps old pages
  lazily during compaction. A single file may have pages encrypted with
  two DEK versions during rotation; each page's encryption header
  includes a 1-byte key_version.

Authenticated additional data (AAD):
  Every encryption includes AAD of (page_number || collection_id ||
  file_UUID) so that pages cannot be relocated between files or
  collections without detection.

[IMPL NOTE] For embedded / MCU profiles without KMS, support a
"passphrase" mode: KEK = Argon2id(passphrase, salt from header 0x00F4).
Use ONLY in single-user scenarios; not suitable for multi-tenant.

Section 6 — Write Path v2.0 (io_uring + BEGIN CONCURRENT)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The v2 write path is re-architected around three principles:
  1. Never block the client on I/O longer than necessary (async all the way).
  2. Batch everything possible (group commit, vector buffer writes).
  3. Allow multiple concurrent writers with MVCC (BEGIN CONCURRENT).

6.1 Write Pipeline Overview (v2)

  Client Thread N                               Background Thread(s)
  ───────────────                               ─────────────────────
  [1]  Client Request
        │
  [2]  MQL Parse + AST
        │
  [3]  Security: auth check + RLS policy eval
        │
  [4]  Trigger Executor (before*) — may modify or reject
        │
  [5]  Schema Validator
        │
  [6]  Relational FK check (INSERT only, strict/soft mode)
        │
  [7]  QE encrypt + tag-generate for encrypted fields
        │
  [8]  MVCC Snapshot → HLC TxID assignment
        │
  [9]  OBE2 Serialize (SIMD) → thread-local buffer
        │
  [10] WAL Writer submit → io_uring SQE → group commit ─→ WAL fsync (batched)
        │
  [11] MemTable Insert (lock-free concurrent skip list)
        │                                      ─→ [12] MemTable Flush → L0 SSTable
  [13] AHIT v2 Index Update (Tier 0 + MemTable;
        Tier 1 learned-index updated lazily)
        │                                      ─→ [14] Compaction (L0 → B+; SSTable merge)
        │                                      ─→ [15] Columnar Flusher (append to ColumnChunks)
        │                                      ─→ [16] Vector Updater (SPFresh incremental)
        │                                      ─→ [17] Zone Map Updater
  [18] Conflict Check (BEGIN CONCURRENT mode)
        │
  [19] Referential commit check (cascade/restrict)
        │
  [20] Audit Log append (OCSF JSON)            ─→ [21] Audit Log Flusher
        │
  [22] Return Ack to Client                    ─→ [23] Change Stream Emitter
                                                ─→ [24] CDC Log Append (durable)
                                                ─→ [25] after* Trigger Exec
                                                ─→ [26] View IVM Refresh
                                                ─→ [27] Unified Expiry Worker

6.2 Step-by-Step Write Path Detail

Step 1 — Client Request
  Same as v1. The write queue is now a multi-ring MPMC with one ring per
  client thread, matching the io_uring ring-per-thread model.

Step 2 — MQL Parsing
  Bounded by max_document_size (default 16 MB; min 4 KB).

Step 3 — Security: Auth + RLS [v2]
  If user authentication is enabled, verify the session token.
  If RLS policies apply to the target collection, evaluate the policy
  expression against the document + session context. Rejection → RLS error.

Step 4 — Trigger Execution (before*)
  Unchanged from v1. beforeInsert/beforeUpdate/beforeDelete can modify
  or reject the write. Timeout default 500 ms.

Step 5 — Document Validation
  Unchanged from v1.

Step 6 — Relational FK Check (INSERT only, non-strict modes)
  Unchanged from v1.

Step 7 — QE Encryption [v2]
  For each field declared as encrypted (via schema):
    - Generate encryption tags using the QE tag-generation algorithm
      for the configured scheme (eq, range, prefix, suffix, substr).
    - Encrypt the value with AES-256-GCM-SIV using the field's DEK.
    - Replace the plaintext with an OBE2 EncryptedField (tag 0x14).
  [IMPL NOTE] Tags are deterministic but randomized via a keyed hash.
  The server can match tags without seeing plaintext; the cryptographic
  details follow the MongoDB QE construction (Cryptography Research Group),
  widely reviewed. Do not invent a new scheme.

Step 8 — MVCC Snapshot & TxID
  HLC-based TxID generation (48b physical ms + 16b logical counter).
  BEGIN CONCURRENT mode: the writer's snapshot is captured here, independent
  of any other active writers.

Step 9 — OBE2 Serialization
  SIMD-accelerated serializer; AVX-512 if available, else NEON, else scalar.

Step 10 — WAL Write (Group Commit by Default)
  WAL record format:
    [4B  record_type]
    [8B  TxID (HLC)]
    [4B  collection_id]
    [16B lsid (zeroed if not retryable)]
    [8B  txnNumber]
    [1B  savepoint_depth]
    [2B  flags: DURABLE_SAVEPOINT | CONCURRENT_WRITER | ENCRYPTED | ...]
    [varint document_length]
    [document_bytes (OBE2, may be encrypted)]
    [4B  CRC32C]

  v2 group commit (enabled by default; pragma group_commit = true):
    1. Writer submits WAL record to an in-memory pending-WAL ring.
    2. WAL Writer thread batches up to max_group_commit_bytes (default 1 MB)
       or max_group_commit_us (default 200 μs), whichever first.
    3. Single io_uring submission with fixed-buffer I/O + fsync.
    4. All writers in the batch are woken once the fsync completes.

  v2 durability levels:
    'D0'               — No fsync. Fastest, data loss possible on crash.
    'D1'  (default)    — Group commit fsync (amortized).
    'D1-strict'        — Per-record fsync (v1 D1 behavior).
    'D2'               — No fsync, but sync to OS page cache (journal_mode=none).
    'D3'               — fsync + fdatasync on a separate disk replica (dist mode).

Step 11 — MemTable Insert
  Lock-free concurrent skip list (crossbeam-skiplist). O(log N).
  Partitioned by collection_id. MemTable freezes at 64 MB by default.

Step 12 — MemTable Flush (Background)
  Unchanged logic from v1 except the flush writes a Bloom filter AND
  updates the Zone Map Segment for the new SSTable.

Step 13 — AHIT v2 Index Update
  For each indexed field:
    - Insert into Tier 0 (ART) if key satisfies ART's capacity.
    - Insert into MemTable (always).
    - Tier 1 (learned index) is NOT updated per-write; rebuilt by the
      AHIT Promoter during compaction when stability threshold met.
  For vector fields:
    - Append to SPFresh update log (durable).
    - Incremental merge during Vector Compaction.
  For full-text fields:
    - Tokenize, update inverted index MemTable.
  For geospatial fields:
    - Encode to S2 cell tokens (2dsphere) or R-tree entry (2d).

Step 14 — Compaction (Background)
  v2 compaction coordinates with AHIT Promoter:
    1. Merge L0 SSTables → merged sorted run.
    2. Merge into permanent B+ tree.
    3. If a merged subtree's access counter qualifies, AHIT Promoter
       bulk-loads a learned-index segment (Tier 1) over it.

Step 15 — Columnar Flusher [v2 NEW]
  Reads committed WAL records, extracts values for columnar-mirrored
  fields, appends them to the in-memory ColumnChunk buffers. When a
  buffer reaches its target size (64 KiB per chunk), encodes it
  (DICT/RLE/BITPACK/DELTA/PLAIN selected per data shape) and writes
  to a new ColumnChunk page. Updates the chunk's zone map.

Step 16 — Vector Updater [v2]
  For HNSW vectors: insert into HNSW graph (incremental, O(log N) expected).
  For DiskANN vectors: route to SPFresh update log; merged during
  Vector Compaction (every 100K updates or 5 minutes, whichever first).
  RaBitQ: quantize during insert; store both full-precision and
  quantized forms.

Step 17 — Zone Map Update [v2 NEW]
  For each page modified by the commit, update its zone map (min, max,
  null_count, distinct_count_hll for each indexed scalar field).

Step 18 — Conflict Check (BEGIN CONCURRENT only) [v2]
  v2 default: optimistic concurrency. Writer's read set is compared
  against committed writes since snapshot time. If any read-set key
  was written by a higher-TxID commit, abort with WRITE_CONFLICT.
  Automatic retry handled by the retry-shim (default max_retries = 3).

Step 19 — Referential Integrity Commit Check
  Unchanged from v1.

Step 20 — Audit Log Append [v2 NEW]
  If auditing is enabled (pragma audit_log = true), construct an OCSF
  JSON event:
    {
      "activity_id": 1 (create) | 2 (update) | 3 (delete) | ...,
      "category_uid": 6 (application activity),
      "class_uid": 6003 (database activity),
      "time": <HLC ms>,
      "severity_id": 1 (informational),
      "actor": { "user": { "name": "<username>", "uid": "<user_id>" } },
      "resource": { "name": "<collection>", "type": "Database.Collection" },
      "operation": { "name": "insert|update|delete|...", "opcode": <int> },
      "status": "success|failure",
      "message": "<summary>",
      "metadata": { "tx_id": "<HLC>", "doc_id": "<UUID>" }
    }
  Append to in-memory audit-log ring buffer. The Audit Log Flusher
  (background worker) writes to the Audit Log Segment.

Step 21 — Audit Log Flusher (Background) [v2]
  Flushes in-memory ring to Audit Log Segment pages.
  Optionally forwards events to an external SIEM (Splunk, Elasticsearch,
  Datadog) via a configured webhook / syslog / OTLP exporter.

Step 22 — Client Acknowledgment
  Sent as soon as WAL fsync completes (Step 10). Ack includes:
  {
    insertedId: '<UUID>',
    txnId:     '<HLC>',
    hlc:       '<HLC readable>',
    persisted: true,
    durability: 'D1' | ...
  }

Step 23 — Change Stream Emitter (Async)
  Unchanged from v1 semantics; now also supports hybrid events when
  a write affects both row and columnar mirrors.

Step 24 — CDC Log Append [v2 NEW]
  Writes a durable CDC record to Segment 0x0B. This persists across
  restarts and is consumable by downstream systems.

Step 25 — After-Write Trigger Execution
  Unchanged from v1.

Step 26 — View IVM Refresh [v2 REPLACES v1 on-write refresh]
  DBSP-based incremental view maintenance (Section 17). The refresh is
  incremental for every aggregation stage v1 supported as full-refresh-only.

Step 27 — Unified Expiry Worker [v2 REPLACES v1 TTL Reaper]
  Combines:
    - TTL-index driven expiry (v1 TTL Reaper).
    - CDC retention cleanup.
    - Change Stream Log rollover.
    - Embedding Cache LRU eviction.
    - GC horizon-TxID version purging.
  Single pass through all these data structures minimizes wakeups.

6.3 Failure Recovery v2

On process restart:
  1. Read Page 0. Verify magic (0x4F564E32) and header HMAC.
  2. Check WAL_ACTIVE flag.
  3. Decrypt header's HLC snapshot; use max(persisted_hlc, wall_clock) as
     starting HLC (preserves monotonicity across crashes).
  4. Scan WAL segment forward from last checkpoint TxID.
     - Decrypt WAL records (if ENCRYPTED flag set).
     - Verify CRC32C of each record.
     - COMMIT markers → re-apply to MemTable.
     - Uncommitted / partial → discard.
     - SVPT_BEGIN without COMMIT → discard (unless durable_savepoints).
  5. Replay Columnar Flusher state (ColumnChunks past watermark = truncated).
  6. Replay Vector SPFresh update log.
  7. Replay Zone Maps (recompute from recovered data).
  8. Rebuild AHIT Tier 0 (ART from MemTable + recent SSTables).
  9. AHIT Tier 1 (learned index): re-verify against persisted segments;
     rebuild if CRC mismatch.
  10. Flush recovered MemTable → L0 SSTable → trigger compaction.
  11. Write new checkpoint to Page 0 (atomic via shadow copy at 0x0FF0).
  12. Clear WAL_ACTIVE flag.

Recovery time typically < 2 s with default 60 s checkpoint interval.

7.1 Read Pipeline Overview (v2)

  [1]  Client Query

        │

7.2 Query Optimizer — Enhanced Plan Costing

v2 cost formula:
  Cost(plan) = (N_pages_read × PAGE_COST) × (1 - ZoneMapPruneRatio)
             + (N_docs_decoded × DECODE_COST)
             + (N_comparisons × CMP_COST)
             + (N_vector_dists × VDIST_COST)
             + (N_encrypted_fields_decrypted × DECRYPT_COST)
             + (if_columnar: N_columns_touched × COL_BATCH_COST)

All cost constants are hardware-calibrated during db.optimize() using a
short micro-benchmark at first call (results cached in Metadata Segment).

Plan types in preference order:
  1. AHIT Tier-0 (ART) point lookup:        O(k) — k = key length
  2. AHIT Tier-1 (Learned) point lookup:    O(log log N) + O(ε)
  3. Indexed point lookup (Tier 2 B+):      O(log N)
  4. Indexed range scan:                     O(log N + K)
  5. Covered index scan:                     O(log N + K), no doc fetch
  6. Columnar scan (analytical):             O(K) with SIMD batch decode
  7. Zone-map-pruned collection scan:        O(N × (1 - ZoneMapPrune))
  8. Geospatial index scan:                  O(log N + K)
  9. Vector ANN scan (HNSW/DiskANN):         O(log N) expected
  10. Full collection scan:                   O(N) — rarely selected

7.3 AHIT v2 Read Path Detail

For a point lookup on indexed field F = v:
  t0: ART.get(v) → hit? return doc_id.                     [Tier 0]
  t1: Learned.find(v) → predicted_pos; verify in range.    [Tier 1]
       On hit return; on miss (ε-miss), fall through.
  t2: MemTable.find(v) → hit? return.                      [Memory LSM]
  t3: For each L0 SSTable, check Bloom filter.             [LSM]
       If possibly-present, read SSTable Index block, binary search.
  t4: Cold Zone B+ tree search.                            [Tier 2]

Typical depth for 1B document collection:
  t0:  20-50 ns (L1/L2 cache resident)
  t1:  60-120 ns (L3 / DRAM resident, 2-3 ns per PGM layer)
  t2:  200-500 ns (MemTable skip list)
  t3:  50 ns per Bloom check + 10-50 μs per SSTable binary search
  t4:  10-200 μs (3-4 page reads, most in buffer pool)

7.4 Zone Map Pruning [v2 NEW]

For a scan with predicate P = "field F > v":
  1. Load the Zone Map Segment (kept mostly in memory; ~100 KB per
     100 MB of data).
  2. For each page P_i in the target range:
     If P_i.zone_map[F].max ≤ v → skip P_i entirely.
     If P_i.zone_map[F].min > v → read all rows (no predicate check needed).
  3. For non-skipped pages, batch-read them via io_uring (async).
  4. Decode and apply predicate in SIMD loop.

Typical pruning ratio: 60-95% on selective scans.
Combined with columnar routing, full-scan queries often hit 10-50× speedup.

7.5 View Resolution

Unchanged semantics from v1. v2 materialized views are now incrementally
maintained via DBSP (Section 17) — reads always see up-to-date results
if the view is 'on_write' mode; 'scheduled' and 'manual' modes behave
as in v1.

Section 8 — Indexing System v2.0
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

All scalar indexes are AHIT v2 — three-tier (ART → Learned → B+/SSTable).
Specialized index types (FTS, geospatial, vector, compound SCO) are
built on top of AHIT's persistence layer.

8.1 Primary Document Index

Unchanged from v1 semantics. _id is indexed via AHIT v2 (all three
tiers eligible). For collections with high _id locality (e.g., UUID v7
or time-ordered IDs), Tier 1 learned index yields O(log log N) point
lookup with tens of bytes per million keys.

8.2 Secondary Indexes

All v1 options preserved: TTL, Sparse, Unique, Partial, Hidden, Wildcard,
Collation, Background.

v2 additions:
  IVF (Inverted File) for low-dim float vectors (experimental alternative
  to HNSW/DiskANN for moderate N).
  Computed-field index: index on the result of a deterministic MQL
  expression (e.g., { $toLower: '$email' }).
    await db.createIndex('users', { $expr: { $toLower: '$email' } });

8.3 Compound Indexes with Skip-Column Optimization (SCO)

Unchanged from v1. v2 extends SCO with a Bloom fingerprint per field
(in addition to the 64-bit hash fingerprint), further reducing false
positives for partial-key queries.

8.4 Full-Text Index (Tantivy-Class)

v2 replaces v1's simple inverted index with a Tantivy-class design:
  - Inverted index with skip lists for fast posting-list intersection.
  - BM25 relevance scoring (replaces v1's TF-IDF) as default.
  - BK-tree for fuzzy (Levenshtein distance ≤ 2).
  - Compressed radix trie for autocomplete.
  - Per-field language-specific analyzers (English Porter, Snowball for
    15+ European languages, CJK bigram, Indonesian stemming).
  - Highlight support (via position-tracked postings from v1).
  - Query operators: $text / $search with BM25, $fuzzy, $phrase,
    $proximity (NEAR), $fieldBoost.

API (mostly compatible with v1):
  await db.createIndex('articles', { title: 'text', content: 'text' }, {
    fuzzy:        { maxEdits: 1, prefixLength: 3 },
    autocomplete: true,
    analyzer:     'english',       // v2 NEW
    fieldBoosts:  { title: 2.0 },  // v2 NEW
    stopwords:    'default'         // v2 NEW: 'default' | 'none' | [list]
  });

8.5 Vector Index v2 (DiskANN + RaBitQ + SPFresh) [v2]

v2 supports three vector index backends, selected at index creation time:

  'hnsw'        In-memory HNSW graph. Best for N ≤ 10M and recall ≥ 0.95.
                Size: ~100-200 bytes per vector + full-precision.
  'diskann'     Disk-resident Vamana graph. Best for N ≥ 10M.
                Size: ~50 bytes per vector in memory + rest on disk.
  'lsm_vec'     LSM-tree-backed graph. Best for dynamic workloads with
                billions of vectors and heavy updates (2025 state of art).

RaBitQ quantization is applied to all three (configurable; can disable
for absolute accuracy):
  - 1-bit-per-dimension quantization by default.
  - Asymptotically optimal error bounds (SIGMOD 2025).
  - Re-rank top-K with full-precision vectors for final ordering.

SPFresh incremental updates: for diskann and lsm_vec, inserts and
deletes go to an update log, merged into the graph during Vector
Compaction (every 100K ops or 5 min).

API:
  await db.createVectorIndex('docs', {
    field:      'embedding',
    dimensions: 1536,
    metric:     'cosine',              // 'cosine' | 'l2' | 'inner_product'
    backend:    'diskann',              // 'hnsw' | 'diskann' | 'lsm_vec'
    quantization: { method: 'rabitq', bits: 1 }, // or false
    ef:         128,                    // for HNSW build/search
    m:          16,                     // for HNSW
    vamanaAlpha: 1.2,                   // for DiskANN
    maxDegree:  64                      // for DiskANN/LSM-VEC
  });

Query API:
  const similar = await db.vectorSearch('docs', {
    vector: queryEmbedding,
    topK:   20,
    filter: { category: 'article', language: 'en' },  // filtered ANN [v2]
    ef:     128,
    rerank: true   // use full-precision for final top-K ordering
  });

Filtered ANN (post-filter + pre-filter hybrid):
  For cheap filters (expected selectivity ≥ 20%): post-filter after ANN
  retrieves 3×topK candidates.
  For selective filters (< 20%): DiskANN's filtered-search mode uses
  attribute-aware graph traversal. Requires compound vector index.

8.6 Learned Index Tier (PGM++ / LMG) [v2 NEW]

(See Section 4.2 for the tier integration.)

Per-segment learned model:
  Piecewise Linear Approximation (PLA) bounded by epsilon (default 64).
  For N keys per segment, expected segments = O(N / ε²).
  Height of PGM hierarchy = O(log log N).

Segment rebuild policy:
  - On bulk-load from Tier 2 B+ subtree: fit segments single-pass in O(N).
  - On insert: if the new key falls within an existing segment's ε
    bound, insert into a "gap" slot (LMG gapped array). If no gap,
    split the segment.
  - On drift detection (fallback rate > 10%): mark segment dirty;
    AHIT Promoter rebuilds during next compaction window.

Memory: 1-8 bytes per key typical; 1000× smaller than a B+ tree.

Safety: every lookup verifies the predicted position against the actual
key (binary search within ε window). Wrong predictions cost at most
log(2ε) additional comparisons.

Rejection criteria (PGM++ won't be used if any of):
  - Data is non-sorted or highly skewed (ART is better).
  - Keys are adversarial (poisoning resistance: v2 uses the
    distribution-aware PGM variant; monitors poisoning metric).

8.7 Geospatial Index (2dsphere / 2d)

Unchanged from v1. v2 adds H3 (Uber's hexagonal hierarchical spatial
index) as an alternative to S2:
  await db.createIndex('stores', { location: 'h3' }, { resolution: 9 });

8.8 Hybrid Search (Reciprocal Rank Fusion) [v2 NEW]

Fuses results from multiple index types. The query engine runs each
branch in parallel, then combines via RRF:
  RRF_score(doc) = Σ_branches (1 / (k + rank_in_branch(doc)))
where k is a constant (default 60, per Microsoft's RRF paper).

API:
  const results = await db.hybridSearch('docs', {
    query: {
      text:   'earthquake retrofitting',
      vector: queryEmbedding,
      fuzzy:  true
    },
    weights: { text: 1.0, vector: 1.0, fuzzy: 0.5 },
    topK:    20,
    filter:  { category: 'engineering' }
  });

This one call runs:
  - $text search on full-text index (BM25)
  - $vectorSearch on vector index (DiskANN/HNSW/LSM-VEC)
  - $text with $fuzzy (BK-tree)
All three branches in parallel, results fused via RRF.

8.9 Zone Maps and Min/Max Sketches [v2 NEW]

(See Section 5.2.6 for the on-disk layout.)

Maintained automatically per page when the Statistics Collector runs
(every 60 s) AND on every write (incremental update to the affected
page's zone map).

Use cases:
  1. Predicate pushdown during full scans → 60-95% page skip rates.
  2. Query planner cost estimation → better selectivity estimates
     without full histograms.
  3. Columnar chunk pruning → skip entire ColumnChunks.

Per-page overhead: 8-24 bytes per indexed scalar field. Well worth it.

8.10 Advanced Index Options

All v1 options retained (unique, sparse, partialFilterExpression, hidden,
collation, background, expireAfterSeconds).

v2 NEW options:
  Option                   | Type    | Description
  -------------------------|---------|---------------------------------------------
  learnedIndexHint         | boolean | Force/disable Learned-Index tier.
  quantization             | object  | Vector quantization config.
  shardKeyCompatible       | boolean | Index is shard-compatible for dist mode.
  encrypted                | boolean | Index built on encrypted field (QE).
  computed                 | object  | Computed field index expression.
  rlsAware                 | boolean | Index respects RLS policies (filtered scan).
  columnar                 | boolean | Index pairs with columnar mirror.

Examples:
  // Encrypted equality-queryable index on email
  await db.createIndex('users', { email: 1 }, {
    unique: true,
    encrypted: true,  // backed by QE tags
    queryable: 'eq'   // 'eq' | 'range' | 'prefix' | 'suffix' | 'substring'
  });

  // Computed field (case-insensitive email)
  await db.createIndex('users', { $expr: { $toLower: '$email' } }, { unique: true });

  // Force AHIT Tier 0 inclusion for a hot field
  await db.createIndex('users', { userId: 1 }, { tier0Pin: true });

Section 9 — Memory Management v2.0
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

9.1 Buffer Pool (ARC Default, CLOCK-Pro Embedded)

v2 replaces v1's segmented-LRU with Adaptive Replacement Cache (ARC):
  ARC maintains two LRU lists:
    T1 — recently accessed pages (cache hits on first access).
    T2 — frequently accessed pages (second+ access).
  Plus two ghost lists (B1, B2) tracking recently evicted entries.
  Adaptively tunes the split between T1 and T2 based on observed hit
  patterns. Outperforms LRU and LFU on most real workloads.

For embedded/MCU targets where ARC's bookkeeping overhead is too high,
v2 falls back to CLOCK-Pro (approximate LRU with O(1) operations,
LIRS-aware).

Default sizes:
  Full profile:     min(256 MB, 25% of available RAM).
  Embedded profile: min(8 MB,   25% of available RAM).
  Edge/WASM:        min(64 MB,  50% of available V8 heap).

Registered buffers (io_uring): the Buffer Pool pages are registered
with io_uring once at startup, enabling zero-copy DMA between NVMe
and pool pages.

9.2 Working Set Estimator v2

Same as v1 (sparse bitmap, 60 s rolling window) but with hysteresis:
  - Grow trigger: working set > 80% for 60 s straight (not momentary).
  - Shrink trigger: working set < 30% for 180 s straight.
  - Minimum pool size: never shrink below 32 MB.
  - Maximum pool size: bounded by pragma max_buffer_pool_size.

9.3 MemTable Memory

Per-collection concurrent skip list. v2 changes:
  - Default size: 64 MB (unchanged).
  - Lock-free inserts via epoch-based reclamation (was hazard pointers).
  - Frozen MemTables (awaiting flush) now count against pool budget,
    not separately tracked.

9.4 AHIT Tier-0 Slab Allocator [v2 renamed from Hot Zone]

Dedicated region for ART nodes. Slab sizes: 48B (Node4), 152B (Node16),
752B (Node48), 2060B (Node256), 64B (Leaf). Tuned to ART's canonical
layout.

9.5 Columnar Cache [v2 NEW]

Separate from Buffer Pool. Sized to min(64 MB, 10% RAM). Caches
decoded batches of ColumnChunks for hot analytical queries. Eviction:
LFU (analytical queries tend to re-scan the same columns).

9.6 PMEM / CXL Memory Tier [v2 NEW, OPTIONAL]

On systems with CXL-attached memory (CXL 2.0+), pragma cxl_tier_gb = N
reserves N GiB of CXL memory as a second-tier Buffer Pool. Pages evicted
from DRAM Buffer Pool fall through to CXL (still accessible in tens of
ns, ~2-4× DRAM latency) before hitting disk.

9.7 Garbage Collection and Version Purging

Unchanged from v1 semantics. v2 adds:
  - HLC-aware horizon TxID (min HLC across local + replicated snapshots).
  - Encrypted-document GC: old versions encrypted with retired keys are
    either re-encrypted with current key or purged, per
    pragma retire_old_keys (default 'purge').

9.8 Anti-Fragmentation

Unchanged mechanisms (slab, fixed-page buffer slots, periodic defrag).
v2 adds:
  - io_uring-batched defragmenter: reads 8 consecutive dirty pages at
    once, rewrites them contiguously. 4× faster than v1's one-page-at-a-time.

Section 10 — Concurrency Control v2.0
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

10.1 MVCC Implementation

Each write creates a new version stamped with an HLC TxID (single-node
mode still uses monotonic u64; distributed mode always HLC).

Version chain: linked list of OBE2-encoded versions, sorted by TxID
descending. Primary B+ tree points to HEAD of the version chain.

Reader walk: traverse chain to find first version with TxID ≤ snapshot
AND tombstone == 0x00. O(V), V typically 1-3.

10.2 Snapshot Isolation + SSI

Default: Snapshot Isolation (SI).
  - No dirty reads.
  - No non-repeatable reads.
  - Phantoms prevented via predicate locking on AHIT ranges.

Optional: Serializable Snapshot Isolation (SSI, Cahill).
  - Detects and aborts dangerous read-write anti-dependencies.
  - Enable: await db.setIsolationLevel('serializable').

v2 SSI extension: cross-ATTACH SSI. When a transaction spans multiple
attached .ovn2 files (cross_db_transactions pragma), the SSI detector
coordinates across files via a shared anti-dependency graph.

10.3 BEGIN CONCURRENT (Multi-Writer MVCC) [v2 NEW]

By default in v2, multiple writers proceed concurrently:
  - Each writer captures its own MVCC snapshot at BEGIN.
  - Writes are queued to per-writer WAL ring buffers.
  - Group commit batches across writers.
  - At commit time, each writer's read set is validated against
    committed writes since snapshot.
  - Lower-TxID writer wins on conflict; higher-TxID writer retries.

Throughput characteristic: on an 8-core machine with typical OLTP
workload, BEGIN CONCURRENT delivers 4-6× the throughput of single-writer
mode, with 0.1-0.5% retry rate.

Opt-out: pragma concurrent_writes = false (uses v1 single-writer WAL).

10.4 Hybrid Logical Clocks (HLC) [v2 NEW]

HLC timestamp format: 64 bits = 48 bits physical ms + 16 bits logical counter.
  On event: hlc = max(last_hlc_physical, wall_clock_ms, recv_hlc_physical)
            if same physical → logical = max(last, recv) + 1
            else logical = 0

Properties:
  - Monotonic across all events on all nodes.
  - Preserves causal ordering (send → recv).
  - Close to wall-clock time (within tens of ms drift).
  - No TrueTime hardware requirement.

Applications in Oblivinx v2:
  - TxID assignment in distributed mode.
  - Change Stream resume tokens (node-agnostic).
  - CDC record ordering across nodes.
  - HLC snapshot persisted in header for crash recovery continuity.

10.5 Deterministic Transactions [v2 NEW, OPT-IN]

For cross-shard transactions where coordination overhead of 2PC is
unacceptable, v2 offers Calvin-style deterministic execution:
  - Client sends full transaction (all reads + writes) to a Sequencer.
  - Sequencer assigns a global order (HLC-based).
  - All shards execute transactions in the same order.
  - No locks, no 2PC. Aborts only on semantic failures (e.g., assertion).

Limitation: transactions must be single-shot (no interactive query-then-commit).

Enable: await db.beginTransaction({ mode: 'deterministic' }).

10.6 Lock-Free Read Strategy

Unchanged from v1. v2 uses epoch-based reclamation (EBR) everywhere
(was hazard pointers in v1).

10.7 Write Conflict Resolution

Unchanged from v1: higher-TxID wins, lower-TxID retries with new snapshot.

10.8 Retryable Writes

Unchanged from v1 in semantics. v2 default max retries raised from 3
to 8 (BEGIN CONCURRENT has higher conflict incidence, benefits from more
retries).

10.9 Savepoints (Now Optionally Durable) [v2]

pragma durable_savepoints:
  false (default) — v1 behavior: in-memory, lost on crash.
  true            — SVPT_BEGIN and SVPT_ROLLBACK records written to WAL.
                    On crash, recovery can resume at last savepoint
                    rather than discarding the entire transaction.

API unchanged from v1.

Section 11 — Background Systems v2.0
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

All workers run at configurable priority levels and are throttled to
max 10% I/O budget when client load > 90% of system capacity.

Change Stream Emitter Detail: unchanged from v1.

Unified Expiry Worker Detail (v2):
  - TTL pass: scan TTL indexes; delete time_value ≤ (now - expireAfterSeconds).
  - CDC retention pass: drop CDC records per retention policy.
  - Change Stream log rollover: advance circular buffer head.
  - Embedding Cache LRU eviction: free least-recently-used content hashes.
  - GC version pass: purge versions beyond horizon TxID.
  All passes in one wake-up minimizes context switching.

View Refresher (IVM) Detail [v2]:
  Uses DBSP (differential dataflow) primitives (Section 17). For every
  aggregation stage, the refresher applies the incremental delta from
  the source collection's write to the materialized snapshot. No more
  full re-runs in 'on_write' mode.

Trigger Executor Detail: unchanged from v1.

CDC Sink Worker Detail [v2]:
  Reads the CDC Log Segment from a persisted offset, transforms to
  Debezium-compatible JSON, forwards to configured sink:
    - Kafka: uses rdkafka-c (embedded); at-least-once delivery.
    - Webhook: HTTPS POST with retries; at-least-once.
    - File: rotating JSONL file.
    - Custom: user-provided callback (WASM or host language).

Vector Compactor Detail [v2]:
  Merges SPFresh update log into DiskANN/LSM-VEC graph. Runs incrementally
  to avoid stop-the-world rebuilds.

11.2 Compaction Strategy (v2)

Tiered compaction remains the default:
  1. L0 count ≥ 4 → merge L0 SSTables into sorted run.
  2. Incrementally merge sorted run into Permanent B+ Tree.
  3. Shadow-copy approach: readers use original subtrees via MVCC.
  4. On completion: atomically update root pointers (single 8B write).

v2 adds:
  - Learned Index rebuild: during compaction, eligible subtrees are
    marked for Tier 1 promotion. AHIT Promoter rebuilds the PGM++
    model in parallel with compaction.
  - Zone Map recomputation: done during compaction (single pass).
  - Key-version re-encryption: during key rotation, compaction rewrites
    pages with the new DEK.
  - Columnar Flusher sync: compaction emits end-of-stream markers so
    ColumnChunks can close cleanly.

11.3 Scheduling and Throttling

Priority-based cooperative scheduler (unchanged from v1).
v2 adds "CPU budget" alongside "I/O budget" — vector index builders
and columnar flushers can saturate CPU, so they're throttled when
client thread count is high.

Section 12 — Optional Distributed Extension (OVN-Dist v2)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

OVN-Dist v2 is a thin layer above the core engine. It uses the engine's
clean internal API as a replication substrate.

12.1 Replication Model (Unchanged Core + Extended)

Single-leader Raft replication (Raft-inspired). Leader applies writes
locally + appends to Replication Log (.ovn-rlog). Followers stream the
log and apply.

Replication Log: append-only binary file, keyed by HLC TxID.

v2 additions:
  - Leaseless leader election (optional): under pragma dist_leaderless,
    uses EPaxos-style consensus to avoid leader bottleneck. Higher write
    latency p99 but no single point of failure.
  - Parallel log apply: followers apply non-conflicting log records in
    parallel (determined by document_id hash).

12.2 Embedded Replicas (Turso-style) [v2 NEW]

[IMPL NOTE] This is the killer feature for AI-agent and local-first
applications.

Architecture:
  - Primary .ovn2 runs on a server or leader node.
  - Client applications hold a local .ovn2 file (the "replica").
  - The replica periodically pulls changes from the primary and applies
    them via the same WAL-apply path as follower replication.
  - Read queries on the replica are fully local (sub-ms latency).
  - Write queries on the replica are either:
    - FORWARDED: proxied to the primary (~20-100 ms network latency).
    - DEFERRED (offline mode): queued locally and synced when online.

API (client side):
  const db = await ovn.open('local.ovn2', {
    sync: {
      primary:       'ovn+https://primary.example.com:8443',
      authToken:     process.env.OVN_TOKEN,
      syncInterval:  1000,              // ms (default 1000; 0 = manual)
      writeMode:     'forward',         // 'forward' | 'deferred'
      conflictResolver: 'last_write_wins'  // or custom function
    }
  });

  // Read: local, fast
  const user = await db.findOne('users', { _id: 'u1' });

  // Write: forwarded or deferred based on writeMode
  await db.update('users', { _id: 'u1' }, { $set: { lastLogin: Date.now() } });

  // Manual sync
  await db.sync();                       // force pull from primary
  await db.sync({ push: true });         // also push deferred writes

12.3 Bi-Directional Sync [v2 NEW]

For offline-first applications where writes can originate anywhere:
  - Each node writes with its own HLC-based TxID.
  - CRDT (Conflict-free Replicated Data Type) merge for specific field
    types (counters, sets, maps) — configurable per field.
  - Default for plain fields: last-write-wins based on HLC timestamp.
  - Custom resolver: per-collection function invoked on merge conflicts.

Enable:
  await db.createCollection('counters', {
    crdt: { count: 'g_counter', tags: 'or_set' }
  });

Merge semantics:
  - Insert + Insert (same _id): conflict → resolver (default: HLC winner).
  - Update + Update: per-field merge (CRDT if declared; LWW otherwise).
  - Delete + Update: "tombstone wins" by default; configurable.
  - Insert + Delete: tombstone wins.

12.4 Distributed Change Streams + CDC

Change Stream events tagged with (node_id, HLC TxID) for globally
consistent ordering. Subscribers on any node receive events in total
HLC order. Buffered locally during network partition; re-sync on reconnect.

CDC Log records are replicated alongside WAL. A downstream CDC sink
can connect to any follower for read scalability.

12.5 Sharding Strategy

Hash sharding (default): 64-bit hash of shard key → shard range.
Range sharding: lexicographic ranges of shard key.
v2 adds Geo sharding: S2-cell-based sharding for geospatial workloads.

12.6 Global Transactions (v2)

Cross-shard transactions now have two modes:
  '2pc' (default)      Two-phase commit with HLC-ordered decisions.
                       Blocking; failure-sensitive; high latency.
  'deterministic'      Calvin-style pre-ordering (Section 10.5).
                       Non-blocking; limited to single-shot txns.

Enable per-transaction:
  const txn = await db.beginTransaction({ mode: 'deterministic', crossShard: true });

Section 13 — Security Layer [v2 NEW MAJOR SECTION]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

13.1 Threat Model

Oblivinx v2 is designed against these threats:
  T1 — Disk-theft attacker:    Adversary obtains the .ovn2 file.
  T2 — Memory-dump attacker:    Adversary reads process memory at one moment.
  T3 — Compromised operator:    Admin user attempts to exfiltrate secrets.
  T4 — Network attacker:        Intercepts client-server traffic.
  T5 — Multi-tenant peer:       Other tenant on same host (RLS scenarios).

Mitigations:
  T1 → Encryption at Rest (Section 13.2) + KMS-held KEK.
  T2 → Memory hygiene (zero-on-free for keys); key isolation in process.
  T3 → KMS requires audit / quorum approval; Oblivinx never prints keys.
  T4 → TLS 1.3 with mutual auth (for OVN-Dist + embedded replicas).
  T5 → RLS (Section 13.4) + collection-level encryption + per-tenant
        attached DBs for hard isolation.

Queryable Encryption (Section 13.3) additionally protects against
a SERVER-SIDE attacker: the server can serve queries on encrypted data
without being able to decrypt plaintext. This is relevant for
trust-minimized deployments (e.g., regulated workloads on untrusted cloud).

13.2 Encryption at Rest

Per-collection Data Encryption Key (DEK), 256-bit AES-GCM-SIV.
DEK wrapped by a Key Encryption Key (KEK) held in external KMS.

Algorithms:
  Default:   AES-256-GCM-SIV (misuse-resistant; same nonce safe).
  Optional:  ChaCha20-Poly1305 (for CPUs without AES-NI: embedded, old ARM).
  Optional:  AES-256-GCM (performance-critical; strict nonce discipline).

Nonce derivation: HKDF(DEK, page_number || TxID || page_type) per
encrypted page. Guarantees nonce uniqueness.

Authenticated additional data (AAD): (page_number || collection_id
|| file_UUID || key_version). Detects page relocation attacks.

Key rotation:
  await db.rotateKey('users');  // rolls DEK for collection 'users'
  Old pages are re-encrypted lazily during compaction, using the key_version
  byte in the encryption header to know which DEK to use.

Whole-database rotation:
  await db.rotateDatabaseKey();  // rolls KEK; all DEKs re-wrapped.

13.3 Queryable Encryption (QE) [v2 NEW FLAGSHIP]

[IMPL NOTE] Oblivinx v2 implements QE following the published design of
MongoDB's Queryable Encryption (Cryptography Research Group, 2022-2025).
DO NOT invent a new scheme. Use well-reviewed primitives. The QE scheme
has been subject to extensive cryptanalysis; a novel scheme will not.

Supported query types on encrypted fields:
  Equality         ($eq, $ne, $in, $nin)            — GA.
  Range            ($gt, $gte, $lt, $lte, $between) — GA since v2.0.
  Prefix           ($regex with ^ anchor)            — GA since v2.0.
  Suffix           ($regex with $ anchor)            — GA since v2.0.
  Substring        ($regex with wildcard anywhere)   — GA since v2.0.
  Full-text        ($text, restricted)               — experimental v2.1.

Cryptographic primitives used:
  - AES-256-GCM-SIV for the ciphertext payload.
  - Deterministic keyed tagging (HMAC-SHA256-based) for equality tags.
  - Order-preserving mapping (OPE-variant with bucket randomization) for
    range tags; randomization bound by log₂(domain_size) tags per value.
  - Prefix/suffix/substring: n-gram decomposition + per-n-gram tagging.

On-disk layout of an EncryptedField (OBE2 tag 0x14, detailed in 5.4):
  [1B scheme]
  [16B key_id]
  [varint nonce_len][nonce]
  [varint ciphertext_len][ciphertext]
  [varint tag_count][tag_count × 16B tags]
  [16B auth_tag]

Key lifecycle:
  Each encrypted field has its own Field Encryption Key (FEK), derived
  from the collection DEK + field name. FEK never leaves the client-side
  crypto driver (in supported driver languages).

Client-side vs server-side:
  STRONG mode:  Encryption happens in the client library (JS/TS driver
                or Python driver). Server sees only ciphertext + tags.
                The strongest protection.
  EMBEDDED mode: Oblivinx is the client (single-process embedded). Keys
                held in-process; still protects against disk theft and
                most memory-dump attacks via key-memory hygiene.

API:
  // Declare encrypted fields in schema
  await db.createCollection('patients', {
    validator: { $jsonSchema: { ... } },
    encryption: {
      fields: [
        { path: 'ssn',           scheme: 'eq'       },
        { path: 'salary',        scheme: 'range'    },
        { path: 'address.zip',   scheme: 'prefix'   },
        { path: 'notes',         scheme: 'substring' }
      ],
      keyId: 'kms://aws/arn:alias/patients-key'
    }
  });

  // Insert — ssn encrypted transparently by client driver
  await db.insert('patients', {
    name:   'Alice',
    ssn:    '123-45-6789',
    salary: 75000,
    address: { zip: '10001' },
    notes:  'Patient has a history of hypertension'
  });

  // Query — ciphertext never sent to server; tags matched
  const found = await db.findOne('patients', { ssn: '123-45-6789' });
  const rich   = await db.find('patients', { salary: { $gt: 50000 } });
  const nyArea = await db.find('patients', { 'address.zip': /^100/ });
  const histo  = await db.find('patients', { notes: /hypertension/ });

13.4 Row-Level Security (RLS) [v2 NEW]

Declarative per-row policies evaluated on read and/or write. Inspired by
PostgreSQL RLS.

Policy definition:
  await db.createPolicy('users', {
    name:       'tenant_isolation',
    operations: ['read', 'write'],
    using:      { tenantId: '$$session.tenant_id' },   // read filter
    withCheck:  { tenantId: '$$session.tenant_id' }   // write enforce
  });

Policy evaluation:
  On read: policy's `using` expression is AND-ed into the query filter.
  On write: policy's `withCheck` expression is evaluated against the
            document; rejection → RLS error.

Session context:
  session.tenant_id, session.user_id, session.roles, session.custom.*
  Set via:
    db.setSessionContext({ tenant_id: 't_42', roles: ['reader'] });

Policy composition:
  Multiple policies on the same collection combine via OR (default)
  or AND (strict mode) — choose per-policy.

Permissive vs restrictive:
  Permissive policy: documents matching ANY permissive policy are visible.
  Restrictive policy: documents must match ALL restrictive policies.

Bypass:
  pragma rls_bypass = true (for superuser maintenance; audit-logged).

13.5 Column-Level Encryption

Finer than collection-level. Each column/field can have its own DEK.
Useful for multi-tenant systems where different tenants hold different
keys for columns of the same shared collection (with RLS also applied).

API:
  await db.setColumnEncryption('users', 'ssn', {
    keyId:    'kms://aws/arn:alias/users-ssn-key',
    scheme:   'aead_only'   // or 'eq'/'range'/... for QE
  });

13.6 Authentication and Authorization

User/role registry in Security Metadata Segment (0x0C):
  await db.createUser('alice', {
    password: 'initialPassword!',    // stored as Argon2id hash
    roles:    ['reader', 'writer_team_42']
  });

  await db.createRole('reader', {
    privileges: [
      { resource: { db: '*', collection: '*' }, actions: ['find'] }
    ]
  });

Privileges form:
  { resource: { db, collection }, actions: ['find','insert','update',...] }

Standard built-in roles:
  readOnly, readWrite, admin, dbOwner, backup, restore.

Authentication mechanisms:
  'password'    Username + password (Argon2id, 64 MB memory, 3 iterations).
  'jwt'         Bearer JWT, validated against a JWKS URL or public key.
  'mutual-tls'  X.509 certificate-based (for OVN-Dist clients).
  'none'        Anonymous (single-user embedded mode only).

13.7 Audit Log (OCSF Schema)

13.8 Key Management (External KMS)

Supported KMS backends:
  aws       AWS KMS (access via IAM role or keypair).
  azure     Azure Key Vault.
  gcp       Google Cloud KMS.
  vault     HashiCorp Vault (Transit secrets engine).
  file      File-based dev KMS (key in plaintext file with mode 0600) —
            DO NOT USE IN PRODUCTION; issues a warning at startup.
  custom    User-provided KMS interface (implement the KmsClient trait).

Configuration:
  await ovn.open('data.ovn2', {
    kms: {
      provider: 'aws',
      keyId:    'arn:aws:kms:us-east-1:123456789012:key/abcd1234-...',
      region:   'us-east-1'
    }
  });

Envelope encryption flow:
  1. Oblivinx generates a fresh DEK (256-bit random).
  2. Oblivinx calls KMS.Encrypt(KEK, DEK) → wrapped_DEK.
  3. wrapped_DEK stored in Security Metadata Segment.
  4. On page encrypt: Oblivinx uses the in-memory DEK.
  5. On process restart: Oblivinx calls KMS.Decrypt(KEK, wrapped_DEK) → DEK.
     Plain DEK held only in process memory (zeroed on close).

13.9 Secrets Redaction in Logs / Metrics

Never logged:
  - Plaintext values of encrypted fields.
  - DEKs, KEKs, password hashes, session tokens.
  - WAL record contents (in log-level ≤ info; debug/trace CAN log with
    explicit opt-in pragma debug_log_secrets = true — audit-flagged).

Always logged:
  - Collection names, field paths, doc IDs.
  - Query shape (with values replaced by '<redacted>').
  - Errors (with sensitive parts redacted).

Section 14 — AI / ML Integration Layer [v2 NEW MAJOR SECTION]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The AI Layer makes RAG (Retrieval-Augmented Generation) pipelines a
first-class database concern rather than application-plumbing. The
database owns: chunking, embedding generation (via pluggable providers),
embedding caching, hybrid retrieval, filtered ANN, and re-ranking.

14.1 Embedding Providers

Built-in providers (compile-time feature flags):
  'openai'           OpenAI text-embedding-3-small / -3-large / ada-002.
                     Requires OPENAI_API_KEY in env or explicit config.
  'cohere'           Cohere embed-english-v3.0, embed-multilingual-v3.0.
  'voyage'           Voyage voyage-2, voyage-large-2.
  'jina'             Jina jina-embeddings-v3.
  'local_onnx'       Local sentence-transformers via ONNX Runtime.
                     No network; model bundled or loaded from disk.
                     Default shipped model: all-MiniLM-L6-v2 (22 MB, 384 dim).
  'local_candle'     Local model via the Candle Rust ML framework (future).
  'custom'           Implement the EmbeddingProvider trait.

EmbeddingProvider trait (Rust):
  trait EmbeddingProvider {
    fn model_id(&self) -> &str;
    fn dimensions(&self) -> usize;
    async fn embed(&self, inputs: &[&str]) -> Result<Vec<Vec<f32>>>;
    async fn embed_batch(&self, inputs: &[&str], batch_size: usize) -> Result<Vec<Vec<f32>>>;
  }

Configuration (JS example):
  await db.setEmbeddingProvider({
    name:       'openai',
    model:      'text-embedding-3-small',
    dimensions: 1536,
    apiKey:     process.env.OPENAI_API_KEY,
    batchSize:  100,
    timeout:    30000
  });

14.2 Native RAG Pipeline

The RAG pipeline is a single API that handles the common flow:
  chunk(text) → embed(chunks) → index(embeddings, chunks) → search + rerank.

API:
  // Index a document: chunked, embedded, cached, indexed.
  await db.ragIndex('knowledge', {
    _id:       'doc_42',
    title:     'Earthquake Retrofitting Guide',
    body:      longTextContent,
    metadata:  { category: 'engineering', version: 3 }
  }, {
    chunkField:   'body',
    chunker:      { strategy: 'recursive', size: 512, overlap: 64 },
    embedding:    { field: 'vector', cache: true }
  });

  // Retrieve with hybrid search + rerank.
  const results = await db.ragSearch('knowledge', {
    query:    'How do I retrofit a pre-1990 RC frame for seismic loads?',
    topK:     10,
    weights:  { text: 0.5, vector: 1.0 },      // RRF weights
    filter:   { 'metadata.category': 'engineering' },
    rerank:   { model: 'cohere-rerank-english-v3', topK: 5 }
  });

Chunk strategies:
  'fixed'        Fixed-size non-overlapping chunks.
  'recursive'    Recursive character splitting (default).
  'sentence'     Sentence-boundary splitting (language-aware).
  'semantic'     Semantic splitting based on embedding similarity drops
                 (requires an embed pass during chunking).
  'markdown'     Preserves Markdown heading hierarchy as boundaries.
  'code'         AST-aware splitting for source code (tree-sitter).
  'custom'       User-provided chunker function.

Chunk storage:
  Each chunk is its own document in a hidden sub-collection
  '_ragchunks_<parent_collection>', with _id = <parent_id>:<chunk_index>,
  and a parentId reference field.

14.3 Hybrid Search (Covered in Section 8.8)

The AI Layer wires hybrid-search into ragSearch automatically — text
branch on chunks' text field, vector branch on chunks' vector field,
optional fuzzy branch on exact keyword matches.

14.4 Filtered ANN (Attribute-Aware Vector Search)

Oblivinx v2 supports filtered ANN in two modes:

Post-filter (default for low-selectivity filters ≥ 20%):
  1. ANN search retrieves top (3 × topK) candidates.
  2. Scalar filter applied to candidates.
  3. If fewer than topK remain, re-query with larger candidate pool.

Pre-filter (default for selective filters < 20%):
  Uses DiskANN's filtered-search mode: at graph-traversal time, the
  scalar predicate is evaluated on each candidate. Requires a compound
  vector index:
    await db.createVectorIndex('docs', {
      field:        'embedding',
      dimensions:   1536,
      backend:      'diskann',
      filterFields: ['category', 'language']
    });

  For LSM-VEC, pre-filter uses per-partition attribute filters.

Selectivity estimation:
  The optimizer uses Zone Maps + distinct-count HLL sketches to estimate
  filter selectivity at query time. Chooses pre- vs post-filter automatically.

14.5 Re-Ranking Interfaces

After retrieval (hybrid or pure vector), an optional re-ranker applies
a more expensive model to re-order the top-K candidates. Built-in
integrations:
  'cohere-rerank'      Cohere Rerank v3 API.
  'voyage-rerank'      Voyage reranker.
  'local-crossencoder' ONNX-exported cross-encoder (ms-marco-MiniLM-L-6-v2, etc).
  'custom'             User function: (query, candidates) → ordered list.

Reranker cache:
  (query_hash, candidate_ids_hash) → ordered list, TTL default 5 min.

14.6 Context Windows and Chunking Best Practices

Default chunker parameters tuned for 512-token embedders:
  size:       512 tokens (≈ 2000 chars English).
  overlap:    64  tokens (≈ 256 chars).
  min_chunk:  100 tokens.

For 8K-token embedders (e.g., Voyage, Cohere multilingual):
  size:       1024 tokens; overlap: 128.

For code:
  Use 'code' chunker with tree-sitter grammar for the language.
  Size: 256 tokens per function/class; preserve whole declarations.

Metadata propagation:
  Each chunk carries: parent_doc_id, chunk_index, chunk_tokens,
  parent_metadata (denormalized), source_offset_start, source_offset_end.
  Query results can return the parent document via a $lookup-equivalent
  rag$expandParent operator.

Section 15 — Columnar / HTAP Engine [v2 NEW MAJOR SECTION]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The HTAP engine provides a columnar mirror of selected fields. OLAP
queries hit the columnar representation with vectorized execution;
OLTP queries continue hitting the row store.

15.1 When Columnar Is Used

Automatic promotion criteria (from Workload Profiler):
  Column is promoted to columnar mirror when:
    scans_per_column + aggregations_on_column ≥ 10 per minute for 5 minutes
    AND average rows_per_scan ≥ 1000
    AND column is scalar (not nested document, not array)

Explicit control:
  await db.setColumnar('events', 'eventType', true);   // force enable
  await db.setColumnar('events', 'payload', false);    // force disable

Query routing:
  Query engine considers columnar for $group, $match over scanned fields,
  $project of scanned fields, $sum/$avg/$count aggregations. Routes if
  estimated cost is lower. Falls back to row path if:
  - Any projected field isn't in the columnar mirror.
  - Query requires document structure (e.g., $unwind of an array field
    that isn't columnarized).
  - Snapshot is newer than columnar watermark.

15.2 Arrow IPC Integration

Columnar representation is designed Arrow-compatible:
  - 64 KiB chunks align with Arrow RecordBatch granularity.
  - Encodings (DICT/RLE/BITPACK/DELTA/PLAIN) map to Arrow equivalents.
  - Zero-copy export to Arrow IPC via:
      const arrowStream = await db.exportArrow('events', {
        fields: ['eventType', 'timestamp', 'userId', 'duration'],
        filter: { timestamp: { $gte: last30Days } }
      });

Zero-copy import from Arrow IPC:
  await db.importArrow('events', arrowStream, {
    primaryKey: 'eventId',
    batch: 10000
  });

15.3 Parquet Import/Export

Parquet support via the 'parquet' feature flag (compile-time):
  await db.importParquet('events', '/data/events-2026-q1.parquet');
  await db.exportParquet('events', '/data/events-2026-q2.parquet', {
    compression: 'zstd',
    rowGroupSize: 1000000
  });

Import is bulk-loaded: skips WAL (opt-in), builds indexes at the end.

15.4 Vectorized Execution Pipeline

Adopts DuckDB's morsel-driven parallelism:
  - Query plan is split into morsels (128-256 KiB of columnar data each).
  - Worker threads pull morsels from a queue.
  - Each operator reads columnar input batches, applies SIMD kernels,
    writes columnar output batches.
  - Pipeline breakers (aggregation, sort, join) buffer output until
    parent consumes.

SIMD kernels (compile-time):
  - Predicate evaluation (eq, ne, lt, gt for i32/i64/f32/f64, string_eq).
  - Hash (for group-by).
  - Sum/Min/Max aggregation (lane-wise then horizontal reduce).
  - Null-mask operations.
  - Dictionary lookup (gather instructions on AVX-512).

Fallback: scalar kernels when SIMD unavailable (WASM, embedded).

15.5 Columnar Zone Maps

Each ColumnChunk carries its own zone map:
  (field_id, min, max, null_count, distinct_count_HLL).
  On query time, the Query Optimizer loads the Zone Map Segment and prunes:
    chunks whose [min,max] doesn't intersect predicate range.
    chunks whose null_count == row_count for IS NOT NULL predicates.
  Pruning ratio: typically 70-95% for range-selective analytical queries.

Section 16 — Change Data Capture (CDC) [v2 NEW]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

16.1 CDC vs. Change Streams

Oblivinx v2 offers both, with different guarantees:

                          | Change Streams         | CDC Log
--------------------------|------------------------|----------------------
Durability                 | In-memory ring buffer  | Durable (Segment 0x0B)
Retention                  | ~64 MB (rollover)      | GB-scale; policy-based
Delivery                   | Best-effort            | At-least-once
Subscriber resume          | Resume token           | Checkpointed offset
Ordering                   | Total per-node         | Total per-partition
Schema evolution metadata  | No                     | Yes (schema change events)
Target                     | In-process subscribers | External systems (Kafka etc)

Use Change Streams for: in-process reactive features, cache invalidation.
Use CDC Log for: replication to data warehouses, audit, Debezium pipelines.

16.2 Debezium-Compatible Envelope

CDC records are serializable to Debezium's standard JSON envelope:
  {
    "op":   "c" | "u" | "d" | "r" | "t",   // create/update/delete/read/truncate
    "ts_ms": 1745352123456,
    "before": {...} | null,                // pre-image (optional)
    "after":  {...} | null,                // post-image
    "source": {
      "db":         "main",
      "collection": "users",
      "tx_id":      "hlc:...",
      "lsn":        "wal_offset",
      "node":       "node_a"
    },
    "transaction": {
      "id":         "hlc:...",
      "total_order": 42,
      "data_collection_order": 1
    }
  }

16.3 CDC Sinks

Built-in sinks (compile-time feature flags):
  'kafka'     Kafka producer (librdkafka). At-least-once; configurable acks.
  'webhook'   HTTPS POST with retries (exponential backoff).
  'file'      Rotating JSONL file.
  'stdout'    stdout for containers.
  'otel'      OTLP Logs exporter.
  'custom'    User callback (WASM or host function).

Configuration:
  await db.configureCdc({
    sink: 'kafka',
    config: {
      brokers: ['kafka1:9092', 'kafka2:9092'],
      topic:   'oblivinx.cdc',
      acks:    'all',
      compression: 'zstd'
    },
    collections: ['users', 'orders'],   // or '*' for all
    retention:   { size: '10GB' }
  });

At-least-once delivery guarantee:
  1. CDC Emitter writes record to Segment 0x0B with fsync.
  2. Ack sent to client.
  3. CDC Sink Worker reads from segment, forwards to sink.
  4. Only after sink confirms, advance CDC offset in Metadata Segment.
  5. On crash: resume from last persisted offset. Possible duplicates
     at the sink; downstream systems must dedupe via (tx_id, doc_id).

Section 17 — Incremental View Maintenance (IVM) [v2 NEW]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

17.1 DBSP-Based Approach

v2's materialized views are maintained via a DBSP (Differential Stream
Processor) engine — the same theoretical foundation used by Turso, Materialize,
and Feldera.

DBSP models every relation as a Z-set (multiset with signed weights).
Every operator has an incremental counterpart that consumes deltas and
emits deltas:
  σ  (filter)     — incremental filter.
  π  (project)    — incremental project.
  ∪  (union)      — sum of Z-sets.
  ⋈  (join)       — incremental join with state-augmented operator.
  γ  (group-by)   — incremental aggregation with retractions.

17.2 Stage-by-Stage Incrementality

v1 supported incremental refresh only for some aggregation stages.
v2 supports incrementality for:
  $match, $project, $addFields, $set, $unset     — pure functional, trivially incremental.
  $group, $sum, $avg, $min, $max, $count         — state-full; maintain per-key aggregate.
  $lookup (inner/outer join)                     — maintain hash index on foreign key.
  $sort (with limit)                             — maintain top-K heap.
  $unwind                                         — flat-map, incremental.

Non-incremental stages (force full-refresh):
  $function (arbitrary JS)    — user-defined; cannot guarantee incrementality.
  $where                      — same.
  $sample (random)            — non-deterministic.

Policy:
  If any stage in a view's pipeline is non-incremental, the view falls
  back to 'scheduled' or 'manual' refresh mode. The developer is informed
  at creation time via a warning.

Memory state:
  IVM requires persistent state (e.g., group-by's intermediate sums).
  State is stored per-view in Metadata Segment.
  Size: O(distinct groups) for group-by; O(|R| + |S|) for joins.

Checkpoint:
  IVM state is checkpointed with the WAL; recovered on restart by
  replaying WAL from the checkpoint TxID.

Section 18 — Edge / WASM / Browser [v2 NEW MAJOR SECTION]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

18.1 OPFS Backend

On browsers that support OPFS (Origin Private File System, available
in all modern browsers as of 2024), Oblivinx v2 runs fully in-browser
with .ovn2 files persisted in OPFS.

Architecture (browser profile):
  - Oblivinx core compiled to WASM (~4 MB gzipped).
  - Core runs in a dedicated Web Worker (Dedicated Worker, not Service
    Worker, for reliable OPFS access).
  - Main thread communicates with Worker via postMessage + SharedArrayBuffer
    for zero-copy data transfer (SAB requires COOP/COEP headers).

OPFS I/O:
  - Sync FileSystemSyncAccessHandle API (available in Workers).
  - No fsync; OPFS flushes are semantically fsync-equivalent per write.
  - Random access (read/write at offset) supported.

18.2 IndexedDB Fallback

On browsers where OPFS isn't available (older Safari, embedded webviews):
  - Fallback: store .ovn2 as chunks in IndexedDB.
  - Chunks: 4 KiB each, keyed by page_number.
  - Slower than OPFS by 3-5× due to IDB transaction overhead.

Selection:
  Auto-detect at open time. Fallback is transparent; file is interchangeable
  (OPFS and IDB-backed files are byte-identical on export).

18.3 Embedded Replicas on the Web

Edge / browser profile supports embedded-replica mode (Section 12.2):
  - Read-through: local .ovn2 in OPFS/IDB is the cache.
  - Writes forwarded to primary over HTTPS (POST /ovn/v2/write).
  - Sync interval: 1-30 s; or push-triggered via SSE/WebSocket.

18.4 Offline-First Sync

Offline detection (navigator.onLine + connectivity probe):
  - Online: writeMode: 'forward' (proxy to primary; low-latency reads local).
  - Offline: writeMode: 'deferred' (queue locally; sync on reconnect).

Conflict resolution during re-sync:
  Per-field CRDT merge if declared; otherwise LWW via HLC.

Storage quota awareness:
  Browser quotas vary (20-60% of disk). Oblivinx monitors
  navigator.storage.estimate() and warns at 80% of quota. Configurable
  eviction of cold ColumnChunks and Embedding Cache entries to reclaim space.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

This section defines the developer-facing API. Presented in TypeScript
notation; equivalent bindings exist in Python, C, Go, and WASM.

19.1 Database Lifecycle

  interface OpenOptions {
    // File
    path:             string;
    create?:          boolean;                 // default true
    readOnly?:        boolean;                 // default false
    pageSize?:        number;                  // default 4096

    // Durability
    durability?:      'D0'|'D1'|'D1-strict'|'D2'|'D3';   // default 'D1'
    groupCommit?:     boolean;                 // default true
    concurrentWrites?:boolean;                 // default true (BEGIN CONCURRENT)

    // I/O
    ioEngine?:        'auto'|'uring'|'iocp'|'kqueue'|'posix'|'opfs'|'indexeddb'|'object';
    directIo?:        boolean;                 // default false
    nvmePassthrough?: boolean;                 // default false
    sqpollCores?:     number;                  // default 0

    // Memory
    bufferPoolSize?:  number|'auto';           // bytes, default 'auto'
    memtableSize?:    number;                  // bytes, default 64MB

    // Compression
    compression?:     'none'|'lz4'|'zstd-1'|'zstd-3'|'zstd-19'|'zstd-dict';

    // Security
    kms?:             KmsConfig;
    encryption?:      CollectionEncryptionSpec[];
    authRequired?:    boolean;
    tls?:             TlsConfig;              // for OVN-Dist clients

    // AI
    embedding?:       EmbeddingProviderConfig;

    // Sync (embedded replica)
    sync?:            SyncConfig;

    // Observability
    otel?:            OtelConfig;
    auditLog?:        boolean|AuditConfig;
  }

  const db = await ovn.open('data.ovn2', options);
  await db.close();
  await db.checkpoint();    // force checkpoint now
  await db.vacuum();        // rebuild file, reclaim space
  await db.optimize();      // calibrate cost constants + rebuild learned idx
  await db.migrate_v1_v2(); // upgrade .ovn → .ovn2 in place

Cursor:
  for await (const doc of cur) { ... }

  cur.batchSize(n); cur.maxTimeMs(n); cur.hint('idx_name');

19.5 Transactions

  const txn = await db.beginTransaction({
    isolation:   'snapshot'|'serializable'|'deterministic',
    mode:        'concurrent'|'exclusive',
    readOnly:    false,
    maxTimeMs:   10000
  });
  try {
    await txn.insert('users', {...});
    await txn.update('orders', {...}, {...});
    const svp = await txn.savepoint('s1');
    try { await txn.insert('audit', {...}); }
    catch (e) { await txn.rollbackTo(svp); }
    await txn.commit();
  } catch (e) { await txn.abort(); throw e; }

Retryable writes:
  await db.withTransaction(async txn => { ... }, { maxRetries: 8 });
  // Automatically retries on WRITE_CONFLICT up to maxRetries times.

19.6 Aggregation (MongoDB MQL-compatible)

  const top = await db.aggregate('orders', [

    { $group: { _id: '$customerId', total: { $sum: '$amount' } } },
    { $sort:  { total: -1 } },

All v1 operators preserved. v2 adds:

  $rerank       { query, model }

  [ ] Encryption at rest enabled for regulated data.

22.1 Compatibility Matrix

Feature                      | v1 file ↔ v2 reader | v2 file ↔ v1 reader
-----------------------------|---------------------|---------------------
Header page                   | ✓ (auto-detect)     | ✗ (different magic)
Data Segment                  | ✓                   | ✓ (if no OBE2 tags 0x13+)
Index Segment (B+/SSTable)    | ✓                   | ✓
WAL Segment                   | ✓ (replayed once)   | ✗
Metadata Segment              | ✓                   | partial
Columnar Segment              | N/A                 | ignored (skipped)
Vector Segment                | ✓ if HNSW-only      | partial
CDC Log / Security / AI       | N/A                 | ignored

22.2 Automatic Migration (Safe)

On first open of a v1 .ovn file with a v2 Oblivinx:
  1. Detected by magic 0x4F564E58 ('OVNX').
  2. Opens read-only by default. Issues WARN log.
  3. Existing queries, reads, aggregations work unchanged.
  4. Any new index, vector, or write operation: migration is triggered
     (or an error if pragma auto_migrate_v1 = false).

Explicit migration:
  await db.migrate_v1_v2();
  /*
    Steps:
     - Create .ovn2 file next to .ovn.
     - Copy all pages; rewrite Page 0 to v2 layout.
     - Rebuild AHIT Tier 1 (learned index) over cold zone.
     - Initialize new segments (Columnar, CDC, Security, Embedding,
       ZoneMap, Audit) as empty.
     - Set Feature Bitmap appropriately.
     - Write HMAC over header.
     - Rename .ovn → .ovn.bak; .ovn2 → .ovn (if legacy-name chosen).
  */

22.3 Breaking Changes Requiring App-Side Action

API-LEVEL (TypeScript bindings):
  1. The raw `buffer` escape hatch was removed. Replace:
       const buf = db._raw.buffer();   // v1
     with:
       const snap = await db.exportSnapshot();   // v2
  2. Pragma values: some booleans now accept enum strings. Review your
     pragma calls — TypeScript types will flag incompatibilities.
  3. Change Stream events now include `hlc` alongside `txnId`. Existing
     `txnId`-based consumers still work.
  4. `db.find(...).buffer()` method removed (Node-specific, leaked internals).

DEFAULT CHANGES:
  5. Default durability changed from 'D1' (fsync per record) to 'D1'
     (group commit). If strict per-record fsync is required, set
     durability: 'D1-strict' explicitly.
  6. Default isolation for `db.beginTransaction()` remains 'snapshot',
     but concurrent writers now allowed by default. For strict serial
     single-writer behavior, set `concurrentWrites: false` at open.

CONFIG CHANGES:
  7. `ioEngine: 'auto'` is new default. Legacy blocking I/O: `'posix'`.
  8. `otel` block replaces `metrics.prometheus`; both maps to OTEL now.

22.4 Data Migration Scripts

v2 ships `ovn2-migrate` CLI:
  ovn2-migrate --in data.ovn --out data.ovn2 \
               --rebuild-vector-indexes \
               --enable-columnar collection1,collection2 \
               --progress

Online migration (zero-downtime, requires OVN-Dist):
  1. Add a v2 follower that receives replication from the v1 leader.
     (v2 supports reading v1 replication log.)
  2. Once caught up, promote v2 follower to leader.
  3. Retire v1 leader.
  4. Run ovn2-migrate on leftover v1 replicas.

22.5 Rollback

If v2 proves problematic:
  1. v2 can be run in "v1-compat" mode: pragma v1_compat = true suppresses
     writing any v2-only segments. The resulting file is readable by v1 v1.2.0.
  2. ovn2-migrate --downgrade .ovn2 → .ovn copies only v1-compatible data.
     (v2-only data — vectors built with DiskANN, encrypted fields, columnar
      mirror — is lost. This is a last resort, with loud warnings.)

  Security-only patches to v1.2.x. No new features.

Phase 1 — Foundation (8-10 weeks)

  1.2  Buffer Pool: ARC default, CLOCK-Pro fallback, registered buffers.

  1.6  OBE2 encoder/decoder (tag 0x01-0x12 preserved; 0x13+ stubs).
  1.7  Basic v1→v2 in-place migration.

Phase 2 — Concurrency & Index Upgrade (8-12 weeks)

Appendix A — Glossary (Updated)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

AAD — Additional Authenticated Data; bytes that are integrity-protected
  but not encrypted by AEAD algorithms.
AEAD — Authenticated Encryption with Associated Data (e.g., AES-GCM-SIV).
AHIT — Adaptive Hybrid Index Tree. v2 is three-tier (ART / Learned / B+).
ANN — Approximate Nearest Neighbors (vector search).
Arrow IPC — Apache Arrow Inter-Process Communication format.
ART — Adaptive Radix Tree.
ARC — Adaptive Replacement Cache (buffer pool algorithm).
BK-tree — Burkhard-Keller tree; metric tree for fuzzy string search.
BM25 — Best Match 25; standard full-text relevance scoring.
BufferPool — In-memory cache of disk pages.
CDC — Change Data Capture; durable, at-least-once stream of data changes.
Change Stream — In-process, best-effort notifications of changes.
CLOCK-Pro — Approximate-LRU buffer pool algorithm; LIRS-aware.
ColumnChunk — Unit of columnar storage (default 64 KiB).
Columnar Mirror — Per-field columnar copy alongside row storage (HTAP).
CRC32C — Cyclic Redundancy Check, Castagnoli polynomial. Hardware-accelerated.
CRDT — Conflict-free Replicated Data Type.
CXL — Compute Express Link; interconnect for memory pooling.
DBSP — Differential Stream Processor; theory for incremental computation.
DEK — Data Encryption Key; encrypts the data itself.
Deterministic Transaction — Calvin-style: pre-order transactions, then
  execute deterministically without locks.
DiskANN — SSD-resident graph-based ANN index (Microsoft Research).
Epoch-based Reclamation — Lock-free memory reclamation technique.
fsync — POSIX syscall to durably commit file data to disk.
FTS — Full-Text Search.
Group Commit — Amortized fsync across multiple transactions.
Header HMAC — Integrity MAC over the file header to detect tampering.
HLC — Hybrid Logical Clock; timestamp combining physical time with a
  logical counter for distributed causality.
HNSW — Hierarchical Navigable Small World; graph-based ANN index.
HTAP — Hybrid Transactional / Analytical Processing.
IOCP — I/O Completion Ports (Windows async I/O).
io_uring — Linux async I/O ring (kernel 5.1+; mature 5.6+).
IVF — Inverted File vector index (clustering-based).
IVM — Incremental View Maintenance.
KEK — Key Encryption Key; wraps DEKs. Typically held in KMS.
KMS — Key Management Service (AWS KMS, Azure Key Vault, GCP KMS, Vault).
kqueue — BSD/macOS event notification subsystem.
LSM — Log-Structured Merge-tree.
LSM-VEC — LSM-tree-backed vector index (2025).
MemTable — In-memory write-back buffer (top of the LSM).
MVCC — Multi-Version Concurrency Control.
Neon — Rust→Node.js bindings library.
NVMe passthrough — Send raw NVMe commands via io_uring, bypassing block layer.
OBE — OVN Binary Encoding (v1). OBE2 is the v2 successor.
OCSF — Open Cybersecurity Schema Framework.
OPFS — Origin Private File System (browser filesystem API).
OTEL — OpenTelemetry (unified observability standard).
OTLP — OpenTelemetry Protocol.
Parquet — Columnar file format (Apache Parquet).
PGM — Piecewise Geometric Model Index (learned index).
PGM++ — 2025 improvement to PGM: branchless search, layer-skipping.
PMEM — Persistent Memory (e.g., Intel Optane; CXL-attached NVRAM successors).
QE — Queryable Encryption (MongoDB-originated; encrypted-index scheme
  supporting eq/range/prefix/suffix/substring).
RaBitQ — Asymptotically optimal scalar/binary quantization (SIGMOD 2025).
Raft — Consensus algorithm for leader election + log replication.
RLS — Row-Level Security.
RRF — Reciprocal Rank Fusion; combines ranked lists.
S2 — Google's hierarchical spherical-geometry library (geospatial).
Savepoint — Nested transaction marker allowing partial rollback.
SBOM — Software Bill of Materials.
SCO — Skip-Column Optimization (Oblivinx compound-index acceleration).
Shard Key — Field(s) determining data's shard assignment.
SI — Snapshot Isolation.
SIMD — Single Instruction, Multiple Data (AVX-512, NEON, SVE).
SPFresh — Incremental vector index update protocol.
SQPOLL — io_uring kernel-side polling mode.
SSI — Serializable Snapshot Isolation (Cahill algorithm).
SSTable — Sorted String Table; immutable file in LSM.
Tantivy — Rust full-text search library (Lucene-class).
Tombstone — Marker indicating a deleted row.
TxID — Transaction ID. HLC in distributed mode; monotonic u64 else.
Vamana — DiskANN's graph structure.
Vectorized Executor — Query executor processing batches of rows, not one at a time.
Vector Index — Data structure for nearest-neighbor search.
WAL — Write-Ahead Log.
WSE — Working Set Estimator.
Z-set — Multiset with signed weights (DBSP foundation).
Zone Map — Per-page min/max/null summary enabling predicate pushdown.

Appendix B — .OVN2 File Format Quick Reference
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Magic:            0x4F564E32 ('OVN2') at offset 0x0000
Version:          Major 2, Minor 0 at offset 0x0004
Page size:        4096 bytes default; valid 512..65536
Page 0 size:      4096 bytes (fixed; not scaled with page size)
Byte order:       little-endian for all integers
String encoding:  UTF-8 in headers/metadata
CRC:              CRC32C (Castagnoli polynomial, hardware-accelerated)

Segment codes:
  0x01 Header              | 0x09 Columnar             | 0x11 SSTableIndex
  0x02 Data (row)          | 0x0A Vector               | 0x12 BloomFilter
  0x03 Index               | 0x0B CDC Log              | 0x13 SecurityMetadata
  0x04 WAL                 | 0x0C Security Metadata    | 0x14 RelationIndex
  0x05 Metadata            | 0x0D Embedding Cache      |
  0x06 Blob                | 0x0E Zone Map             |
  0x07 ChangeStream        | 0x0F LearnedIndex         |
  0x08 Attached            | 0x10 Audit Log            |

OBE2 Type Tags:
  0x01 Null          0x09 Document     0x11 GeoJSON       0x19 EmbeddingRef [v2]
  0x02 Bool true     0x0A Array        0x12 RelRef         0x1A TaggedUnion  [v2]
  0x03 Bool false    0x0B Timestamp    0x13 Vector [v2]
  0x04 Int32         0x0C ObjectId     0x14 EncryptedField [v2]
  0x05 Int64         0x0D Decimal128   0x15 BigDecimal    [v2]
  0x06 Float64       0x0E Varint       0x16 DateTimeTZ    [v2]
  0x07 String        0x0F BlobRef      0x17 Duration      [v2]
  0x08 Binary        0x10 GeoPoint     0x18 JSONPatch     [v2]

Page Types:
  0x01 Leaf B+       0x07 ChangeStream   0x0D Audit         0x13 SecurityMetadata [v2]
  0x02 Interior B+   0x08 View           0x0E EmbeddingCache 0x14 RelationIndex
  0x03 Overflow      0x09 ColumnChunk    0x0F ZoneMap
  0x04 WAL record    0x0A VectorGraph    0x10 LearnedIndex
  0x05 Free          0x0B VectorCodebook 0x11 SSTableIndex
  0x06 BlobChunk     0x0C CDC            0x12 BloomFilter

Header Flag Bits (64-bit, offset 0x003C):
  0 WAL_ACTIVE            8  HAS_COLUMNAR [v2]    16 CONCURRENT_WRITES [v2]
  1 ENCRYPTED              9  HAS_VECTOR [v2]      17 SIGNED_RELEASE [v2]
  2 COMPRESSED             10 HAS_CDC [v2]         18 ZSTD_DICT
  3 HAS_BLOB               11 HAS_QE [v2]          19 DIRECT_IO
  4 HAS_CHANGE_STREAM      12 HAS_RLS [v2]         20 WASM_OPFS
  5 HAS_RELATIONS          13 HAS_EMBEDDING_CACHE  21 REMOTE_BACKED [v2]
  6 HAS_VIEWS              14 HAS_LEARNED_INDEX    22 PMEM_CHECKPOINT [v2]
  7 HAS_TRIGGERS           15 HAS_HLC [v2]         23-63 Reserved

Encryption:
  Alg default:     AES-256-GCM-SIV (ChaCha20-Poly1305 on AES-NI-less CPUs)
  Nonce:           HKDF(DEK, page_number || TxID || page_type)
  AAD:             page_number || collection_id || file_UUID || key_version

Compression:
  Default page:    zstd-3
  Default doc:     none (≤ 512 B) / zstd-3 (> 512 B)
  WAL:             never compressed
  Security meta:   never compressed (encrypted only)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Date:       $year, $month, $dayOfMonth, $dayOfWeek, $dayOfYear,

              $isoWeekYear, $isoDayOfWeek, $dateToString, $dateFromString,
              $dateDiff, $dateAdd, $dateSubtract, $dateTrunc.
  Conditional:$cond, $ifNull, $switch, $case.
  Type:       $convert, $toInt, $toLong, $toDouble, $toDecimal, $toString,

                    additive minors; documented deprecation cycle;
                    [[FILE-13]] §11.
        Owner: API governance.

  R-11  Long-offline mobile sync outbox overflow
        L=M  I=M
        Mitigation: bounded outbox (default 100 MiB) with overflow event;
                    CRDT writes never dropped (separate quota); operator
                    tunable; [[FILE-09]] §13.6.
        Owner: replication / mobile SDK.

  R-12  Plugin author writes UB plugin → engine instability
        L=L  I=L
        Mitigation: WASM sandbox prevents memory unsafety; quarantine after
                    repeated faults; per-call fuel + epoch caps;
                    [[FILE-14]] §16.
        Owner: plugins.

  R-13  Compression dictionary blow-up (many dicts, never GC)
        L=M  I=L
        Mitigation: lifecycle states (active/decoder-only/garbage); periodic
                    full-scan reference count during vacuum; alert
                    obx_compress_dict_count; [[FILE-11]] §6.4.
        Owner: storage.

  R-14  Bench infrastructure drift hides perf regressions
        L=M  I=M
        Mitigation: dedicated perf hardware in CI; baseline updated only on
                    release tags; macro benches re-run on RC; [[FILE-17]] §11.
        Owner: build / perf.

  R-15  Documentation lag blocks adoption at v1.0
        L=M  I=H
        Mitigation: docs gated as part of milestone Definition of Done
                    [[FILE-18]] §16; mdBook builds in CI; OpenAPI rendered
                    each PR; [[FILE-16]] §12.
        Owner: docs / DevRel.

For implementers:
  - Start at Phase 1 of the roadmap (Section 23).
  - Keep v1 and v2 crates side-by-side during the transition.
  - Every [IMPL NOTE] block contains a direct, unambiguous decision.
    Do not deviate without updating this spec.

For reviewers:
  - The header HMAC (Section 5.1) is new. If you're reviewing a v2 file
    without HMAC verification infrastructure, you're running v1-compat
    mode.
  - Queryable Encryption follows MongoDB's published design (Section 13.3).
    Do not ship a reimplementation without cryptographic review.

For operators:
  - The audit log is on by default (pragma audit_log). Route it somewhere.
  - The CDC log is durable and can grow — set a retention policy.
  - NVMe passthrough is opt-in and requires device-exclusive access.
