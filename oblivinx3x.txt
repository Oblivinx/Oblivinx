Section 1 — Ecosystem Analysis
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

This section provides a comprehensive architectural analysis of modern database
systems across all major paradigms. Each system is examined through the lens
of its storage engine, indexing design, query execution, transaction model, and
concurrency strategy. This analysis forms the empirical foundation from which
the Oblivinx3x architecture is derived.

1.1 Document Databases
──────────────────────

1.1.1 MongoDB
Storage Engine:
WiredTiger (replaced MMAP in v3.2). Uses WAL + B-tree checkpoint pages.
Collections and indexes each live in separate .wt files.

Disk Layout:
data/ directory with WiredTiger files, journal/ for WAL. BSON encoding.
BSON is length-prefixed — O(1) document-length reads.

Indexing:
B-tree for all index types. _id index always created and unique.
Secondary indexes store key + document _id pointer. Covered queries avoid
full document fetch.

Query Engine:
Multi-stage pipeline. Cost-based planner races candidate plans in parallel
("trial period"). Winning plan is cached. Aggregation pipeline is streaming.

Transactions:
ACID multi-document since v4.0. MVCC at storage level (WiredTiger).
Pessimistic document-level locking with deadlock detection.
Distributed transactions since v4.2.

Concurrency:
Optimistic with document-level locking. Intent locks at collection/DB level.
Readers do not block writers (MVCC). Ticket-based lock manager.

Strengths:
- Mature document model (nested objects + arrays).
- Rich aggregation pipeline.
- Change streams for real-time event processing.
- GridFS for large binary storage > 16MB.
- Time-series collections with automatic bucketing.
- Retryable writes.
- Geospatial 2dsphere indexes (full GeoJSON support).

Weaknesses:
- High memory (~300MB process, WiredTiger 50% RAM default).
- Not embeddable (requires running mongod server).
- Write amplification from B-tree splits + journaling.

1.1.2 CouchDB
Append-only B-tree ("couch file format"). All writes appended at end of .couch
file; B-tree root pointer updated atomically. Crash recovery = truncate to last
valid root. Each database = one .couch file. Compaction rewrites to reclaim space.
Secondary indexing via MapReduce views (JavaScript) or Mango queries.

Strengths: Offline-first sync via Couch Replication Protocol. HTTP-native API.
Weaknesses: Expensive MapReduce views. No joins. Slow compaction on large data.

1.1.3 ArangoDB
Uses RocksDB (LSM-tree) since v3.4. Multi-model: documents, key-value, graphs
via AQL. Single engine reduces infrastructure complexity.
Weaknesses: Abstraction overhead. RocksDB needs manual tuning.

1.1.4 Firestore
Fully managed, serverless, built on Bigtable. Strong consistency with
multi-region replication. Composite indexes must be pre-defined.
Weaknesses: Vendor lock-in. Pricing complexity. No self-hosted option.

1.2 Key-Value Databases
───────────────────────

1.2.1 Redis
In-memory store. Persistence via RDB (snapshots) + AOF (append-only log).
Rich data structures: strings, hashes, lists, sets, sorted sets, streams.
Strengths: Sub-millisecond latency. Pub/Sub.
Weaknesses: Dataset limited by RAM. Persistence complex to tune.

1.2.2 DynamoDB
Fully managed, multi-region. Consistent hashing. B-tree pages on SSDs.
Single-digit ms latency at scale. TransactWrite/TransactGet for ACID.
Weaknesses: Hot partitions. Eventually consistent GSI. Limited query flexibility.

1.2.3 RocksDB
Embedded LSM-tree KV store (Facebook). Writes: MemTable → L0 SSTable → Ln.
Compaction: Leveled (default), FIFO, Universal.
Strengths: Very high write throughput. Tunable amplification. Column families.
Weaknesses: Read amplification (must check all levels). Compaction I/O spikes.
No native document model.

1.3 Column-Family Databases
────────────────────────────

1.3.1 Apache Cassandra
LSM-tree (Bigtable-style). WAL + MemTable → immutable SSTables.
Compaction: STCS, LCS, TWCS. Wide-column: partition key + clustering columns.
Masterless ring with consistent hashing, configurable replication factor.
Strengths: Write throughput. Linear horizontal scaling. Tunable consistency.
Weaknesses: Poor secondary indexes. Operational complexity.

1.3.2 Apache HBase
Distributed column-family on HDFS (Bigtable model). RegionServers hold data
regions as HFiles (SSTable). Strong row-level consistency.
Weaknesses: Full Hadoop/ZooKeeper stack required. No native secondary indexes.

1.3.3 ScyllaDB
C++ reimplementation of Cassandra. Shared-nothing, one shard per CPU core.
Seastar async I/O eliminates context-switch overhead. 10-20x Cassandra throughput.
Weaknesses: Same data model limits as Cassandra.

1.4 Graph Databases
────────────────────

1.4.1 Neo4j
Native graph storage. Nodes, relationships, properties in separate fixed-size
store files. Index-free adjacency: each node stores pointer to first relationship
(doubly-linked list per node) — O(1) neighbor lookup.
Cypher query language. APOC procedures.
Strengths: Deep traversals are fast. ACID.
Weaknesses: Poor for global analytical queries. Not a general-purpose doc store.

1.5 Embedded Databases
───────────────────────

1.5.1 SQLite
Single-file database. Fixed-size pages (default 4KB). B-tree per table/index.
WAL mode (-wal, -shm files) for read/write concurrency. Five locking levels.
ACID with WAL. ~700KB binary, zero dependencies.
Used in iOS, Android, Firefox, Chrome.
Weaknesses: No native document model. Single-writer bottleneck. B-tree write
amplification on random inserts.

1.5.2 LevelDB
Google's embedded LSM-tree KV store. MemTable (skip list) → L0–L6 SSTables.
Foundation for RocksDB.
Weaknesses: Single-threaded compaction. No column families. No document model.

1.5.3 Realm (MongoDB Atlas Device SDK)
Mobile-first. Custom memory-mapped format. Zero-copy reads. MVCC concurrency.
Optional sync to MongoDB Atlas.
Weaknesses: Limited query language. Atlas coupling. Not server-side friendly.

Comparison Table:

MongoDB     | Document     | WiredTiger (B-tree)   | No       | Yes        | Moderate
CouchDB     | Document     | Append-only B-tree    | No       | Yes        | Low
ArangoDB    | Multi-model  | RocksDB (LSM)         | No       | Yes        | Low
Firestore   | Document     | Bigtable              | No       | Yes        | N/A (cloud)
Redis       | Key-Value    | In-Memory + AOF/RDB   | Partial  | No         | High (AOF)
RocksDB     | Key-Value    | LSM-tree              | Yes      | No         | Low

Section 2 — Gap Discovery
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

After exhaustive analysis of Section 1, we identify five fundamental gaps.
These gaps define the design space for Oblivinx3x.

Gap 1 — No Embedded Native Document Database
No production-grade, embedded document database exists without a server process.
SQLite's JSON1 is bolted on. MongoDB requires mongod. Realm is mobile-only.
Backend APIs, edge workers, serverless functions, IoT firmware, and test harnesses
all need "SQLite for documents" — single file, no config, in-process, with MQL.

Gap 2 — Write Amplification in B-tree Embedded Stores
SQLite B-tree: page splits cascade upward, WAL records all modifications → 5-10x.
LSM (RocksDB): compaction can be 10-30x. Neither model solves it completely —
they shift it. Oblivinx3x addresses this with the Hybrid B+/LSM approach.

Gap 3 — Inefficient Compound and Partial Field Indexing
MongoDB's "leftmost prefix rule" forces redundant single-field indexes or full
collection scans when querying non-leftmost compound key fields. No system
auto-adapts index structure based on observed query patterns.

Gap 4 — Memory Pressure in Mixed Read/Write Workloads
MongoDB WiredTiger defaults to 50% RAM (512MB on 1GB edge node). RocksDB block
cache needs manual tuning. No embedded DB implements a self-tuning buffer pool
that dynamically adjusts based on actual working set size.

Gap 5 — No Lightweight System with Both LSM Writes and ACID Semantics
LSM stores (RocksDB) lack full isolation without external coordinators.
B-tree stores (SQLite) have good transactions but write amplification and single-
writer bottlenecks. The hybrid architecture with native MVCC fills this gap.

3.1 Target Deployment Environments

Backend APIs (Node.js, Go, Python, Rust):
  Loaded as a native library module. Used for user-specific data, session stores,
  request caching, and feature flags. Must handle thousands of concurrent reads/s
  with single-writer serialization.

Edge Computing (Cloudflare Workers, Fastly Compute@Edge):
  WASM-compatible sandbox, 128MB memory limit. .ovn file from object storage
  (R2, S3), opened read-only. Cold-start initialization < 5ms.

Microservices:
  Each microservice owns its .ovn file as a local cache or primary store for
  its bounded context. Supports sidecar deployment with shared volumes.

Serverless (AWS Lambda, Google Cloud Functions):
  Bundled within the function package. .ovn from EFS or object storage.
  Read-heavy; writes only on cold-start initialization.

Embedded Systems (ARM Cortex-M4+, RISC-V):
  Cross-compiled with minimal feature set (no replication, no full-text search,
  no vector index). .ovn on flash storage. Runtime memory ≤ 2MB.

responsibility and communicates with adjacent layers through well-defined
interfaces. The architecture is deliberately layered to enable selective
compilation — embedded targets can strip distributed, full-text, and vector
layers.

4.2 The Adaptive Hybrid Index Tree (AHIT) — Architectural Innovation

[IMPL NOTE] AHIT is not a standard B-tree. It is Oblivinx3x's core innovation.
You MUST implement it as described here, not as a plain B+tree or LSM index.

AHIT divides every index into two zones:
  Hot Zone  — In-memory B+ tree (concurrent skip list). Serves all writes and
              frequently accessed reads with zero disk I/O.
  Cold Zone — Immutable, sorted SSTable files on disk. Accessed for infrequent
              data. Protected by Bloom filters for O(1) miss detection.

A background "Promoter" thread monitors access frequency counters (lock-free
atomic increments per node). When a Cold Zone node's counter exceeds the
"promotion threshold" (default: 1000 accesses / 60 seconds), the node and its
subtree are loaded into the Hot Zone.

Demotion: Hot Zone nodes whose counter drops below the eviction threshold
(default: 100 accesses / 60 seconds) are evicted. Hot Zone max size =
configurable (default 25% of buffer pool). LRU eviction when Hot Zone is full.

Why AHIT improves performance:
  Write-heavy: New entries go to Hot Zone MemTable → O(1) amortized.
  Read-heavy:  Popular entries stay in Hot Zone → O(log N), no disk I/O.
  Cold reads:  Bloom filter prevents unnecessary disk reads → O(1) miss.

4.3 Storage Engine Layer

Hybrid B+/LSM architecture:
  Writes → MemTable (concurrent skip list, O(log N) insert)
          → flushed to L0 SSTables
          → background compaction merges into Permanent B+ Tree

The Permanent B+ Tree provides efficient point lookups and range scans for
stable data. The LSM layer is the write intake pipeline — transparent to
upper layers. This gives LSM write throughput + B+ tree read efficiency.

4.4 Document Layer

Implements OVN Binary Encoding (OBE, Section 5.4). Handles:
  - Document parsing and field extraction
  - Dot-notation path resolver for nested access (e.g. "address.city")
  - Type system and type validation
  - Optional JSON Schema validation per collection

Schema validation behavior (mirroring MongoDB):
  validationLevel:
    'strict' (default)  — validate all inserts and updates.
    'moderate'          — validate inserts and updates to documents that
                          already conform; skip re-validation of existing
                          non-conforming documents.
  validationAction:
    'error' (default)   — reject the write, return error to client.
    'warn'              — allow the write, log a validation warning.

[IMPL NOTE] Validation is optional. If no schema set, zero overhead.

4.5 MVCC Transaction Layer

MVCC at the document level. Each write creates a new document version stamped
with a TxID (monotonically increasing 64-bit unsigned integer). Previous version
is retained until garbage collected. A GC thread purges versions beyond the
"horizon TxID" (lowest active snapshot TxID).

Readers see only versions with TxID ≤ snapshot_TxID and tombstone_flag == 0x00.

[IMPL NOTE] v1.2.0 adds Savepoints to this layer — see Section 10.7.

4.6 Query Engine Layer

MQL-compatible subset with Oblivinx-specific extensions.
Pipeline: Lexer → Parser → AST → Logical Plan → Physical Plan → Executor.

Optimizer transformations:
  1. Predicate pushdown (filters before projection)
  2. Index selection (index scan over collection scan)
  3. Sort merging (eliminate redundant sorts on indexed fields)
  4. Geospatial plan selection (S2 cell range scans, R-tree scans)

4.7 Relational Compatibility Layer [NEW]

[IMPL NOTE] This is an optional layer that sits above the storage engine.
It does NOT change the document storage model. It adds relationship metadata
and enforcement mechanisms on top of document collections.

Although Oblivinx3x is a document database, it provides an optional relational
compatibility layer to support application patterns found in SQLite-like systems.
This layer enables foreign-key-like references, ON DELETE/ON UPDATE behaviors,
and declarative relationship metadata across collections.

Features:
  - Foreign key-like references between collections (with optional enforcement)
  - ON DELETE behavior: RESTRICT, CASCADE, SET NULL, NO ACTION
  - ON UPDATE behavior: CASCADE, RESTRICT, SET NULL
  - Declarative relationship metadata stored in Metadata Segment
  - Optional JOIN optimization hints for the query planner
  - Savepoint-aware referential checks during transactions

Relationship Model:
  - one-to-one
  - one-to-many
  - many-to-many (via junction collections)
  - Inverse relationship metadata for fast traversal queries

API:
  await db.defineRelation({
    from: 'posts.user_id',        // source collection + field
    to: 'users._id',             // target collection + field
    type: 'many-to-one',
    onDelete: 'cascade',          // RESTRICT | CASCADE | SET NULL | NO ACTION
    onUpdate: 'restrict',         // CASCADE | RESTRICT | SET NULL
    indexed: true                 // auto-create index on 'from' field
  });

  // List defined relations
  const relations = await db.listRelations();

  // Remove a relation definition
  await db.dropRelation({ from: 'posts.user_id', to: 'users._id' });

[IMPL NOTE] Relation definitions are stored in the Metadata Segment.
Enforcement runs at commit-time inside the MVCC transaction layer when
4.8's referential integrity mode is set to 'strict' or 'soft'.
Cascade operations execute as sub-writes within the same transaction.

4.8 Referential Integrity Validation [NEW]

Oblivinx3x can optionally validate cross-collection references during write
and commit phases.

Modes:
  'off'    — No validation. Default. Any reference is allowed.
  'soft'   — Warn-only. Log invalid references, never reject writes.
  'strict' — Reject write with a ReferentialIntegrityError if the referenced
             document does not exist, or if a referenced document is being
             deleted and ON DELETE = RESTRICT.

[IMPL NOTE] In 'strict' mode, validation occurs as follows:
  - On INSERT: verify that referenced _id exists in target collection.
  - On DELETE: check if any other document references this document's _id
    via a defined relation with ON DELETE = RESTRICT; reject if found.
  - On CASCADE DELETE: delete all referencing documents in the same txn.
  - On SET NULL: set the referencing field to null in all referencing docs.
Validation is deferred until commit (not per-operation) to avoid blocking the write path.

Configure:
  await db.setReferentialIntegrity('strict'); // 'off' | 'soft' | 'strict'
  const mode = await db.getReferentialIntegrity();

Section 5 — .OVN Database File Format
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The .ovn file is the single foundational storage primitive of Oblivinx3x.
It is a single-file, page-oriented, self-describing binary format.
All integers: little-endian. All strings in headers/metadata: UTF-8.

5.1 File Header (Page 0 — Always 4096 bytes)

Offset   Size   Field
──────────────────────────────────────────────────────────────
0x0000    4     Magic Number: 0x4F564E58 ('OVNX')
0x0004    2     Format Version Major (e.g., 0x0001)
0x0006    2     Format Version Minor (e.g., 0x0002)
0x0008    4     Page Size in bytes (default 4096; valid: 512 – 65536)
0x000C    8     Total file size in bytes
0x0014    8     Root Page Number of Primary B+ Tree
0x001C    8     Root Page Number of Metadata B+ Tree
0x0024    8     WAL Segment Start Offset (page number)
0x002C    8     WAL Segment Size in bytes
0x0034    4     Number of collections
0x0038    4     Checksum (CRC32 of bytes 0x0000–0x0034)
0x003C    4     Flags:
                  Bit 0: WAL active (set on open, cleared on clean checkpoint)
                  Bit 1: Encrypted (per-collection AES-256-GCM)
                  Bit 2: Compressed (page-level LZ4 active)
                  Bit 3: Has Blob Segment
                  Bit 4: Has Change Stream Log
                  Bit 5: Has Relational Layer metadata [NEW]
                  Bit 6: Has View definitions [NEW]
                  Bit 7: Has Trigger definitions [NEW]
0x0040    8     Creation timestamp (Unix epoch, milliseconds)
0x0048    8     Last checkpoint timestamp
0x0050    16    Database UUID (128-bit random)
0x0060    64    Application identifier (UTF-8, null-padded)
0x00A0    ...   Reserved for future use
0x0FF0    16    Shadow copy: magic + version + page size (corruption detection)

5.2 Segment Layout

Segments are page-aligned. Segment Directory is at Page 1 and maps type codes
to [start_page, end_page] ranges.

Segment Type         | Code | Contents                                  | Start Page
---------------------|------|-------------------------------------------|------------
Header Segment       | 0x01 | File header, DB metadata, collection reg. | 0
Data Segment         | 0x02 | Document leaf pages, overflow pages        | 2
Index Segment        | 0x03 | AHIT B+tree nodes, SSTable index blocks,  | Dynamic
                     |      | inverted index, BK-tree, autocomplete     |
                     |      | trie, geospatial R-tree/S2, HNSW graph    |
WAL Segment          | 0x04 | Write-ahead log records, txn journal       | Dynamic
Metadata Segment     | 0x05 | Collection schemas, statistics, version   | Dynamic
                     |      | history, session registry, idempotency    |
                     |      | cache, relation defs, view defs,          |
                     |      | trigger defs, pragma state [NEW fields]   |
Blob Segment         | 0x06 | BLOBs > 16MB, chunked at 255KB boundaries | Dynamic
Change Stream Log    | 0x07 | Append-only circular buffer of events     | Dynamic
Attached DB Index    | 0x08 | Map from alias → .ovn file path [NEW]     | Dynamic

Blob Segment (0x06) Detail:
  GridFS-equivalent for binaries > 16MB.
  - Blobs split into 255KB chunks, stored as page sequences.
  - Blob Directory B+ tree: blob_id → chunk page ranges.
  - Referenced from documents via OBE type tag 0x0F (BlobRef).
  - Blob writes follow WAL path for durability.
  - Per-chunk LZ4 or zstd compression; configured independently from
    document compression.
  - delete: marks chunks free; Blob Compactor reclaims space.

Change Stream Log (0x07) Detail:
  Append-only circular buffer.
  Each record: [8B TxID][4B collection_id][1B operation_type: ins/upd/
    rep/del/drop][16B document_id][varint pre_img_len][pre_img OBE bytes]
    [varint post_img_len][post_img OBE bytes]
  Default size: 64MB. When full, oldest records are overwritten.
  Subscribers that fall behind receive CHANGE_STREAM_INVALIDATE and must
  re-establish subscription from a fresh snapshot.

Attached DB Index (0x08) [NEW]:
  Stores the map of alias → absolute file path for ATTACH/DETACH.
  Format: [varint alias_len][alias UTF-8][varint path_len][path UTF-8]
  Updated atomically on db.attach() and db.detach() calls.

5.3 Page Layout

Every page is exactly [page_size] bytes (default 4096).
Every page begins with a standard 32-byte Page Header.

Page Header (32 bytes):
  Offset  Size  Field
  0x00     4    Page Type:
                  0x01 = Leaf B+tree page
                  0x02 = Interior B+tree page
                  0x03 = Overflow page (large field values)
                  0x04 = WAL record page
                  0x05 = Free page
                  0x06 = BlobChunk page
                  0x07 = ChangeStreamRecord page
                  0x08 = ViewPage (materialized view snapshot) [NEW]
  0x04     8    Page Number (0-indexed)
  0x0C     8    Transaction ID (TxID) that last modified this page
  0x14     2    Number of document slots (leaf pages only; 0 for others)
  0x16     2    Free space offset (bytes from start of payload area)
  0x18     4    Right sibling page number (B+ tree leaf chain; 0 = last leaf)
  0x1C     4    CRC32 checksum of page bytes 0x20 to end

Leaf Page Payload:
  [Slot Array — 4 bytes per slot, grows forward from payload start]
    Slot N: { doc_offset: u16, doc_length: u16 }
  [Document Data — packed from payload end, growing backward]
    doc_N: OBE-encoded document bytes

Interior Page Payload:
  [Key-Pointer Pairs — repeated: (key_length: varint, key_bytes, child_page: u64)]
  [Rightmost child pointer — 8 bytes]

5.4 OVN Binary Encoding (OBE) Format

[IMPL NOTE] OBE is inspired by BSON but with two key improvements:
  1. Field IDs (2-byte) replace repeated field name strings for schema-aware
     collections — reduces storage 40-60%.
  2. Variable-length integers (LEB128 varint) minimize space for small values.

OBE Document Structure:
  [4 bytes]   Document total length (includes this field itself)
  [16 bytes]  Document ID (128-bit UUID v4 or user-provided _id)
  [8 bytes]   Transaction ID (version stamp, u64 little-endian)
  [1 byte]    Tombstone flag (0x00 = live, 0xFF = deleted)
  [varint]    Number of fields
  [Fields...] (see OBE Field Structure below)

OBE Field Structure:
  [varint]    Field name length:
                > 0 → followed by that many UTF-8 bytes (field name)
                == 0 → followed by 2-byte Field ID (schema-aware mode)
  [bytes]     Field name (UTF-8) OR Field ID (u16 little-endian)
  [1 byte]    Type tag (see Type Tag Table below)
  [...]       Value bytes (type-dependent encoding)

Type Tag Table:
  Tag   | Type                 | Encoding
  ------+----------------------+--------------------------------------------------
  0x01  | Null                 | No value bytes
  0x02  | Boolean True         | No value bytes
  0x03  | Boolean False        | No value bytes
  0x04  | Int32                | 4 bytes little-endian (i32)
  0x05  | Int64                | 8 bytes little-endian (i64)
  0x06  | Float64              | 8 bytes IEEE 754 (f64)
  0x07  | String               | varint byte_length + UTF-8 bytes
  0x08  | Binary               | varint byte_length + raw bytes
  0x09  | Document (nested)    | Recursive full OBE document (has its own 4B length prefix)
  0x0A  | Array                | varint element_count + element values (each with type tag)
  0x0B  | Timestamp            | 8 bytes Unix epoch milliseconds (u64)
  0x0C  | ObjectId             | 12 bytes: 4B unix time + 5B machine ID + 3B counter
  0x0D  | Decimal128           | 16 bytes IEEE 754 decimal128
  0x0E  | Varint               | 1–9 bytes LEB128 signed integer
  0x0F  | BlobRef              | 16B blob_id (UUID) + 8B total_size (u64)
  0x10  | GeoPoint             | 16B: 8B latitude (f64) + 8B longitude (f64) IEEE 754
  0x11  | GeoJSON              | varint byte_length + GeoJSON UTF-8 bytes
  0x12  | RelRef [NEW]         | varint coll_name_len + coll_name + 16B target_doc_id
        |                      | (a typed cross-collection reference for the relational layer)

[IMPL NOTE] RelRef (0x12) is used by the Relational Compatibility Layer (Section 4.7)
to store typed references in documents. When a field is declared as a relation via
defineRelation(), the engine automatically stores references as RelRef tags.
The referential integrity validator inspects RelRef fields at commit time.

5.5 Compression

Two independent levels; both optional; configured per-collection.

Page-level compression (LZ4):
  Compress the payload of Data Segment and Index Segment pages (never WAL pages).
  LZ4 chosen for decompression speed (> 4 GB/s) so reads are not bottlenecked.
  Applied/removed transparently — the Page Header's type field indicates raw vs.
  compressed page only at the 32-byte header level; bytes 0x20 onward are
  LZ4-compressed when the collection has page_compression = 'lz4'.

Document-level compression (zstd):
  Compress individual documents > 512 bytes using Zstandard with a shared
  per-collection dictionary trained on sample documents.
  Dictionary stored in Metadata Segment; loaded into memory on collection open.
  Shared dictionary dramatically improves ratio for structurally similar docs
  (e.g., all user records share field name patterns).

Section 6 — Write Path Micro Process
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The write path minimizes client latency. The client is unblocked after WAL
flush (Step 6); all heavy I/O is delegated to background threads.

6.1 Write Pipeline Overview

  Client Thread                              Background Thread
  ─────────────                              ─────────────────
  [1]  Client Request (insert/update/delete)
        │
  [2]  Query Parser → validate MQL syntax
        │
  [3]  Trigger Executor → run beforeInsert / beforeUpdate / beforeDelete [NEW]
        │                  (sync triggers can modify or reject the write)
  [4]  Document Validator → check JSON Schema (if set)
        │
  [5]  Relational Layer checks (if relations defined + strict mode) [NEW]
        │                  (foreign key existence check on INSERT)
  [6]  MVCC Snapshot → assign TxID
        │
  [7]  Serialize to OBE binary (including RelRef tags if needed)
        │
  [8]  WAL Writer → fsync .ovn WAL segment ────────→ WAL Cleaner
        │
  [9]  MemTable Insert (skip list, O(log N))
        │                                    ────────→ [10] MemTable Flush → L0 SSTable
  [11] AHIT Index Update (Hot Zone)
        │                                    ────────→ [12] Compaction (L0 → Permanent B+Tree)
  [13] Referential integrity commit check (cascade/restrict) [NEW]
        │
  [14] Return Acknowledgment to Client        ────────→ [15] Change Stream Emitter (async)
                                              ────────→ [16] afterInsert / afterUpdate / afterDelete trigger (async) [NEW]
                                              ────────→ [17] TTL Reaper check (timer)
                                              ────────→ [18] Materialized View Refresh (if triggered) [NEW]

6.2 Step-by-Step Write Path Detail

Step 1 — Client Request
  Client calls db.insert(), db.update(), or db.delete() via JavaScript/FFI API.
  Marshaled to OVN API request format. Submitted to Engine's write queue.
  Write queue: lock-free MPSC (multi-producer, single-consumer) to prevent
  thundering herd from concurrent writers.

Step 2 — Query Parsing
  Tokenize and parse MQL filter + update operator expressions into an AST.
  Bounded by configurable max document size (default 16MB, min 4KB).
  Documents > 16MB must use BlobRef (Section 5.2, Blob Segment).

Step 3 — Trigger Execution (beforeWrite) [NEW]
  If beforeInsert / beforeUpdate / beforeDelete triggers are defined for the
  collection (Section 13.9), run them synchronously on the client thread.
  Triggers run inside the transaction context and can:
    - Modify the document (return modified doc)
    - Reject the write (throw an error)
  Trigger timeout: configurable (default 500ms). Timeout → write rejected.

Step 4 — Document Validation
  If collection has a JSON Schema (db.setValidator()), validate using a
  streaming validator. Behavior controlled by validationAction and
  validationLevel (Section 4.4). Zero overhead when no validator set.

Step 5 — Relational Layer Checks (INSERT only) [NEW]
  If referential integrity mode = 'strict' or 'soft' and relations are defined,
  verify that referenced document IDs exist in their target collections.
  'strict': reject write on failure.
  'soft': log warning, allow write.
  This check is a fast AHIT primary index lookup per RelRef field.

Step 6 — MVCC Snapshot and TxID Assignment
  Assign a monotonically increasing TxID (u64). Source: hardware clock
  (rdtsc on x86, cntvct_el0 on ARM) + thread-local sequence counter.
  For retryable writes: WAL record also embeds lsid (16B) + txnNumber (u64).
  For savepoints: TxID is the parent transaction's TxID; savepoint state is
  stored in the transaction context in-memory (not WAL) — see Section 10.7.

Step 7 — OBE Serialization
  Serialize document to OBE binary (Section 5.4).
  Schema-aware collections: resolve field name → 2-byte Field ID (40-60% reduction).
  Write to thread-local serialization buffer (avoid heap allocation on hot path).

Step 8 — Write-Ahead Log
  WAL record format:
    [4B record_type]
    [8B TxID]
    [4B collection_id]
    [16B lsid (zeroed if not retryable)]
    [8B txnNumber (0 if not retryable)]
    [1B savepoint_depth (0 if not in savepoint)] [NEW]
    [varint document_length]
    [document_bytes (OBE)]
    [4B CRC32]
  WAL writer calls fsync() / fdatasync() after each write.
  Group commit: batch multiple pending WAL records into one fsync when multiple
  clients are writing concurrently.

Step 9 — MemTable Insert
  Concurrent skip list, O(log N). Partitioned by collection ID.
  When MemTable size > threshold (default 64MB), freeze current MemTable,
  create new one, queue frozen MemTable for background flush.

Step 10 — MemTable Flush (Background)
  Serialize frozen MemTable → L0 SSTable in Index Segment.
  SSTable: sorted by document ID, includes Bloom filter, min/max keys.
  After fsync, corresponding WAL records become eligible for reclamation.

Step 11 — AHIT Index Update
  For each indexed field: extract value via OBE decoder, insert into Hot Zone
  skip list (lock-free CAS). O(log N), non-blocking for readers.
  For geospatial fields: encode GeoPoint/GeoJSON as S2 cell token → insert
  into Geospatial Index (Section 8.7).

Step 12 — Compaction (Background)
  When L0 SSTable count ≥ 4: merge into Permanent B+ Tree.
  Merge is shadow-copy — readers use original subtrees via MVCC during compaction.
  On completion: atomically update B+ tree root pointers (single 8-byte write).

Step 13 — Referential Integrity Commit Check [NEW]
  At commit time (for multi-statement transactions), run the full referential
  integrity validation:
    - CASCADE DELETE: delete all referencing documents in same txn.
    - SET NULL: update referencing fields in same txn.
    - RESTRICT: abort txn if any constraint would be violated.
  This step runs before Step 14 (client acknowledgment).

Step 14 — Client Acknowledgment
  Client receives ack as soon as WAL write completes (Step 8).
  Ack includes: assigned document _id, TxID.

Step 15 — Change Stream Emitter (Background, Async)
  Write change event to Change Stream Log (Segment 0x07).
  Event includes: operation type, doc ID, pre-image (from MVCC version chain),
  post-image (newly written doc). Notify active subscribers asynchronously.

Step 16 — After-Write Trigger Execution (Async) [NEW]
  Run afterInsert / afterUpdate / afterDelete triggers asynchronously.
  Errors in async triggers are logged to observability; they do NOT roll back
  the committed write. Use afterWrite triggers for side effects (logging,
  cache invalidation, notification delivery).

Step 17 — TTL Reaper (Timer-Based)
  Runs every 60 seconds. Scans TTL indexes for expired documents.
  AHIT range scan: time value ≤ (current_unix_ms - expireAfterSeconds * 1000).
  Batch deletes: max 1000 per cycle. Each batch = single WAL write.

Step 18 — Materialized View Refresh (Async) [NEW]
  If there are materialized views defined with refresh policy 'on_write',
  the View Refresher worker checks if any affected collection was written.
  If yes, it re-runs the view's aggregate pipeline and updates the view's
  snapshot pages (PageType 0x08) in the Index Segment.

6.3 Failure Recovery

On process restart:
  1. Read Page 0. Verify magic (0x4F564E58) and CRC32.
  2. Check WAL Active flag. If set: previous session did not cleanly checkpoint.
  3. Scan WAL segment forward from last checkpoint TxID.
  4. For each WAL record with COMMIT marker: re-apply write to MemTable.
  5. For each WAL record without COMMIT marker: discard (partial txn).
  6. Flush recovered MemTable → L0 SSTable → trigger compaction.
  7. Write new checkpoint to Page 0. Clear WAL Active flag.

Recovery time ∝ size of uncommitted WAL since last checkpoint.
With 60s checkpoint interval: recovery typically < 2 seconds.

[IMPL NOTE] Savepoint rollbacks (partial transaction undos) do not write to WAL.
Only full transaction commit or abort appears in WAL. On crash, savepoint state
is lost — the entire transaction is treated as uncommitted and discarded.

Section 7 — Read Path Micro Process
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Reads are fully concurrent — no write locks acquired. MVCC provides
consistent snapshots across concurrent writes.

7.1 Read Pipeline Overview

  [1]  Client Query: db.find('users', { age: { $gt: 18 } })
        │
  [2]  Query Parser → AST
        │
  [3]  View Resolution [NEW] — if collection is a logical view, redirect
        │                       to source collection + inject pipeline
  [4]  Explain / Plan Inspection [NEW] — if db.explain() called, return
        │                                 plan without executing
  [5]  Query Optimizer → Index Selection / Plan Costing
        │
  [6]  MVCC Snapshot Acquisition → record minimum active TxID
        │
  [7]  AHIT Index Lookup (if indexed field)
        │        ↓ miss
  [8]  Buffer Pool Check → page cache hit?
        │        ↓ miss
  [9]  SSTable Bloom Filter Check → key definitely absent?
        │        ↓ possible hit
  [10] Disk Page Fetch → read B+ Tree leaf page
        │
  [11] OBE Document Decoding → reconstruct document
        │
  [12] MVCC Version Filtering → keep only TxID ≤ snapshot_TxID, tombstone = 0x00
        │
  [13] Predicate Evaluation → apply remaining filter predicates
        │
  [14] Projection → select requested fields
        │
  [15] Response Serialization → return to client

7.2 Query Optimizer — Plan Costing

Cost formula:
  Cost(plan) = (N_pages_read × PAGE_COST)
             + (N_docs_decoded × DECODE_COST)
             + (N_comparisons × CMP_COST)

PAGE_COST, DECODE_COST, CMP_COST are tunable constants calibrated to hardware.
Statistics (updated by Statistics Collector worker every 60s):
  - document count per collection
  - average document size
  - field cardinality histograms
  - page fill ratios

Plan types in preference order:
  1. Indexed point lookup:        O(log N)
  2. Indexed range scan:          O(log N + K) — K = result count
  3. Covered index scan:          O(log N + K), no document fetch
  4. Geospatial index scan:       S2 cell range / R-tree neighbor
  5. Partial collection scan:     high-selectivity filter
  6. Full collection scan:        fallback

7.3 View Resolution [NEW]

  [IMPL NOTE] When the queried collection name matches a registered view
  (Section 13.8), the read path transparently:
    1. Loads the view definition from Metadata Segment.
    2. Substitutes the query's source with the view's source collection.
    3. Prepends the view's stored pipeline stages to the query's pipeline.
    4. For materialized views: reads from the precomputed ViewPage snapshots
       directly, bypassing stages 7-12 entirely.
  The client receives results as if it queried the view directly. The view
  mechanism is invisible to the client.

7.4 Explain / Plan Inspection [NEW]

  If the query is executed via db.explain(), the optimizer produces a plan
  document but does NOT execute the query. See Section 13.10 for the API.
  The plan document includes: chosen index, scan type, estimated cost,
  estimated documents examined, estimated documents returned, and fallback
  reason if full scan was selected.

7.5 MVCC Snapshot Semantics

  Snapshot TxID = minimum active TxID at time read begins.
  Visible versions: TxID ≤ snapshot_TxID AND tombstone = 0x00
                   AND TxID > last_compaction_TxID.
  Version chain walk: O(V) where V = versions in chain. Typically V = 1–3.
  GC thread bounds V by purging versions below horizon TxID.

7.6 Bloom Filter Optimization

  Each SSTable has a Bloom filter: k=7 hash functions, 1% false-positive rate.
  For 1M document SSTable: ≈ 1.2MB per filter.
  On SSTable miss (key definitely absent): O(1), no disk I/O.

7.7 Complexity Analysis

  Operation                | Best     | Avg        | Worst      | Notes
  -------------------------|----------|------------|------------|-------------------
  Point lookup (indexed)   | O(log N) | O(log N)   | O(log N)   | B+ tree traversal
  Point lookup (unindexed) | O(1)*    | O(N)       | O(N)       | *Bloom filter hit
  Range scan (indexed)     | O(log N+K)| O(log N+K)| O(log N+K) | K = result count
  Range scan (unindexed)   | O(N)     | O(N)       | O(N)       | Full collection
  Geospatial ($near)       | O(log N+K)| O(log N+K)| O(N)       | R-tree/S2
  Insert                   | O(log N) | O(log N)   | O(log N)   | MemTable skip list
  Update (indexed field)   | O(log N) | O(log N)   | O(log N)   | Includes index upd.
  Delete                   | O(log N) | O(log N)   | O(log N)   | Tombstone insert
  Aggregation (indexed)    | O(K log N)| O(K log N)| O(N log N) | K = matching docs
  View read (logical)      | O(log N) | O(log N)   | O(N)       | Passes thru pipeline
  View read (materialized) | O(1)     | O(1)       | O(K)       | Precomputed pages

Section 8 — Indexing System
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

All indexes are AHIT-based (Section 4.2). The AHIT is the universal index
structure — it wraps every index type (secondary, compound, full-text,
geospatial, vector) with Hot Zone / Cold Zone adaptive caching.

8.1 Primary Document Index

  Every collection has a primary index on _id. This is a persistent B+ tree
  in the Index Segment. Never stored in AHIT Hot Zone — it is always compact
  enough to be highly cache-efficient via the Buffer Pool.
  
  Fanout: (page_size - 32) / (key_size + pointer_size)
    Default: (4096 - 32) / (16 + 8) = 169 ≈ fanout ~170
  For 1 billion documents: ceil(log₁₇₀(10⁹)) = 4 levels → ≤ 4 page reads per lookup.

8.2 Secondary Indexes

  Key format: [field_value_bytes | document_id_bytes]
  Value: document_id (not full document, to minimize index size).
  Stored in OBE encoding for consistency with the document store.

  TTL Index:
    Special secondary index on a Timestamp or Int64 (Unix epoch) field.
    Created with expireAfterSeconds option.
    TTL Reaper (Section 11.1) scans for time_value ≤ (now - expireAfterSeconds).
    Single-field only. Cannot be compound. Incompatible with capped collections.

  Sparse Index [NEW]:
    Only indexes documents that contain the indexed field.
    Documents without the field are not inserted into this index.
    Useful for optional fields where full-collection index size would be wasteful.

  Unique Index [NEW]:
    Enforces field value uniqueness across the collection.
    Checked at AHIT insert time: if the value already exists in the index,
    reject the write with a UniqueConstraintError before WAL write.

  Partial Index [NEW]:
    Only indexes documents matching a partialFilterExpression.
    partialFilterExpression is a standard MQL filter object.
    The AHIT Promoter respects partial index filters — documents not matching
    the filter are never inserted into the index.

  Hidden Index [NEW]:
    A hidden index exists on disk and is kept up-to-date on writes, but is
    NEVER selected by the query optimizer. Used to preview impact of dropping
    an index without actually dropping it.
    Toggle: await db.hideIndex('coll', 'index_name');
            await db.unhideIndex('coll', 'index_name');

  Wildcard Index [NEW]:
    Indexes all fields (or a subtree of fields) in documents, regardless of
    field names. Useful for collections with dynamic/unknown schemas.
    Created as: await db.createIndex('coll', { '$**': 1 })
                 or a sub-path: { 'metadata.$**': 1 }
    Internally: one AHIT entry per (field_path, value, doc_id) tuple.

8.3 Compound Indexes and the Skip-Column Optimization (SCO)

  Standard compound index on (A, B, C) in a plain B-tree cannot be used for
  queries filtering only on B or C (leftmost prefix rule).

  SCO Design:
    Each compound index node stores:
      1. Primary sort key (A, B, C concatenated in declared order)
      2. A 64-bit hash fingerprint for each field: fingerprint[A], fingerprint[B], fingerprint[C]
    
    For a partial-key query (e.g., only filtering on B):
      The AHIT scan applies bitwise AND masks to the fingerprints to eliminate
      B+ tree pages that cannot contain matching documents BEFORE reading
      the full key. Reduces scan from O(N) to O(sqrt(N)) for uniform data.
  
  [IMPL NOTE] SCO is stored in each B+ tree interior node as a 64-bit
  fingerprint array. Memory overhead per interior node: 8 * num_fields bytes.

8.4 Full-Text Index

  Inverted index in Index Segment. Maps stemmed, lowercased tokens to posting lists.
  Posting list entry: [doc_id, field_id, term_frequency, list_of_positions]
  Positions enable phrase queries and proximity searches.

  Tokenization: Unicode Text Segmentation (UAX #29) — language-agnostic.
  Stemmers: Porter (English), Snowball (European languages) — registered via plugin.
  Storage: B+ tree keyed on token; overflow pages for high-frequency terms.

  8.4.1 Fuzzy Matching
    BK-tree (Burkhard-Keller tree) as companion to the inverted index.
    Organized by Levenshtein edit distance. A fuzzy query retrieves all tokens
    within edit distance N from the query term, resolves their posting lists.

    Fuzzy options in $text operator:
      { $text: { $search: 'wiget', $fuzzy: { maxEdits: 1, prefixLength: 3 } } }
      maxEdits: Levenshtein distance (1 or 2; 0 = exact match).
      prefixLength: Leading chars that must match exactly before fuzzy expansion.
    
    BK-tree built lazily during index compaction; stored serialized in Index Segment.

  8.4.2 Autocomplete Index
    Compressed radix trie (Patricia trie) in Index Segment.
    Each node: compressed edge label + pointer to list of doc IDs in subtree.
    Trie construction: O(L) per token (L = token length).
    Prefix lookup: O(L + K) (K = results returned).
    Can be disabled per-collection to save memory.

    API:
      const suggestions = await db.autocomplete('products', {
        field: 'name', query: 'wid', maxResults: 5
      });
      // Returns: ['Widget A', 'Widget B', 'Widget Pro', ...]

8.5 Optional Vector Index (ANN Search)

  HNSW (Hierarchical Navigable Small World) algorithm.
  Stored as serialized HNSW graph in Index Segment.
  ANN search: O(log N) query complexity; configurable recall-precision trade-off.
  Dimensions: 2 to 2048 (float32). Distance metrics: Euclidean (L2), Cosine, Inner Product.
  Built asynchronously by background HNSW Builder thread.

8.6 AHIT Hot Zone / Cold Zone Promotion

  Access frequency counter per B+ tree node: lock-free atomic increment.
  Promotion threshold (default): 1000 accesses in last 60 seconds.
    → Load node + subtree into Hot Zone (memory-resident B+ tree mirror).
    → All subsequent reads from this node: zero disk I/O.
  Demotion threshold (default): < 100 accesses in 60 seconds.
    → Evict node from Hot Zone.
  Hot Zone max size: 25% of buffer pool (configurable). LRU eviction when full.
  Memory: Hot Zone uses dedicated slab allocator (Section 9.4).

8.7 Geospatial Index (2dsphere / 2d)

  Index Types:
    2dsphere — Earth-surface spherical coordinates (longitude/latitude).
               Documents must store as GeoJSON:
                 Point, LineString, Polygon, MultiPoint, MultiLineString,
                 MultiPolygon, GeometryCollection.
               Internally: GeoJSON decomposed into S2 cell tokens at levels 1–20
               (configurable). S2 tokens stored as compound AHIT keys.
    
    2d       — Flat 2D plane coordinates. Uses an R-tree for bounding-box and
               nearest-neighbor queries. For game maps, floor plans, etc.

  Supported Query Operators:
    { location: { $near: { $geometry: { type: 'Point', coordinates: [lng, lat] }, $maxDistance: 5000 } } }
    { location: { $nearSphere: { $geometry: { type: 'Point', coordinates: [lng, lat] }, $maxDistance: 1000 } } }
    { location: { $geoWithin: { $geometry: { type: 'Polygon', coordinates: [[[...]]] } } } }
    { location: { $geoWithin: { $centerSphere: [[lng, lat], radiusInRadians] } } }
    { location: { $geoIntersects: { $geometry: { type: 'LineString', coordinates: [[...]] } } } }

  Index Creation:
    await db.createIndex('stores', { location: '2dsphere' });
    await db.createIndex('tiles', { position: '2d' }, { min: -1000, max: 1000 });

  Performance:
    $near: S2 cell range scan expanding outward in rings. O(log N + K).
    $geoWithin polygon: S2 cell covering of polygon; scan for tokens in cover;
    exact containment test filters false positives from cell over-approximation.

8.8 Advanced Index Options [NEW]

  All index creation calls accept an options object with these properties:

  Option                   | Type    | Description
  -------------------------|---------|---------------------------------------------
  unique                   | boolean | Enforce uniqueness. Reject duplicate values.
  sparse                   | boolean | Only index docs containing the field.
  partialFilterExpression  | object  | MQL filter — only index matching docs.
  hidden                   | boolean | Maintain but never select in query planning.
  collation                | object  | Locale-aware string comparison rules.
  background               | boolean | Build index without blocking writes (default true).
  expireAfterSeconds       | number  | TTL index. Single-field Timestamp/Int64 only.

  Example:
    // Unique sparse index — email optional, but unique when present
    await db.createIndex('users', { email: 1 }, { unique: true, sparse: true });

    // Partial index — only index non-archived orders
    await db.createIndex('orders', { status: 1 }, {
      partialFilterExpression: { archived: false }
    });

    // Hidden compound index for impact testing
    await db.createIndex('events', { userId: 1, createdAt: -1 }, { hidden: true });

    // Wildcard index for dynamic schemas
    await db.createIndex('logs', { '$**': 1 });

Section 9 — Memory Management
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Oblivinx3x implements a multi-tier memory architecture that adapts to
available system memory and observed workload patterns.

9.1 Buffer Pool

  Central page cache. Fixed-size slots: buffer_pool_size / page_size entries.
  Segmented LRU eviction:
    Probationary segment — pages on first access.
    Protected segment — pages accessed a second time are promoted here.
    Eviction targets probationary first → protects hot pages from scan pollution.
  
  Default size: min(256MB, 25% of available RAM).
  Adjusts automatically via the Working Set Estimator (Section 9.2).

9.2 Working Set Estimator (WSE)

  Tracks distinct pages accessed within a rolling 60-second window (sparse bitmap).
  Cardinality of bitmap = estimated working set size.
  If working set > 80% of Buffer Pool → signal Pool Manager to increase allocation.
  If working set < 30% for > 120 seconds → reduce Buffer Pool allocation.
  WSE Adjuster worker runs every 30 seconds (Section 11.1).

9.3 MemTable Memory

  Default: 64MB max per collection (concurrent skip list).
  At limit: freeze MemTable (immutable), activate new one, queue frozen for flush.
  Max 3 frozen MemTables simultaneously before client write throttling.

9.4 AHIT Hot Zone Memory

  Dedicated memory region separate from Buffer Pool (prevents competition).
  Slab allocator: pre-sized slabs of 64B, 128B, 256B, 512B.
  Eliminates external fragmentation from variable-size AHIT node allocations.

9.5 View Materialization Memory [NEW]

  Materialized views cache their precomputed result sets in ViewPage (0x08)
  pages in the Index Segment. Not in the Buffer Pool proper — they use their
  own memory-mapped region (configurable per-view max size, default 64MB).

9.6 Garbage Collection and Version Purging

  GC Thread runs every 5 seconds.
  Computes horizon TxID = minimum TxID of all active snapshots.
  Marks versions older than horizon TxID (that have been superseded) as garbage.
  Scans Data Segment for garbage versions; reclaims page space; updates
  free-page bitmap in Metadata Segment.

9.7 Anti-Fragmentation Strategy

  Three mechanisms:
  1. Slab allocation for AHIT nodes (fixed-size = no external fragmentation).
  2. Page-aligned Buffer Pool slots (fixed-size = no heap fragmentation).
  3. Periodic Data Segment compaction during low-activity windows (rewrites
     document pages to eliminate internal fragmentation from updates/deletes).

Section 10 — Concurrency Control
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

10.1 MVCC Implementation

  Each write creates a new document version with a unique TxID.
  Previous version is NOT overwritten — retained until GC.
  Version chain: linked list of OBE-encoded versions, sorted by TxID descending.
  Primary B+ tree points to HEAD of the version chain for each document.

  Reader walk: traverse chain to find first version with TxID ≤ snapshot_TxID
  AND tombstone_flag == 0x00. O(V), V typically 1–3.

10.2 Snapshot Isolation

  Default isolation level: Snapshot Isolation (SI).
  - No dirty reads (uncommitted writes invisible).
  - No non-repeatable reads (snapshot is immutable).
  - Phantom reads prevented for collection scans (predicate locking on AHIT ranges).
  
  Optional Serializable Snapshot Isolation (SSI, Cahill algorithm):
    Detects and aborts transactions with dangerous read-write anti-dependencies.
    Enable: await db.setIsolationLevel('serializable');

10.3 Optimistic Concurrency Control

  Both writers proceed without acquiring locks.
  At commit: writer checks if any field it read (read set) was modified by a
  concurrent committed transaction with higher TxID.
  Conflict → lower TxID transaction is aborted.
  Retry: built-in, configurable max count (default 3) + exponential backoff.

10.4 Lock-Free Read Strategy

  Critical path (Buffer Pool, AHIT Hot Zone, MemTable scan) uses lock-free
  algorithms with atomic CAS operations:
    Buffer Pool page table: lock-free hash map + Hazard Pointer (prevents ABA).
    AHIT Hot Zone skip list: CAS-based atomic pointer updates.
    MemTable reads: epoch-based reclamation for safe access during concurrent writes.

10.5 Write Conflict Resolution

  Strategy: higher TxID (later writer) wins.
  Lower TxID transaction is aborted → WRITE_CONFLICT error to client.
  Client's retry logic re-executes entire transaction: new snapshot + re-eval reads.
  Avoids thundering herd of pessimistic locking.

10.6 Retryable Writes

  Transparent driver-level retry for transient failures (network interruption,
  primary failover, temporary throttle).

  Session Management:
    Each connection → logical session (lsid, 128-bit UUID).
    Sessions expire after configurable idle timeout (default 30 minutes).
    Session registry in Metadata Segment.

  Idempotency Key:
    [lsid (16B)] + [txnNumber (u64)] = idempotency key.
    Embedded in WAL record. On retry: check WAL + Idempotency Cache (LRU map).
    If found: return original result without re-executing.

  Supported: insertOne, insertMany, updateOne (unique filter), replaceOne, deleteOne.
  Non-idempotent ops ($inc, $push on arrays) require explicit opt-in.

  Configuration:
    const db = await ovn.open('data.ovn', { retryableWrites: true });

10.7 Savepoints and Nested Transaction Control [NEW]

  Savepoints allow partial rollback within a transaction without aborting
  the entire transaction. Inspired by SQLite's SAVEPOINT syntax.

  [IMPL NOTE] Savepoints are implemented as in-memory snapshots of the
  transaction's write set at the point savepoint() is called.
  They do NOT write to WAL. If the process crashes, the transaction is
  treated as uncommitted (all savepoints are lost with the parent txn).
  Savepoint state is stored in the TxContext struct, not on disk.

  Savepoint depth: unlimited in theory; configurable max depth (default 16).
  Exceeding max depth raises a SavepointDepthError.

  Use cases:
    - Multi-step import pipelines (rollback one batch without losing all batches)
    - Batch updates with partial recovery
    - Safe retry of sub-operations without full transaction restart

  API (see also Section 13.5):
    const txn = await db.beginTransaction();
    await txn.savepoint('step1');
    try {
      // risky sub-operation
      await txn.insert('items', { ... });
      await txn.releaseSavepoint('step1');
    } catch (err) {
      await txn.rollbackToSavepoint('step1');
      // outer transaction still active; can continue or commit
    }
    await txn.commit();

  Operations:
    txn.savepoint(name)           — Create savepoint named 'name' at current state.
    txn.rollbackToSavepoint(name) — Undo all writes since 'name' was created.
    txn.releaseSavepoint(name)    — Discard savepoint (no rollback); merges forward.
    txn.commit()                  — Commit all unreleased savepoint contents + parent txn.
    txn.rollback()                — Abort entire transaction including all savepoints.

Section 11 — Background Systems
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

All background workers run at configurable priority levels and are throttled
during high client load (max I/O budget = 10% of normal during high load).

11.1 Background Worker Registry

Worker Name             | Trigger                        | Priority | Function
------------------------|--------------------------------|----------|---------------------------------------
MemTable Flusher        | MemTable size > threshold      | High     | Flush frozen MemTable → L0 SSTable
Compaction Worker       | L0 SSTable count ≥ 4           | Medium   | Merge L0 SSTables → Permanent B+ Tree
WAL Cleaner             | Post-MemTable flush            | Low      | Purge committed WAL records
AHIT Promoter           | Access counter > threshold     | Low      | Promote hot AHIT subtrees to Hot Zone
AHIT Demoter            | Hot Zone > 80% capacity        | Low      | Evict cold AHIT subtrees
GC Thread               | Timer: every 5 seconds         | Low      | Purge MVCC version chains beyond horizon
Statistics Collector    | Timer: every 60 seconds        | Minimal  | Update cardinality histograms
WSE Adjuster            | Timer: every 30 seconds        | Minimal  | Resize Buffer Pool based on working set
Checkpoint Writer       | Timer: 60s OR WAL > 16MB       | Medium   | Write B+ tree root checkpoint to Page 0
Defragmenter            | Fragmentation ratio > 30%      | Minimal  | Compact Data Segment pages
Change Stream Emitter   | Post-WAL write commit          | Medium   | Write change events; notify subscribers
TTL Reaper              | Timer: every 60 seconds        | Low      | Delete expired documents
HNSW Builder            | New vector index / bulk insert | Low      | Build/update HNSW graph asynchronously
Blob Compactor          | Blob Segment fragmentation>30% | Minimal  | Rewrite Blob Segment; reclaim space
View Refresher [NEW]    | Write to source collection     | Low      | Refresh on-write materialized views
                        | OR Timer: configurable         |          |
Trigger Executor [NEW]  | Post-WAL write commit          | Medium   | Execute async (after-write) triggers
Relation Checker [NEW]  | Transaction commit             | High     | Run cascade/restrict referential checks

Change Stream Emitter Detail:
  Runs asynchronously after each committed write (post-WAL, post-ack).
  Constructs change event document:
  {
    _id:           { _data: '<resume_token>' },
    operationType: 'insert' | 'update' | 'replace' | 'delete' | 'drop' | 'rename',
    ns:            { db: '<db_name>', coll: '<collection_name>' },
    documentKey:   { _id: '<document_id>' },
    fullDocument:  { ... },              // insert/replace/update(if updateLookup)
    updateDescription: {                 // update operations only
      updatedFields:   { ... },
      removedFields:   [ ... ],
      truncatedArrays: [ ... ]
    },
    clusterTime:   <TxID>
  }
  Resume tokens allow restart from last processed event after disconnect.

TTL Reaper Detail:
  Iterates all collections with ≥ 1 TTL index.
  AHIT range scan: time_value ≤ (now_unix_ms - expireAfterSeconds * 1000).
  Batch: max 1000 deletions per cycle → commit as single WAL write (minimize fsync).

View Refresher Detail [NEW]:
  Two refresh modes per materialized view:
    'on_write'   — Refresh whenever the source collection is written.
                   Triggers after the write completes (async, non-blocking).
    'scheduled'  — Refresh on a configurable interval (e.g., every 5 minutes).
    'manual'     — Only refresh when db.refreshView(name) is called explicitly.
  Incremental refresh (when possible): re-run only the changed document through
  the pipeline and merge result into the precomputed ViewPage snapshot.
  Full refresh: re-run entire pipeline from scratch. Used when incremental
  merge is not possible (e.g., $group across all documents).

Trigger Executor Detail [NEW]:
  After-write triggers are queued after the write is acknowledged.
  They run in the Trigger Executor worker thread pool.
  Errors in async triggers are logged to observability; they never roll back
  the committed write. Trigger execution is bounded by a timeout (default 5s);
  on timeout, the trigger is logged as failed and skipped.

11.2 Compaction Strategy

  Tiered compaction specific to Hybrid B+/LSM:
  1. When L0 SSTable count ≥ 4: merge all L0 SSTables into one sorted run.
  2. Incrementally merge sorted run into Permanent B+ Tree (merge-scan algorithm):
     read sorted SSTable pages + corresponding B+ tree leaf pages in parallel
     → produce merged B+ tree structure.
  3. Shadow-copy approach: readers use original subtrees via MVCC during compaction.
  4. On completion: atomically update B+ tree root pointers (single 8-byte write).

11.3 Scheduling and Throttling

  Priority-based cooperative scheduler.
  I/O budget per worker: max page reads/writes per 100ms scheduler tick.
  When client I/O rate > 90% of system capacity:
    → Background workers reduce I/O budget to 10% of normal.
  WAL Cleaner: two-phase (mark reclaimable → rewrite WAL without reclaimable records).
  WAL cleaning during idle periods only.

Section 12 — Optional Distributed Extension (OVN-Dist)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

OVN-Dist is an optional layer on top of the core engine. It does NOT modify
the core engine internals — it uses the engine's clean internal API as a
replication substrate.

12.1 Replication Model

  Single-leader replication (Raft-inspired).
  One node in a cluster = Leader. All writes go to Leader.
  Leader applies write to local .ovn file + appends WAL record to Replication Log.
  Followers receive WAL records via streaming replication; apply locally.

  Replication Log: separate append-only binary file (.ovn-rlog).
  All committed write operations in TxID order.
  Followers maintain replication offset = their position in .ovn-rlog.
  On partition heal: Followers replay from last offset.

12.2 Leader Election

  Raft leader election protocol.
  Each node: term counter + vote log.
  Follower receives no heartbeat within election timeout (150–300ms random)
    → transitions to Candidate, requests votes from other nodes.
  Candidate becomes Leader on majority vote.
  Safety guarantee: at most one Leader per term.

12.3 Distributed Change Streams [NEW]

  Change stream events are tagged with node ID and TxID for global ordering.
  The Change Stream Emitter embeds a cluster-wide vector timestamp (node_id + TxID).
  Subscribers on any node receive events in globally consistent TxID order.
  On network partition: subscribers buffer locally; re-sync on reconnect.

12.4 Distributed Query Execution

  Scatter-gather model. Query coordinator (Leader or designated Coordinator node)
  parses query, identifies which collection partitions are involved.
  Sends sub-queries to relevant nodes. Each node executes locally.
  Coordinator merges, sorts, and projects combined results.

12.5 Sharding Strategy

  Hash Sharding (default):
    64-bit hash of shard key field value → shard whose range covers hash.
  Range Sharding:
    Lexicographic ranges of shard key → shards.
    Efficient for range queries across shards.

Section 13 — Library API Design
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The Oblivinx3x API mirrors MongoDB MQL while being idiomatic for the host
language. Primary binding: JavaScript via Neon (Node.js native addon).
All async operations return Promises.

[IMPL NOTE] The JavaScript API layer (packages/oblivinx3x/) MUST NOT expose
raw Rust pointers, buffer offsets, or internal type tags to consumers. All
internal details must be hidden behind the clean TypeScript interface.

13.1 Database Lifecycle

  const ovn = require('oblivinx3x');

  // Open or create a database
  const db = await ovn.open('data.ovn', {
    pageSize:        4096,      // Page size in bytes (512 to 65536, power of 2)
    bufferPool:      '128MB',   // Buffer pool size
    walMode:         true,      // Enable WAL for concurrent reads
    compression:     'lz4',    // Page compression: 'none' | 'lz4' | 'zstd'
    readOnly:        false,     // Open in read-only mode
    retryableWrites: true,      // Enable retryable writes (Section 10.6)
    foreignKeys:     'strict',  // Referential integrity: 'off' | 'soft' | 'strict' [NEW]
    attachments:     []         // Pre-attach databases on open [NEW]
  });

  await db.close();             // Graceful close: flush MemTable + checkpoint
  await db.checkpoint();        // Manual checkpoint: flush MemTable + WAL → B+ Tree
  await db.backup('copy.ovn');  // Online hot backup (Section 15.2)
  await db.export('json', 'out.jsonl', { collection: 'users' }); // Export

13.5 Transactions (including Savepoints)

  // ACID transaction
  const txn = await db.beginTransaction();
  try {
    await txn.insert('accounts', { userId: 'u1', balance: 1000 });
    await txn.update('accounts', { userId: 'u1' }, { $inc: { balance: -200 } });
    await txn.update('accounts', { userId: 'u2' }, { $inc: { balance:  200 } });
    await txn.commit();
  } catch (err) {
    await txn.rollback();
    throw err;
  }

  // Transaction with Savepoints [NEW]
  const txn2 = await db.beginTransaction();
  await txn2.savepoint('before_bulk');
  try {
    for (const item of largeBatch) {
      await txn2.insert('items', item);
    }
    await txn2.releaseSavepoint('before_bulk');  // success: merge forward
  } catch (err) {
    await txn2.rollbackToSavepoint('before_bulk'); // failure: undo bulk, keep outer txn
    // can continue with alternative logic
  }
  await txn2.commit();

  // Transaction spanning attached databases [NEW]
  // (requires db.pragma('cross_db_transactions', true))
  const txnMulti = await db.beginTransaction({ crossDb: true });
  await txnMulti.insert('users', { name: 'Bob' });
  await txnMulti.insert('analytics.events', { type: 'user_created' });  // 'analytics' = attached db
  await txnMulti.commit();

13.6 Aggregation Pipeline

  // MongoDB-compatible stages
  const result = await db.aggregate('orders', [
    { $match:  { status: 'completed', year: 2025 } },
    { $group:  { _id: '$customerId', totalAmount: { $sum: '$amount' }, orderCount: { $sum: 1 } } },
    { $sort:   { totalAmount: -1 } },
    { $limit:  20 }
  ]);

  // $facet — multiple sub-pipelines in one pass
  const facetResult = await db.aggregate('products', [
    { $match: { status: 'active' } },
    { $facet: {
      byCategory: [
        { $group: { _id: '$category', count: { $sum: 1 } } },
        { $sort: { count: -1 } }
      ],
      priceDistribution: [
        { $bucket: {
          groupBy:    '$price',
          boundaries: [0, 25, 50, 100, 250, 500],
          default:    'Other',
          output:     { count: { $sum: 1 }, avgPrice: { $avg: '$price' } }
        }}
      ],
      topBrands: [
        { $group: { _id: '$brand', revenue: { $sum: '$price' } } },
        { $sort: { revenue: -1 } },
        { $limit: 5 }
      ]
    }}
  ]);

  // $bucketAuto
  const priceBuckets = await db.aggregate('products', [
    { $bucketAuto: { groupBy: '$price', buckets: 5, output: { count: { $sum: 1 }, avgPrice: { $avg: '$price' } } } }
  ]);

  // $setWindowFields — running totals, moving averages, window ranks
  const runningTotals = await db.aggregate('sales', [
    { $setWindowFields: {
      partitionBy: '$region',
      sortBy:      { date: 1 },
      output: {
        runningTotal: { $sum: '$amount', window: { documents: ['unbounded', 'current'] } },
        movingAvg:    { $avg: '$amount', window: { documents: [-2, 0] } },
        rank:         { $rank: {} }
      }
    }}
  ]);

  // $merge — write aggregation results into target collection
  await db.aggregate('orders', [
    { $match: { year: 2025 } },
    { $group: { _id: '$customerId', totalSpend: { $sum: '$amount' } } },
    { $merge: {
      into:           'customerSummary',
      on:             '_id',
      whenMatched:    'merge',   // merge | replace | keepExisting | fail
      whenNotMatched: 'insert'   // insert | discard | fail
    }}
  ]);

  // $replaceRoot / $replaceWith
  const normalized = await db.aggregate('users', [
    { $replaceRoot: { newRoot: '$address' } }
  ]);

  // $unwind with preserveNullAndEmptyArrays
  const unwound = await db.aggregate('orders', [
    { $unwind: { path: '$items', preserveNullAndEmptyArrays: true } }

13.6.1 Join and Relationship Operators [NEW]

  Oblivinx3x supports MongoDB-style $lookup for join-like document association
  and $graphLookup for relationship-heavy graph traversal workloads.
  These operators integrate with the Relational Compatibility Layer (Section 4.7)
  when relations are defined — the query planner can use relation metadata and
  relation indexes to optimize join plans.

  // $lookup — standard foreign-key-like join
  const postsWithAuthors = await db.aggregate('posts', [
    {
      $lookup: {
        from:         'users',
        localField:   'user_id',
        foreignField: '_id',
        as:           'author'
      }
    }
  ]);

  // $lookup with pipeline (arbitrary sub-pipeline on joined collection)
  const enriched = await db.aggregate('orders', [
    {
      $lookup: {
        from:     'products',
        let:      { productId: '$productId' },
        pipeline: [
          { $match: { $expr: { $eq: ['$_id', '$$productId'] } } },
          { $project: { name: 1, price: 1 } }
        ],
        as:       'product'
      }
    }
  ]);

  // $graphLookup — recursive graph traversal (e.g., org charts, category trees)
  const orgChart = await db.aggregate('employees', [
    {
      $graphLookup: {
        from:             'employees',
        startWith:        '$reportsTo',
        connectFromField: 'reportsTo',
        connectToField:   '_id',
        as:               'reportingChain',
        maxDepth:         5,
        depthField:       'depth',
        restrictSearchWithMatch: { active: true }
      }
    }
  ]);

13.8 Views and Materialized Views [NEW]

  [IMPL NOTE] Views are stored as pipeline definitions in the Metadata Segment.
  Logical views: redirect reads to source + prepend pipeline; no data stored.
  Materialized views: precomputed result stored in ViewPage (0x08) pages;
  reads are O(1) to O(K); writes trigger async refresh (Section 11.1).

  // Create a logical view (stored query — always live data)
  await db.createView('active_users', {
    source:   'users',
    pipeline: [
      { $match:   { active: true } },
      { $project: { name: 1, email: 1 } }
    ]
  });

  // Query a view exactly like a collection
  const users = await db.find('active_users', { name: /Alice/ });

  // Create a materialized view (precomputed — fast reads; async refresh)
  await db.createView('sales_summary', {
    source:   'orders',
    pipeline: [
      { $match: { status: 'completed' } },
      { $group: { _id: '$region', total: { $sum: '$amount' } } }
    ],
    materializedOptions: {
      refresh:  'on_write',    // 'on_write' | 'scheduled' | 'manual'
      schedule: '*/5 * * * *', // cron expression (for 'scheduled' mode)
      maxSize:  '64MB'         // maximum storage for precomputed results
    }
  });

  // Manually refresh a materialized view
  await db.refreshView('sales_summary');

  // List all views
  const views = await db.listViews();

  // Drop a view
  await db.dropView('active_users');

  // Modify a view definition
  await db.command({
    collMod: 'active_users',
    viewOn:  'users',
    pipeline: [
      { $match:   { active: true, verified: true } },
      { $project: { name: 1, email: 1 } }
    ]
  });

13.9 Triggers and Event Hooks [NEW]

  [IMPL NOTE] Triggers are stored as serialized function definitions in the
  Metadata Segment. The engine does NOT store arbitrary JavaScript code in the
  .ovn file — instead, triggers are registered as named hooks; the host process
  provides the function implementation at open time via the hook registry.
  This ensures the .ovn file format remains language-agnostic.

  Supported trigger events:
    beforeInsert    — Sync; can modify or reject the document. Runs before WAL write.
    afterInsert     — Async; runs after WAL commit. Cannot reject the write.
    beforeUpdate    — Sync; receives (filter, update, ctx). Can modify or reject.
    afterUpdate     — Async; runs after WAL commit.
    beforeDelete    — Sync; receives (filter, ctx). Can cancel the deletion.
    afterDelete     — Async; runs after WAL commit.

  [IMPL NOTE] Sync triggers run on the client thread (in the write path,
  Step 3 of Section 6.2). Async triggers are queued through the Trigger Executor
  background worker (Section 11.1). A sync trigger that throws cancels the write
  with a TriggerError. An async trigger that throws is logged; it does NOT
  affect the committed write.

  // Register triggers on db.open() or db.createCollection()
  await db.createTrigger('users', 'beforeInsert', async (doc, ctx) => {
    if (!doc.email) throw new Error('email is required');
    doc.createdAt = Date.now();
    return doc;   // return modified document
  });

  await db.createTrigger('users', 'afterInsert', async (doc, ctx) => {
    // e.g., send welcome email, update cache, publish to message bus
    await notificationService.send(doc.email, 'welcome');
  });

  await db.createTrigger('orders', 'beforeDelete', async (filter, ctx) => {
    const order = await ctx.db.findOne('orders', filter);
    if (order && order.status !== 'cancelled') {
      throw new Error('Can only delete cancelled orders');
    }
  });

  await db.createTrigger('posts', 'afterUpdate', async (filter, update, ctx) => {
    // Invalidate cache, etc.
  });

  // List triggers on a collection
  const triggers = await db.listTriggers('users');

  // Remove a trigger
  await db.dropTrigger('users', 'beforeInsert');

  // Trigger context object (ctx):
  // {
  //   db:        <database handle — can read but not write outside trigger scope>,
  //   txnId:     <current TxID>,
  //   sessionId: <client session lsid>,
  //   collection: '<collection name>'
  // }

13.10 Explain and Query Diagnostics [NEW]

  [IMPL NOTE] db.explain() must NOT execute the query. It runs only the

  databases without performance impact.

  // Explain a find query
  const plan = await db.explain('users', { age: { $gt: 18 } });
  console.log({
    chosenIndex:        plan.index,          // e.g., 'age_1' or null (full scan)
    scanType:           plan.scanType,       // 'indexScan' | 'collectionScan' | 'coveredScan'
    estimatedCost:      plan.estimatedCost,  // numeric cost value
    docsExamined:       plan.docsExamined,   // estimated docs read
    docsReturned:       plan.docsReturned,   // estimated docs matched
    fallbackReason:     plan.fallbackReason, // string if full scan selected, null otherwise
    stages:             plan.stages          // array of execution stage descriptions
  });

  // Explain an aggregate pipeline
  const aggPlan = await db.explainAggregate('orders', [
    { $match:  { status: 'completed' } },
    { $group:  { _id: '$customerId', total: { $sum: '$amount' } } }

  // Verbose explain (includes actual execution stats — DOES execute the query)
  const verbosePlan = await db.explain('users', { age: { $gt: 18 } }, {
    verbosity: 'executionStats'  // 'queryPlanner' (default) | 'executionStats' | 'allPlansExecution'
  });

13.11 Pragmas and Engine Directives [NEW]

  [IMPL NOTE] Pragmas are stored in the Metadata Segment and persist across
  db.open() calls. They override the defaults set in the open() options object.
  A pragma set during one session takes effect immediately and persists
  for all subsequent sessions unless explicitly reset.

  Supported pragmas:
    Pragma Name              | Type    | Description
    -------------------------|---------|----------------------------------------------
    foreign_keys             | boolean | Enable/disable relational layer enforcement
    synchronous              | string  | WAL fsync mode: 'off' | 'normal' | 'full'
    auto_vacuum              | boolean | Enable automatic Data Segment compaction
    cache_size               | number  | Buffer pool size in bytes
    page_size                | number  | Read-only after DB creation; shows current size
    journal_mode             | string  | 'wal' (default) | 'none'
    referential_integrity    | string  | 'off' | 'soft' | 'strict'
    cross_db_transactions    | boolean | Allow transactions across attached DBs [NEW]
    wal_checkpoint           | string  | Action: 'passive' | 'full' | 'restart' | 'truncate'
    optimize                 | boolean | Trigger Statistics Collector + AHIT optimization
    integrity_check          | string  | Scope: 'file' | 'wal' | 'indexes' | 'full'
    max_savepoint_depth      | number  | Maximum nested savepoint depth (default 16)
    materialized_view_size   | string  | Default max size for materialized views

  Examples:
    await db.pragma('foreign_keys', true);
    await db.pragma('synchronous', 'full');
    await db.pragma('cache_size', 268435456);   // 256MB in bytes
    await db.pragma('referential_integrity', 'strict');
    await db.pragma('integrity_check', 'full'); // scans all pages + indexes for corruption
    await db.pragma('wal_checkpoint', 'truncate');

  // Read current pragma value
  const mode = await db.pragma('synchronous');
  console.log(mode); // 'full'

13.12 Attached Databases [NEW]

  [IMPL NOTE] Attached databases are fully independent .ovn files that are
  opened and registered under an alias. They share the same session and can
  participate in cross-database queries and transactions (when pragma
  cross_db_transactions = true). Each attached database maintains its own
  Buffer Pool, MemTable, and WAL. They do NOT share a buffer pool with the
  primary database.

  Attachment metadata is stored in the Attached DB Index segment (0x08) and
  persists across db.close() / db.open() cycles (auto-reattached on open).

  // Attach an existing .ovn file
  await db.attach('analytics.ovn', 'analytics');
  // Now accessible as collection path: 'analytics.events'

  // Query an attached database
  const events = await db.find('analytics.events', { type: 'purchase' });

  // Cross-database aggregation
  const crossResult = await db.aggregate('orders', [
    { $lookup: {
      from:         'analytics.events',   // cross-db reference
      localField:   '_id',
      foreignField: 'orderId',
      as:           'events'
    }}
  ]);

  // List attached databases
  const attached = await db.listAttached();
  // Returns: [{ alias: 'analytics', path: 'analytics.ovn', status: 'open' }]

  // Detach an attached database (closes it; removes from alias registry)
  await db.detach('analytics');

  // Rules:
  //   - Each attached database has its own namespace (alias prefix)
  //   - Cross-database joins are explicitly enabled via pragma cross_db_transactions
  //   - Transactions spanning attached databases require beginTransaction({ crossDb: true })
  //   - Attached database alias cannot conflict with existing collection names
  //   - Attached database cannot attach other databases (no recursive attach)

15.1 Crash Recovery

  Guarantees recovery to last committed transaction after: SIGKILL, power failure, OS crash.

  Recovery procedure:
    1. Read Page 0. Verify magic (0x4F564E58) and CRC32.
    2. If WAL Active flag set: previous session did not cleanly checkpoint.
    3. Scan WAL segment forward from last checkpoint TxID.
    4. For each WAL record with COMMIT marker: re-apply to MemTable.
    5. For each WAL record without COMMIT marker: discard (partial transaction).
    6. Flush recovered MemTable → L0 SSTable → trigger compaction.
    7. Write new checkpoint to Page 0. Clear WAL Active flag.

  Recovery time ∝ WAL size since last checkpoint.
  With default 60-second checkpoint interval: recovery typically < 2 seconds.

  [IMPL NOTE] Savepoint state is in-memory only (Section 10.7). On crash,
  the entire parent transaction is treated as uncommitted — all savepoints lost.

15.2 Backup Systems

  Hot Backup (Online, no writes paused):
    db.backup(destinationPath) → triggers checkpoint → streaming page copy
    while tracking concurrent writes → modified pages re-read and appended.
    Result: point-in-time consistent copy of the .ovn file.

  WAL-Based Incremental Backup:
    OVN-Dist Replication Log (.ovn-rlog) streamed to remote backup store.
    Backup agent connects as read-only replica. RPO = seconds.

  Snapshot Export:
    db.export(format, path, options) → exports a collection.
    Formats: 'json' (JSON Lines), 'csv', 'bson'.
    Consistent read via MVCC snapshot.

15.3 Durability Guarantees

  D0 — No fsync (in-memory only; data loss on crash). Highest throughput.
  D1 — WAL writes fsync'd before client ack (default). Full durability.
  D2 — Group fsync (batch WAL records, latency/durability trade-off).
  Configure per-write: await db.insert('users', doc, { durability: 'D0' });

GeoJSON                   | Open standard for encoding geographic data (Point, Polygon, etc.) as JSON.

lsid                      | Logical Session ID — 128-bit per-connection session identifier.
LSM Tree                  | Log-Structured Merge-tree — stages writes in memory before sequential disk flush.
Materialized View         | Precomputed, stored result of a pipeline query; refreshed asynchronously.
MVCC                      | Multi-Version Concurrency Control — multiple doc versions coexist simultaneously.
OBE                       | OVN Binary Encoding — Oblivinx3x's document serialization format.
OVN                       | Oblivinx Storage Format — the .ovn file format.
Partial Index             | Index that only includes documents matching a partialFilterExpression.
R-tree                    | Spatial index for multi-dimensional data (used for 2d planar geospatial index).
Referential Integrity     | Enforcement of cross-collection reference validity (off / soft / strict).
RelRef                    | OBE type tag (0x12) for typed cross-collection document references.
Resume Token              | Opaque token in change stream events for restarting after disconnect.
Retryable Write           | Driver-level idempotent write retry using session-level (lsid, txnNumber).
S2 Geometry               | Google's spherical geometry library partitioning Earth into quadrilateral cells.
Savepoint                 | Partial rollback point within a transaction; in-memory only; not WAL-persistent.
SCO                       | Skip-Column Optimization — enables non-leftmost compound index queries.
Sparse Index              | Index that skips documents where the indexed field is absent.
SSI                       | Serializable Snapshot Isolation — full serializability via SI + anomaly detection.
SSTable                   | Sorted String Table — immutable sorted KV file in LSM-tree architecture.
Time-Series Collection    | Collection optimized for time-ordered measurements via automatic bucketing.
Trigger                   | Collection-level hook: sync (beforeWrite) or async (afterWrite) function.
TTL Index                 | Secondary index with expireAfterSeconds; causes TTL Reaper deletions.
TxID                      | Transaction ID — monotonically increasing u64 identifying each write transaction.
Unique Index              | Index that rejects writes with duplicate values.
validationAction          | 'error': reject invalid doc; 'warn': log warning and allow write.
validationLevel           | 'strict': validate all writes; 'moderate': skip non-conforming existing docs.
View                      | Named stored query (logical) or precomputed snapshot (materialized).
WAL                       | Write-Ahead Log — writes logged sequentially before applying to main structure.
Wildcard Index            | Index that covers all fields ($**) or a subtree of fields in documents.
Working Set               | Set of pages actively accessed in the current workload's rolling time window.
WSE                       | Working Set Estimator — dynamically adjusts Buffer Pool size based on WSE.

Appendix B — .OVN File Format Quick Reference
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

File Layout (default 4096-byte pages):

  Page 0    : File Header (magic, version, page size, root pointers, flags)
  Page 1    : Segment Directory (maps segment type codes → page ranges)
  Page 2-N  : Data Segment (document leaf pages, overflow pages)
  Page N-M  : Index Segment (AHIT B+ tree nodes, SSTable index blocks,
                              inverted index, BK-tree, autocomplete trie,
                              geospatial R-tree / S2 cell index, HNSW graph,
                              ViewPage snapshots for materialized views)
  Page M-P  : WAL Segment (write-ahead log records)
  Page P-Q  : Metadata Segment (collection schemas, statistics, version history,
                                 session registry, idempotency cache, relation
                                 definitions, view definitions, trigger hook
                                 registry, pragma state)
  Page Q-R  : Blob Segment (chunked binary large objects, Blob Directory B+ tree)
  Page R-S  : Change Stream Log (append-only circular buffer of change events)
  Page S-T  : Attached DB Index (alias → .ovn path map) [NEW]

Header Flags (byte at 0x003C):
  Bit 0: WAL active
  Bit 1: Encrypted
  Bit 2: Compressed (LZ4)
  Bit 3: Has Blob Segment
  Bit 4: Has Change Stream Log
  Bit 5: Has Relational Layer metadata [NEW]
  Bit 6: Has View definitions [NEW]
  Bit 7: Has Trigger definitions [NEW]

Page Header (32 bytes, every page):
  [4B] Page Type | [8B] Page Number | [8B] TxID | [2B] Slot Count |
  [2B] Free Space Offset | [4B] Right Sibling | [4B] CRC32

OBE Document (variable length):
  [4B] Total Length | [16B] Document ID | [8B] TxID | [1B] Tombstone |
  [varint] Field Count | [Field...] (Name/ID + Type Tag + Value)

OBE Type Tags Quick Reference:
  0x01 Null        0x02 BoolTrue   0x03 BoolFalse  0x04 Int32
