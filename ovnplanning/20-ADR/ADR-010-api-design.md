# ADR-010 — Multi-Surface API with Rust as the Canonical Definition

**Status:** Accepted, 2026-04
**Owners:** API / SDK
**Cross refs:** `[[FILE-13]]`

---

## Context

Oblivinx3x must be usable from many environments:

1. Embedded Rust applications (tightest integration).
2. Node.js servers and Electron apps (Neon).
3. Python data pipelines (PyO3).
4. Other languages via C ABI (Go, Swift, Java, .NET, Zig).
5. Optional sidecar over REST and gRPC for non-embedded callers.
6. Browsers (WASM + OPFS).

The design challenge is keeping all surfaces semantically uniform without diluting any one of them.

## Decision

* **Rust API is canonical.** Every other surface is defined as a faithful mirror; non-Rust surfaces document themselves as "the X-equivalent of Rust function Y".
* Async-by-default in host languages that support it; sync wrappers exist but documented as "costs a thread".
* **Builder pattern** for option-heavy ops (Find); option struct (`#[non_exhaustive]`) for Engine.
* Errors as values in Rust (`Result<_, OvnError>`); exceptions in Node/Python; status codes + thread-local last-error in C.
* Cursor model: pull-based with batching; WebSocket for change streams.
* C ABI is **the** integration point for everything that isn't Rust/Node/Python/WASM. The C header is generated by `cbindgen` from Rust definitions to keep drift minimal.
* REST is path-versioned (`/v1/...`); response error envelope identical across endpoints; HTTP status codes follow strict mapping (`OvnError` → status).
* gRPC offered for callers that prefer it but lives behind the same engine handle.

Stability promises:

* SemVer for crates and packages.
* Wire/REST: removing a field is breaking; adding optional fields is non-breaking.
* C ABI: `OVN_ABI_VERSION` constant; major bump on breaking change.
* Deprecations: 2 minor releases of warnings before removal.

## Consequences

**Positive**

* Single source of semantic truth (Rust); doc/test cost reduced.
* New surfaces can be added (e.g. PHP, Ruby) without re-deciding semantics.
* C ABI from cbindgen keeps multi-language SDKs aligned.

**Negative**

* Multiple surfaces means multiple test matrices.
* Driver authors must respect the canonical types or subtle bugs propagate.
* Async/sync duality requires careful documentation.

## Alternatives considered

* **REST/gRPC only** — rejected: defeats embedded use case.
* **Server-only architecture** — rejected: misaligned with project's embedded-first stance.
* **Hand-written C header** — rejected: drift inevitable; `cbindgen` keeps Rust→C in sync.

## Open questions

* GraphQL surface: community plugin vs core?
* Multi-document atomic bulk across collections (would require global txn coordination) — deferred.
* Rust async runtime independence (Tokio vs Smol vs custom) — currently Tokio-native; revisit when async-trait stabilizes further.

*End of ADR-010.*
